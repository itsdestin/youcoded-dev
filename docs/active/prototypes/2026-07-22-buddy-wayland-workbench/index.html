I2 · click-through + forward events

Turn this on, then move the mouse over the empty area. If pointer events keeps climbing, forward works and hover tracking is viable. If it freezes, the one-window model needs a different hover strategy.

I3 · setShape input region

Binds input to the mascot rect only. If clicks land on the square but pass through everywhere else, this is the clean solution — no polling needed.

I4 · always-on-top

Click another window to focus it. This overlay should stay visible above it. KWin reports keepAbove=false for this window, so this may well fail — that is exactly what we need to know.

I5 · compositor-driven window move

Wayland forbids the app from moving its own window, but it CAN ask the compositor to drag it. Drag the orange strip below: if the whole window moves, dragging is possible without the one-window rewrite (though the app still can't know where the window ended up).
