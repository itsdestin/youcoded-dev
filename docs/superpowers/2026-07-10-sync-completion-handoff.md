### A01. Project LIFECYCLE вЂ” rename / un-sync DELIVERED with A00; residual pieces genuinely deferred (updated 2026-07-13)
**Rename (display-name) + un-sync (stop) are now IMPLEMENTED on `feat/sync-project-discovery`** as part of A00 (the design absorbed them from the start вЂ” cheaper than bolting on later). What shipped:
- **Rename a synced project** вЂ” display-name-only, propagates to every device via the synced `displayName` field (LWW merge) + a read-time overlay in the status payload (no folder move, no `youcoded-folders.json` writeback). - **Un-sync ("Stop syncing")** вЂ” a `stopped` tombstone (monotonic, `stopped`-dominates) + `engine.removeSpace` that KEEPS each device's local copy; the `activeManagedSpaces()` gate stops it respawning on any device. Consequence-gated Project View button; gray "Sync stopped" dot.
- **Delete-propagation / respawn** is now moot for the sanctioned path: "Stop syncing" is the tombstoned detach. Raw OS-delete of a still-active folder still restores (documented, standard sync semantics).

**Genuinely deferred (NOT release-gating on their own вЂ” the core lifecycle is covered; spec В§15):**
- **Resume syncing a stopped project.** `stopped` is a permanent tombstone (monotonic so a stale rename can't un-stop it). Clean upgrade path: a per-field timestamp on `state` (or a generation counter) so a resume outranks the stop.
- **True on-disk folder rename** (coordinated multi-device folder move + transcript-slug/artifact-sidecar/central-index remap Г— N devices) вЂ” display-name rename covers "same name everywhere" without it.
- **Remote GitHub repo cleanup on stop** вЂ” the provisioned repo is orphaned, not deleted (destructive cross-account action).
- **Project View 3-category UI** (native-synced / native-unsynced / external + `(i)` explainer) вЂ” the status dot is the interim legibility; do it once discover/rename/stop is dogfooded.
These residuals can build on Plan 2b's coordination primitives if/when picked up; none blocks a release under the "sync complete" rule now that appearance + rename + stop work.

### D. Two-device dogfood (THE GATE before Plan 2b вЂ” needs a 2nd machine)
Dev builds on a second machine: sign in, enable Sync, verify Personal + a project space converge, conflict copies materialize correctly, second-device provisioning reuses device 1's repos, and signals make it near-instant. **Procedure (both machines run a DEV build вЂ” `bash scripts/run-dev.sh` вЂ” not the released app):**
1. **Both machines:** launch `bash scripts/run-dev.sh` (dev window on shifted ports; shares `~/.claude` but its own userData profile). First launch runs the reconciler вЂ” expect it to populate `~/YouCoded/Personal/Conversations/` locally (records + mirrored transcripts). With youcoded#118 shipped, conversations key to their REAL projects (e.g. `youcoded-dev`), not all bucketed to `desti`. **Sanity-check this first** (`ls ~/YouCoded/Personal/Conversations/claude/transcripts/` should show real project names, not just `desti`).
2. **Both machines:** sign in to the SAME account (marketplace auth), then **enable Sync** in the SyncPanel. Machine 1 provisions the personal-space GitHub repo; machine 2 must REUSE it (provisioning treats an existing repo as success вЂ” verify no duplicate repo, no error).
3. **Conversation convergence (the Plan 2a payoff):** run a short conversation on machine A в†’ within a few seconds (SyncHub signal) it should appear in machine B's **Resume Browser**. Open it on B and confirm **resume works** (`claude --resume` finds a materialized transcript). Then the reverse (Bв†’A).
4. **Project-space convergence:** edit a file in a synced project folder on A в†’ confirm it converges on B; make conflicting edits offline on both в†’ confirm the convergent policy (remote wins canonical, local becomes a visible `(from <device>)` conflict copy) and that BOTH devices end identical.

**What to watch / known 2a caveats (not bugs):**
- **A LIVE (running) session is deliberately NOT offered for resume on the other device** (no leases until 2b вЂ” the sweep + browser both skip live sessions to avoid two `claude --resume` on one transcript). Takeover UX is Plan 2b.
- **Accepted caveat (reduced by #120):** an ENDED session stays sweep-guarded until app restart (nothing clears the live-sessions map on exit yet вЂ” `noteSessionEnded` lands in 2b). A peer version now reliably materializes **on the next launch** thanks to the startup catch-up sweep (youcoded#120); applying it **without** a restart is the deferred Part-2-of-the-dogfood-fix, owned by 2b (design В§3 "Materialize-on-release"). Before #120, it did NOT materialize even after a restart вЂ” that was the Bug 2 below.
- A store-only Resume Browser row can briefly read **"Not synced to this device yet"** until the same-pull sweep materializes its transcript вЂ” self-corrects on the next browser open.
- Transcripts **>50 MB** won't push (engine `MAX_SYNC_FILE_BYTES` cap) вЂ” they list but won't materialize cross-device.
- The **live-intake path** (an in-app turn writing to the store) is UNVERIFIED at runtime вЂ” the 2a smoke test only exercised the startup reconciler. Confirming a fresh in-app conversation shows up on the other device is the key new thing this dogfood proves.
