### A01. Project LIFECYCLE — rename / un-sync DELIVERED with A00; residual pieces genuinely deferred (updated 2026-07-13)
**Rename (display-name) + un-sync (stop) are now IMPLEMENTED on `feat/sync-project-discovery`** as part of A00 (the design absorbed them from the start — cheaper than bolting on later). What shipped:
- **Rename a synced project** — display-name-only, propagates to every device via the synced `displayName` field (LWW merge) + a read-time overlay in the status payload (no folder move, no `youcoded-folders.json` writeback). Project View "Rename" routes through `syncSpaces.renameProject` for synced projects; local folders keep `folders.rename`.
- **Un-sync ("Stop syncing")** — a `stopped` tombstone (monotonic, `stopped`-dominates) + `engine.removeSpace` that KEEPS each device's local copy; the `activeManagedSpaces()` gate stops it respawning on any device. Consequence-gated Project View button; gray "Sync stopped" dot.
- **Delete-propagation / respawn** is now moot for the sanctioned path: "Stop syncing" is the tombstoned detach. Raw OS-delete of a still-active folder still restores (documented, standard sync semantics).

**Genuinely deferred (NOT release-gating on their own — the core lifecycle is covered; spec §15):**
- **Resume syncing a stopped project.** `stopped` is a permanent tombstone (monotonic so a stale rename can't un-stop it). Clean upgrade path: a per-field timestamp on `state` (or a generation counter) so a resume outranks the stop.
- **True on-disk folder rename** (coordinated multi-device folder move + transcript-slug/artifact-sidecar/central-index remap × N devices) — display-name rename covers "same name everywhere" without it.
- **Remote GitHub repo cleanup on stop** — the provisioned repo is orphaned, not deleted (destructive cross-account action).
- **Project View 3-category UI** (native-synced / native-unsynced / external + `(i)` explainer) — the status dot is the interim legibility; do it once discover/rename/stop is dogfooded.
These residuals can build on Plan 2b's coordination primitives if/when picked up; none blocks a release under the "sync complete" rule now that appearance + rename + stop work.

**Procedure (both machines run a DEV build — `bash scripts/run-dev.sh` — not the released app):**

3. **Conversation convergence (the Plan 2a payoff):** run a short conversation on machine A → within a few seconds (SyncHub signal) it should appear in machine B's **Resume Browser**. Open it on B and confirm **resume works** (`claude --resume` finds a materialized transcript). Then the reverse (B→A).
4. **Project-space convergence:** edit a file in a synced project folder on A → confirm it converges on B; make conflicting edits offline on both → confirm the convergent policy (remote wins canonical, local becomes a visible `(from <device>)` conflict copy) and that BOTH devices end identical.

**What to watch / known 2a caveats (not bugs):**
- **A LIVE (running) session is deliberately NOT offered for resume on the other device** (no leases until 2b — the sweep + browser both skip live sessions to avoid two `claude --resume` on one transcript). Takeover UX is Plan 2b.
- **Caveat RESOLVED by Plan 2b** (was: an ENDED session stayed sweep-guarded until app restart): `noteSessionEnded` now clears the live-sessions guard on `session-exit` and runs a targeted quiescence-gated `materializeOne`, so a peer version applies **without** a restart. The startup catch-up sweep (youcoded#120) remains as the launch-time backstop. Before #120 it did NOT materialize even after a restart — that was Bug 2.
- A store-only Resume Browser row can briefly read **"Not synced to this device yet"** until the same-pull sweep materializes its transcript — self-corrects on the next browser open.
- Transcripts **>50 MB** won't push (engine `MAX_SYNC_FILE_BYTES` cap) — they list but won't materialize cross-device.
- The **live-intake path** (an in-app turn writing to the store) is UNVERIFIED at runtime — the 2a smoke test only exercised the startup reconciler. Confirming a fresh in-app conversation shows up on the other device is the key new thing this dogfood proves.
