# Pitfalls & Architectural Invariants

Every item here is a lesson learned the hard way or a constraint that's invisible from reading code alone. Violating these silently breaks things.

## Cross-Platform (Desktop + Android)

- **`preload.ts` and `remote-shim.ts` must expose the same SHARED `window.claude` shape.** If one has a shared API the other lacks, React crashes on that platform. Intentional exceptions: `window.claude.window` (Electron-only, for minimize/maximize/close) and `window.claude.android` (Android-only).
- **Desktop handlers return raw values; Android wraps in JSONObject.** The shim must normalize both to a consistent shape before React sees them.
- **Message type strings must be identical across preload.ts, ipc-handlers.ts, and SessionService.kt.** A typo silently fails on one platform.
- **The React UI bundle must be present in `app/src/main/assets/web/assets/` before APK packaging** or the Android app launches with a blank WebView (`index.html` references JS/CSS bundles that aren't shipped → `net::ERR_FILE_NOT_FOUND` → empty `<div id="root">`). The `bundleWebUi` Gradle task in `app/build.gradle.kts` runs `scripts/build-web-ui.sh` automatically before `preBuild`, with input tracking against `desktop/src/` so it skips when nothing changed. **Do not bypass the task** (e.g., `-x bundleWebUi`) without manually running `bash scripts/build-web-ui.sh` first — that's how this footgun returned in April 2026 after a rebase that committed a new `index.html` bundle hash without rebuilding the assets locally.
- **When you add CC-coupled code, add an entry to `youcoded/docs/cc-dependencies.md`.** That spine doc feeds the `review-cc-changes` release agent, which maps Claude Code CHANGELOG entries to YouCoded code that might break. An omitted touchpoint silently downgrades the agent to free-reasoning-only mode for that area — don't rely on the agent to catch a coupling it doesn't know exists. Coupling includes: parsing CC output (transcript JSONL, statusline JSON), consuming a CC file (settings.json, installed_plugins.json), depending on CLI behavior (flags, exit codes, prompt text), or matching a CC text pattern (spinner glyphs, prompt markers).
- **Transcript parser has two implementations behind a `TranscriptSource` interface.** Default is the legacy Kotlin `TranscriptWatcher`. Opt-in `TranscriptWatcherProcess` spawns the bundled `transcript-watcher-cli.js` (same TS code that runs in-process on Electron) under Termux Node and forwards NDJSON events. Toggle via `transcriptWatcher.useNodeProcess: true` in `~/.claude-mobile/config.json`. **The Node-CLI path is the single source of truth** — when modifying transcript parsing, edit `desktop/src/main/transcript-watcher.ts` first, then mirror only what's necessary into `app/.../parser/TranscriptWatcher.kt` (which exists solely as a fallback when the CLI bundle is missing). Drift is caught by `desktop/tests/transcript-parity.test.ts` running the canonical fixtures at `youcoded/shared-fixtures/transcript-parity/`.
- **Android-only events the Kotlin watcher doesn't emit but the Node CLI does:** `user-interrupt` (Claude Code's `[Request interrupted by user]` markers; Kotlin currently renders these as user bubbles instead of cleanly ending the turn) and `assistant-thinking` (extended-thinking heartbeats). Opting into the Node CLI path fixes both. The Kotlin TranscriptEvent enum has the matching types (UserInterrupt, AssistantThinking) for the CLI path's deserialization, even though the Kotlin parser doesn't emit them.
- **`shared-fixtures/transcript-parity/` is the contract.** Hand-curated JSONL inputs paired with `<name>.expected.json` event streams. The desktop parity test runs `parseTranscriptLine` against each fixture and asserts byte-equal match (timestamps stripped). Adding a new event shape or parsing behavior MUST come with a fixture that pins it down. The same fixtures will gate Phase-3 verification of the Node CLI on Android once a Kotlin or instrumented test harness is added.

## Multi-Model Provider Seam (Phase 0 — youcoded PR #115, 2026-07-10)

Governing docs: `docs/superpowers/specs/2026-07-09-platform-vision-roadmap.md`, `2026-07-10-phase0-foundations-design.md`, ADRs 006–010 in `docs/decisions/`.

- **`SessionProvider` is `'claude' | 'native'`. The `'gemini'` provider is GONE** (Google discontinued the Gemini CLI June 2026; Gemini models return via the native runtime through OpenRouter/direct key in Phase 1). Don't reintroduce a gemini branch, toggle, or PTY command.
- **`'native'` has NO runtime yet.** `SessionManager.createSession` throws loudly for any non-claude provider — deliberate guard so a stray native create (e.g. from a remote client payload) fails instead of spawning a broken PTY. Phase 1 branches BEFORE the PTY worker spawn.
- **`window.claude.native.supported` is the ONLY gate for the dormant native UI** — no settings flag (decided 2026-07-10). preload computes it from `YOUCODED_NATIVE=1` (shell env at launch — run-dev.sh does NOT set it); remote-shim hardcodes `false`; shape parity pinned in `ipc-channels.test.ts` ("native runtime capability parity"). It is a plain boolean, NOT an IPC channel — no ipc-handlers/SessionService.kt row exists on purpose.
- **Sandboxed preload DOES have `process.env`.** Verified empirically 2026-07-10 against the built preload in a production-identical sandboxed window; `desktop/CLAUDE.md`'s old "no process.env" claim was corrected. The sandbox blocks module imports, not Electron's polyfilled `process`.
- **Runtime selector** (`Claude Code | YouCoded`) renders in the new-session form ONLY when `native.supported`; the YouCoded option is disabled until Phase 1; `handleCreate` is hardwired to `'claude'` — wiring the selector state into create is a Phase 1 task.
- **Reasoning segments are dormant on the CC path — don't "light them up" casually.** `assistant-thinking` transcript events WITH `data.text` dispatch `TRANSCRIPT_ASSISTANT_REASONING` (per-token deltas merged by `partId` — UNLIKE the text path, which appends whole blocks); payload-less events remain heartbeats. CC's transcript-watcher emits `data: {}`, so no CC user sees the disclosure today. If you ever make the watcher emit thinking text, that needs a parity-fixture update AND a UX decision first. **App.tsx and BubbleFeed.tsx MUST use the identical predicate** (`event.data?.text` truthiness) — divergence puts main and buddy windows out of sync.
- **Phase 1 must route chat-view PTY send paths through the harness for native sessions.** Cataloged in PR #115's body: InputBar send, `guardedPtySend`, ESC interrupt, `useSubmitConfirmation` retry, ChatView Ink-menu keys, ToolCard permission keys, TrustGate, BubbleFeed send; also gate the Shift+Tab permission-cycle handler; subagent reasoning needs `parentAgentToolUseId` routing in the reasoning reducer case.
- **`feat/opencode-mvp` is ARCHIVED, not dead** — `OPENCODE-MVP-ARCHIVED.md` on that branch lists what Phase 1/2 should mine (event-translation patterns, subprocess supervision + tests, OllamaDetector, capability probe). Do not merge it; do not delete it.

## Chat Reducer

- **`toolCalls` Map is never cleared.** ToolCards need old results for display. Use `activeTurnToolIds` (a Set) for current-turn status checks, not the full Map.
- **Always use `endTurn()` helper when adding turn-ending code paths.** Don't manually clear `isThinking`, `streamingText`, `currentTurnId`, `attentionState`, etc. — `endTurn()` handles all of them consistently. `SESSION_PROCESS_EXITED` is the one handler that spreads `endTurn()` and then overrides `attentionState: 'session-died'`.
- **Dedup uses the `pending` flag on user timeline entries.** `USER_PROMPT` always appends with `pending: true`. `TRANSCRIPT_USER_MESSAGE` finds the oldest matching pending entry and clears the flag; if none matches, appends a new `pending: false` entry. Replaces the prior last-10-entries content match, which silently dropped legitimate rapid-fire duplicates. Don't "simplify" back to content match.
- **`attentionState` is classifier-driven, not timer-driven.** The 30s `thinkingTimedOut` flag is gone. `useAttentionClassifier` reads the xterm buffer every 1s and dispatches `ATTENTION_STATE_CHANGED`; transcript events + `PERMISSION_REQUEST` reset it to `'ok'`. **Classifier patterns in `attention-classifier.ts` are Claude Code CLI-version sensitive** — regex drift is the most likely source of banner false positives/negatives. Keep the version-anchor comment current and treat it like a dated fixture.
- **`AttentionState` is `'ok' | 'stuck' | 'session-died' | 'error'` — four reachable states.** Writers: `'stuck'` ← the PTY classifier; `'session-died'` ← `SESSION_PROCESS_EXITED`; `'error'` ← `NATIVE_SESSION_ERROR` (dispatched from `'session-error'` transcript events, **native sessions only** — CC sessions never emit `session-error`, so a CC user never enters `'error'`). `NATIVE_SESSION_ERROR` follows the same spread-`endTurn()`-then-override pattern as `SESSION_PROCESS_EXITED`, additionally stashing the provider message on `SessionChatState.errorMessage` for the `AttentionBanner`. History note: `'awaiting-input' | 'shell-idle' | 'error'` were all deleted in the 2026-04-26 audit as writer-less dead branches; `'error'` was reintroduced in Phase 1 Plan A (2026-07) **with** the `NATIVE_SESSION_ERROR` writer. If you reintroduce any other state, wire up its dispatcher too — adding a state to the union without a writer puts dead branches back in the `AttentionBanner` switch.
- **Spinner classifier matches glyph + gerund + ellipsis ONLY — no seconds counter, no "esc to interrupt".** CC v2.1.119 emits `<glyph> Gerund…` with no suffix. The previous regex required `(Ns · esc to interrupt)` and silently failed on every real turn → the no-spinner-20s escalation flashed the wrong "Still waiting on Claude" banner during every long thinking turn. Active-vs-stalled is now decided by **glyph rotation** (same glyph for ≥30s = stalled, threshold raised from 10s after empirical capture showed CC holds the same glyph for 10–20s during silent thinking). To verify on a CC bump, run `node test-conpty/test-spinner-fullcapture.mjs` and `node test-conpty/test-attention-states.mjs` — both are wired into `cc-dependencies.md` "PTY spinner regex" and capture the live behavior. As of 2026-04-30 (commit `669fc86d`) the classifier ALSO matches a parallel paren-wrapped `(Nm Ns · …)` counter via `COUNTER_RE`; an advancing counter is an OR liveness signal independent of glyph rotation. The hook layer feeds `previousCounterSeconds` back through `ClassifierContext` so the no-signal-20s escalation is gated on both glyph absence AND counter non-advancement. `SPINNER_RE` was also widened from `[A-Za-z]+…` to `[A-Za-z][A-Za-z +\-]*…` to match multi-word gerunds like "Installing + verifying on device…" — needed by the CC tool-running variant captured 2026-04-30.
- **`SPINNER_RE` is anchored to start of line — DO NOT remove the `^`.** Without the anchor, the false-match probe (`test-conpty/test-attention-false-match.mjs` using `@xterm/headless` for production-accurate buffer rendering) confirmed the classifier matches Claude's response text containing `* Loading…`-style markdown bullets, echoed user prompts containing literal spinner glyphs (`❯ Output…: ✻ Pondering…`), and assistant turns prefixed with `●`. Real CC spinner lines always have the glyph at column 0 with no leading content; the `^` anchor preserves that exclusively. The pattern STOPS at `…`, so it still matches the hook-execution variant `✶ Channelling… (running stop hook · 3s · ↓ 1 tokens)` — verified live. If you tighten the regex further (e.g., requiring trailing whitespace), the hook variant will silently stop matching.
- **`readNewLines` must isolate each emit in try/catch.** `session.offset` advances before the emit loop, so a throwing listener aborting the loop strands every subsequent chunk in the batch (next `readNewLines` reads from the advanced offset forward). This was a root cause of "rare Claude message not appearing." Keep the per-emit try/catch in `transcript-watcher.ts` — do not "clean up" to a batch-level wrapper that loses the per-event isolation.
- **`readNewLines` is SERIALIZED per session (`reading` flag + coalesced rerun) — don't remove the guard.** (2026-07-10, both `transcript-watcher.ts` and `subagent-watcher.ts`.) The read is async (stat → open → read) and is triggered concurrently by fs.watch bursts, the global poll, and manual calls. Un-serialized, two overlapping reads consumed the same byte range (duplicate user bubbles, tool cards flapping back to `running`) and — because the read POSITION argument was re-evaluated after the first read advanced the offset while `bytesRead` was ignored — a short/empty second read decoded its zero-filled buffer into the partial-line carry, wedging NUL bytes in and silently dropping the next message at `JSON.parse`. The other root cause of "rare missing Claude message." Tests pin it in `transcript-watcher.test.ts → read integrity`.
- **The incomplete-line carry is BYTES (`partialBytes: Buffer`), not a decoded string.** A string carry decodes each half of a multi-byte UTF-8 char to U+FFFD independently — emoji/CJK split across a read boundary garbled permanently. Stitch bytes before decoding. Don't "simplify" back to `partialLine: string`.
- **`getHistory` replay dedups by uuid with the SAME semantics as the live path** (skip repeated `assistant-text`, first write wins; tool-use/result/turn-complete still emit). Comments used to claim a renderer-side dedup existed — it never did; the reducer appends duplicates. If replay semantics change, change the live path in the same commit.
- **`usePromptDetector` reads `getVisibleScreenText` (screen + margin), NOT the full scrollback; the classifier's IPC eval passes a 120-row tail.** Serializing the whole 1000+-row buffer per rAF flush was the top renderer CPU cost while streaming. The tail walk-back in `terminal-registry.getScreenText` never starts mid-wrapped-line — keep it. Side effect that is now load-bearing: menus that scrolled into scrollback can no longer shadow a live Ink menu.
- **SubagentWatcher polls are slow (5s) safety nets by design — the fast paths are event-driven.** `TranscriptWatcher` calls `kickScan()` when a parent Agent tool_use lands (instant discovery of the subagents dir/new files) and `settleByParent()` when the parent's tool-result lands (final read, then the per-file stat poll stops; fs.watch stays attached). Don't speed the polls back up "for responsiveness" — that regresses the idle-CPU accumulation this fixed — and don't remove the kick/settle calls in `transcript-watcher.ts`, which are what make slow polls safe.
- **`<local-command-stdout>` / `<local-command-stderr>` are STRIPPED ENTIRELY in `stripSystemTags` — DO NOT switch back to unwrapping.** CC writes these as dimmed (`[2m...[22m`) status echoes of slash-command output. After every `/compact` it writes a follow-up user-type line whose content is `<local-command-stdout>[2mCompacted (ctrl+o to see full summary)[22m</local-command-stdout>`. Unwrapping let CC's echo reach the reducer's `TRANSCRIPT_USER_MESSAGE` "no pending match" path, which **both** (a) appended a fake user bubble "Compacted (ctrl+o to see full summary)" AND (b) set `isThinking: true` with no transcript turn to ever clear it — leaving chat permanently stuck thinking after every compaction. Both the TS and Kotlin parsers strip these tags entirely now; regression tests in `transcript-watcher.test.ts` pin the exact JSONL fixture line. The terminal pane already shows raw CC stdout for users who want to see slash-command output. If you ever consider re-enabling unwrap for a different slash command's output, route it through a new event type — don't reintroduce the user-message path.
- **The `compact-summary` transcript event carries the full summary text in `data.summary`.** Used by `SystemMarker.tsx` to render click-to-expand on the otherwise-thin "Compacted · freed X tokens" divider. Both `App.tsx` (main window) and `BubbleFeed.tsx` (buddy window) forward `event.data.summary` into the `COMPACTION_COMPLETE` action; the reducer stores it on the `SystemMarker.summary` field. Aborted/watchdog completions don't carry a summary — keep the `expandable = !!marker.summary` gate in the renderer so the chevron + "view summary" affordance only appears when there's something to expand.

## Android Runtime

- **`LD_LIBRARY_PATH` is mandatory. DO NOT remove.** Termux binaries are relocated from `/data/data/com.termux/files/usr` to `context.filesDir/usr`; `DT_RUNPATH` baked into binaries is stale.
- **All binaries route through `/system/bin/linker64`** for SELinux W^X bypass. Three layers with distinct roles: LD_PRELOAD for C/Rust, claude-wrapper.js for runtime quirks, linker64-env.sh for Go binaries.
- **`TMPDIR` points to `$HOME/.cache/tmpdir`, not `$HOME/tmp`.** The specific path avoids Termux Node.js's compiled-in `/tmp` rewriting double-applying. Don't "fix" it to `$HOME/tmp`.
- **Use the linker variant of `termux-exec`.** Bootstrap copies `libtermux-exec-linker-ld-preload.so` over the primary `.so` after install.
- **Runtime fixes must work in BOTH `PtyBridge` and `DirectShellBridge`.** Both share `Bootstrap.buildRuntimeEnv()` and `Bootstrap.deployBashEnv()` — fix once, applies everywhere.
- **DO NOT poll `isRunning`.** Use the reactive `sessionFinished` `StateFlow` (fed by JNI waitpid thread).
- **`claude-wrapper.js` canonical source is `app/src/main/assets/claude-wrapper.js`.** Edit this file directly. It's deployed to `~/.claude-mobile/` at every launch.
- **`~/.claude-mobile/exec-wrappers/*` must be chmod 0755, not 0700.** Java's default `File.setExecutable(true)` gives 0700 under Android's 0077 umask. Those wrappers are exec'd by subprocesses spawned from Go binaries (notably `gh` spawning `git` during `gh auth setup-git`). At 0700 the shebang interpretation path via `/system/bin/sh` fails with EACCES even though the child runs as the same app uid — shebang exec is stricter than a direct `linker64` invocation. Fix: call `setReadable(true, false)` + `setExecutable(true, false)` in `Bootstrap.deployBashEnv()`. Wider perms stay inside the app sandbox (uid-isolated by Android at `/data/data/<pkg>/`) so nothing new is exposed. Don't "tighten" back to 0700.
- **Git HTTPS auth uses `~/.netrc`, NOT `gh auth setup-git`.** Go's raw-syscall exec breaks `gh auth setup-git` on Android (it tries to configure git's credential helper subsystem, which re-spawns processes that can't traverse the exec-wrapper path). Instead, the OAuth token from `~/.config/gh/hosts.yml` is mirrored into `~/.netrc` as `machine github.com login x-access-token password <token>` (mode 0600). libcurl reads `.netrc` natively, so `git push` over HTTPS needs no credential helper exec. Two places keep `.netrc` current: `Bootstrap.syncGhTokenToNetrc()` runs at session-start, and the `gh` bash wrapper in `linker64-env.sh` runs `_youcoded_sync_gh_netrc` after any `gh auth login/logout/refresh/token/switch`. If you add a new gh-auth-changing command, add it to the post-hook's case list so mid-session auth changes propagate without a session restart. Do NOT reintroduce a call to `gh auth setup-git` anywhere in setup flows — it will fail silently or with EACCES.
- **`gh auth login --web` polling is flaky — retry once if it dies with "error connecting to github.com".** The device-flow long-poll Go HTTP call sometimes errors out mid-poll on Android even when `curl` to github.com works fine. Root cause is most likely Go's HTTP/2 client behavior on Android's network stack (not YouCoded's runtime). Observed 1-of-3 success rate in the wild. Treat it as a known transient; don't paper over it with a retry wrapper in `gh()` (would double-prompt the user with a new device code on retry). If this gets worse, consider `GODEBUG=http2client=0` at the env level to force HTTP/1.1 polling.

## Vendored Termux terminal-emulator

- **`terminal-emulator-vendored/` is pinned to Termux v0.118.1 with a single documented patch** (a `RawByteListener` hook on `TerminalEmulator.append()`). `VENDORED.md` in that directory is the source of truth. Never edit files in this module outside the documented patch — if a new concern needs more, revisit the decision to vendor.
- **The vendored emulator is now headless** as of Tier 2 (xterm-in-WebView). The native Termux `TerminalView` Compose block was removed from `ChatScreen.kt` along with the `terminal-view:v0.118.1` Maven dep. `TerminalSession` still owns the PTY fork + JNI waitpid loop + `TerminalEmulator.append()`, and `RawByteListener` is the single tap point that feeds bytes to React xterm via `pty:raw-bytes`. Don't reintroduce a native render path — the vendored module exists solely to produce the byte stream now.
- **`RawByteListener` fires on the terminal thread** (same thread that calls `TerminalEmulator.append()`). Listener implementations MUST copy bytes before any async work — Termux reuses the same `byte[]` across PTY reads. `PtyBridge.rawByteFlow` uses `tryEmit` on a bounded `MutableSharedFlow` so a slow downstream consumer drops bytes rather than blocking the terminal thread.
- **`pty:raw-bytes` payload is base64-encoded.** JSON can't carry raw binary, UTF-8 corrupts high-bit bytes (common in ANSI control sequences), and base64 over localhost WebSocket has negligible cost. Never change the encoding without updating `raw-byte-listener-contract.test.ts` and every consumer at once.
- **Classifier fixtures at `youcoded/shared-fixtures/attention-classifier/` are the contract.** Adding a `BufferClass` or tweaking a regex in `classifyBuffer` requires a fixture change in the same commit. `attention-classifier-parity.test.ts` enforces this.
- **`pty:raw-bytes` is full three-surface parity now.** Tier 2 added the desktop consumer surfaces — `preload.ts` (no-op stub for shape parity), `remote-shim.ts` (real per-session dispatch), and the `usePtyRawBytes` hook in the renderer. The `ipc-channels.test.ts` parity describe asserts presence in `preload.ts`, `remote-shim.ts`, and `SessionService.kt` (no `ipc-handlers.ts` entry — it's a push event, not a request-response handler).
- **xterm display-only on touch.** `TerminalView` passes `disableStdin: true` to the Terminal constructor on touch platforms (Android + remote browser) and skips the `terminal.onData → sendInput` registration. Typing flows through the existing `InputBar` minimal-mode `<textarea>` instead. Don't reintroduce xterm-side input on touch — it would re-expose the historical xterm.js mobile IME issues (xterm.js #2403) and conflict with InputBar.
- **Single-finger touch scroll is custom JS, not native.** `TerminalView` registers capture-phase `touchstart`/`touchmove`/`touchend` handlers on the xterm container that route to `terminal.scrollLines()`. xterm's default mouse logic (which would interpret touch-drag as text selection) is bypassed via `e.preventDefault()` in `touchmove`. Trade-off: text selection on touch is unavailable. Don't try to "fix" by removing the handlers — scroll is more important than selection on mobile and chat-view doesn't have touch selection either.
- **xterm scrollback can show duplicated TUI chrome.** Claude Code (Ink) redraws its full TUI on certain events; when the cumulative TUI exceeds visible rows, older content scrolls into xterm's scrollback as new content arrives. This is Claude-Code-emission + xterm-standard behavior — there's no xterm option to discard programmatically-scrolled content (alt-screen mode would lose ALL scrollback). Mitigation deferred; bumping `scrollback` to 5000+ would let conversation history coexist with banner duplicates.

## Bundled Plugins & Hooks (`youcoded-core`)

`youcoded-core/` is a bundled Claude Code plugin contributed alongside `wecoded-themes-plugin` and `wecoded-marketplace-publisher`. It's currently being deprecated (see `docs/superpowers/plans/2026-04-21-deprecate-youcoded-core.md`) — `write-guard.sh` is moving into the app natively and the repo will be archived after release N+1. Prefer fixing bugs over adding features here; new functionality belongs in the app or in a dedicated marketplace plugin.

- **Never edit hooks in `settings.json` directly — update `youcoded-core/hooks/hooks-manifest.json`.** The desktop app's `HookReconciler` merges the manifest into settings.json at launch and after install/update. Direct edits get overwritten.
- **`.sh` files MUST have execute bit set.** On Windows, Git does not set this automatically. Use `git update-index --chmod=+x path/to/file.sh`. Missing execute bit is the #1 cause of "hook does nothing" bugs.
- **`config.json` is portable, `config.local.json` is machine-specific.** config.local.json is rebuilt every session by session-start.sh — don't sync it.
- **Feature work requires `git worktree add`**, not `git checkout -b`, in the plugin dir. `worktree-guard.sh` blocks branch switches.
- **The plugin is flat — there is no `core/`, `life/`, or `productivity/` subdirectory.** Phase 3 (commits `d54bbf9` / `0d5ca0a`) removed the former three-layer decomposition. Skills that used to live under those layers (journal, encyclopedia, task inbox, theme-builder, skill-creator, google services) now ship as independent marketplace plugins.

## Sync Warnings

- **`~/.claude/.sync-warnings.json` is the authoritative sync warning file.** It holds a typed `SyncWarning[]` array consumed by SyncPanel, StatusBar chips, and the gear-icon red dot. The legacy string-code `~/.claude/.sync-warnings` is still written by bash `statusline.sh` for the terminal statusline — the desktop app no longer writes or reads that file.
- **Two writers, non-overlapping codes.** `runHealthCheck()` owns `OFFLINE`, `PERSONAL_NOT_CONFIGURED`, `PERSONAL_STALE`, `SKILLS_UNROUTED`, `PROJECTS_UNSYNCED`. Push methods (`pushDrive`/`pushGithub`/`pushiCloud`) own per-backend codes (`CONFIG_MISSING`, `AUTH_EXPIRED`, `QUOTA_EXCEEDED`, `NETWORK`, `RCLONE_MISSING`, `UNKNOWN`). Every push-path warning has a `backendId`; health-check ones don't. If you add a new code, pick an owner — the health-check merge replaces only its own codes on every run, so a push-path-owned code must never appear in `runHealthCheck`.
- **`.sync-error-<backendId>` is retired.** The old per-backend error file format is deleted on startup by `SyncService.cleanupStaleBackendErrorFiles()`. Do not reintroduce that file. All per-backend error state now lives in `.sync-warnings.json` entries keyed by `backendId`.
- **Push-failure warnings are non-dismissible.** `dismissWarning()` enforces `dismissible: false` server-side — the UI's dismiss button is hidden for those, but even if it weren't, the server rejects the call. That's deliberate: silencing "Google Drive isn't connected" risks data loss. Don't add a force-dismiss path.
- **Remove a backend → clear its warnings.** `removeBackend()` calls `clearWarningsByBackend(id)` after removing the backend from config. Skipping this leaves a phantom warning with no way to dismiss (the backend it pointed to is gone).
- **Classifier stderr patterns are version-sensitive.** `sync-error-classifier.ts` uses rclone-version-sensitive substring matches. When rclone changes its error messages (major version bumps), patterns can silently drop to `UNKNOWN`. The test file is the source of truth for what stderr each code matches — keep it alongside any pattern change.
- **Node-killed timeouts produce empty stderr — use `extractStderr()` from `sync-error-classifier.ts`.** When `execFileAsync(..., { timeout })` fires, Node SIGTERMs the child before it can flush stderr, so the rejection has empty `e.stderr` and only the wrapper's `Command failed: <cmd>` in `e.message`. Naively returning `e.stderr || e.message` makes every timeout classify as `UNKNOWN` — which is exactly what happened to Drive backups when slug sizes outgrew the original 60s `RCLONE_TIMEOUT` (the youcoded-dev slug grew past 156 MB). The `extractStderr(e, timeoutMs)` helper checks `e.killed && e.signal` and prepends a `TIMEOUT_SENTINEL` so the `TIMEOUT` pattern in the classifier matches. **Don't reintroduce raw `e.stderr || e.message` in `rclone()` / `gitExec()` / future helpers** — wire them through `extractStderr` and pass the timeout used.

## Resume Browser & Conversation Identity

Invariants from the 2026-06-12 missing-sessions investigation (221 conversations deleted by CC's default cleanup; 627 topic files mass-rebumped; phantom index ids; flags landing on pre-/clear session ids). All shipped in the `fix/resume-browser-reliability` branch.

- **Topic-file mtime IS the conversation-index's `lastActive` source.** `updateConversationIndex()` upserts an entry whenever the topic file's mtime is newer than the stored `lastActive`. Consequence: anything that (re)writes topic files MUST preserve the original timestamp (`fs.utimesSync` to the entry's lastActive) — `regenerateTopicCache()` is the canonical writer and does this. Writing with "now" mtimes creates a feedback loop that bumps EVERY session's lastActive each cycle, so the 30-day index prune never fires and the recent-50 backup pull fetches the wrong sessions. Observed in the wild before the fix: all 627 topic files ≤2 days old, dozens of entries sharing one `lastActive` second.
- **`sessionIdMap` remaps ONLY on `SessionStart` hook events.** CC rotates its session id mid-PTY on `/clear` (verified 2026-06-12; `--resume` does NOT rotate, `/compact` rewrites the same file). The map follows the rotation so close-time flags and topic lookups land on the live conversation — but the remap gate must stay SessionStart-only: subagent/tool hook events can carry child session ids and would poison the map. The remap also assumes the rotated transcript starts EMPTY; see the INVARIANT comment in the ipc-handlers listener and `docs/cc-dependencies.md → Session-id rotation semantics`.
- **Resume Browser name precedence: topic file > conversation-index topic > derived-from-first-user-message > "Untitled".** The derived title (`readTranscriptMeta`, mirrored in Kotlin `SessionBrowser.readTranscriptMeta`) and content-timestamp ordering exist because (a) the auto-title hook is PostToolUse-gated — chat-only sessions never fire it — and depends on model compliance, and (b) sync restores clobber file mtimes, so `stat.mtimeMs` lies about recency. Don't "simplify" ordering back to mtime or drop the head/tail parse. Reads are bounded (256KB head only when untitled / 64KB tail) so 100MB transcripts stay cheap. On Android the final precedence is applied in `SessionService`'s `session:browse` handler (the Kotlin lister carries `derivedTitle` separately); the IPC payload shape is unchanged.
- **Literal `Untitled` topic-file content is a placeholder.** Both desktop `readTopic` and Kotlin `SessionBrowser` normalize it (alongside blank and `New Session`) so the index/derived fallback chain applies. Older clients synced such files; treating the literal string as a real name silently bypasses both fallbacks.
- **Conversation-index keys are CC session UUIDs — `SESSION_UUID_RE` gates the topic scan.** The auto-title flow has the in-session model hand-type `echo "Title" > topics/topic-<id>`; a typo'd id used to create phantom index entries pointing at no transcript (seen in the wild: `3f3a5cccc-…`, nine c's). The scan guard blocks CREATION from malformed ids (existing entries may still be updated — `setSessionFlag` legitimately seeds flag-bearing entries before a topic file exists), and the prune loop self-heals malformed entries that carry no flags. **Flagged malformed entries are kept** — deleting a user's tag is worse than carrying a dead row. If you ever key index entries by anything other than a CC UUID, revisit this guard first.
- **`cleanupPeriodDays` is seeded (365) when absent — never overwritten.** CC's built-in default is 30 days and silently deletes transcripts; the Resume Browser lists from those files. `retention-default.ts` (desktop) and `RetentionDefault.kt` (Android) seed only when the key is missing, respect explicit user values, and never rewrite a corrupt settings.json (it carries hooks + enabledPlugins). One-time healing note: after the regenerate-mtime fix, index entries whose lastActive was artificially bumped will correctly age out of the index — that's the fix working, not data loss (their transcripts were already gone).

### Slug→path resolution for resume (2026-07-12, two-device dogfood)

A CC project slug is the cwd with every path separator flattened to `-`, so reversing a slug to a path is AMBIGUOUS when a folder's own name contains a hyphen (`C--Users-desti-youcoded-dev` = `…\desti\youcoded-dev` OR `…\desti\youcoded\dev`). This is the direction that resume depends on.

- **`walkSlugParts` (`session-browser.ts`) MUST try the LONGEST leading segment first — do NOT revert to shortest-first.** Shortest-first greedily descends into a stray shorter sibling dir: with `C:\Users\desti\youcoded` present, `C--Users-desti-youcoded-dev` resolved to the nonexistent `…\youcoded\dev`, and `session-manager` then silently launched `claude --resume` in `$HOME` → **every hyphenated-folder session "resumed from the home directory."** Longest-first picks the real `youcoded-dev`. Pinned by `desktop/tests/slug-path-resolution.test.ts`.
- **Store-backed rows override the slug-walk with the store's exact-basename resolution** (`resolveLocalProject` in `conversations/resolve-local-project.ts`, shared by the materialize sweep AND the Resume Browser via `buildLocalProjectResolver`), but only when the resolved local folder actually holds the transcript. This is unambiguous (no slug guessing) and makes materialize + resume agree on every OS. Cross-OS store-only rows resolve by `projectName` basename too (originalPath is the FOREIGN device's path and won't exist locally) — pinned by the cross-OS test in `session-browser.test.ts` + the pure `resolve-local-project.test.ts`.
- **`session-manager.ts`'s `existsSync(cwd) ? cwd : os.homedir()` is a SILENT MASK — it now warns on a resume with a missing cwd.** Behavior is unchanged (a new session with no cwd still defaults to home), but the warning exists so a future upstream path-resolution regression is visible instead of resurfacing as a silent wrong-directory resume. Don't remove the warn; better still, gate resume on a resolvable row upstream (the Resume Browser already does).
- **ANDROID MIRRORS THE ORIGINAL (PRE-FIX) BUG — all three failure modes are still live there (deferred with Android sync, Phase 3).** `app/.../runtime/SessionBrowser.kt`'s `walkSlugParts` is still SHORTEST-first (its doc comment even wrongly claims longest-first); the `session:create` bridge handler in `SessionService.kt` **drops `resumeSessionId`** (so Android "resume" launches a FRESH session) and passes the raw cwd to the PTY with no existence check; there is NO `resolveLocalProject`/store overlay on Android; and `SessionBrowser.listPastSessions` does NOT skip the legacy home-slug aggregation symlinks (so a foreign-only session resolves to `$HOME`). `SessionRegistry.resumeSession` has the CORRECT cwd derivation but is dead code (no caller). When Android resume/sync is picked up, port: longest-first `walkSlugParts`, thread `resumeSessionId` through the bridge + a cwd existence guard, and add a store/basename resolver. See the two-device-dogfood findings.

## PTY Writes (Desktop + Android)

Writing programmatically into Claude Code's TUI input bar exposes a stack of behaviors that have to be respected, none of them documented contracts. The submit logic lives in `desktop/src/main/pty-worker.js` (case `'input'`) and `app/.../runtime/PtyBridge.kt` (`writeInput`). A renderer-side safety net is `desktop/src/renderer/hooks/useSubmitConfirmation.ts`.

### The mechanism (April 2026 audit)

- **Paste classification is LENGTH-GATED.** Empirical bisection against CC v2.1.119 (`test-conpty/cc-snapshot.mjs`) showed: an atomic 63-byte write (62-char body + `\r`) submits cleanly, an atomic 64-byte write (63-char body + `\r`) leaves the body in the input bar with literal `\n` + cursor on the next line — i.e. `\r` was absorbed as paste content. The threshold is **exactly 64 bytes** for CC v2.1.119 (likely a buffer-size constant in Ink); future versions may shift it. The bisected value lives in `youcoded/desktop/test-conpty/snapshots/cc-<version>.json` — re-run `node test-conpty/cc-snapshot.mjs` on each CC bump and diff against the prior snapshot. The previous PITFALLS claim that "any write ≥2 chars is paste" was wrong on detail; see `docs/superpowers/investigations/2026-04-24-chat-to-pty-submit-reliability.md` Phase 10.
- **CC echoes typed bytes back through stdout** (Ink re-renders the input bar). Cold-start delay is ~6.7 s on Windows; warm-session delay is sub-second. This is universal TUI behavior but also a CC-internal contract — see `docs/cc-dependencies.md` "PTY input-bar echo" entry.
- **Windows ConPTY drops bytes on large single writes** (>~600). Bracketed-paste markers don't survive ConPTY at all (verified by `test-conpty/harness.mjs` Phase 8). Both constraints are addressed by chunking the body in writes ≤56 bytes.

### The submit protocol (current, post-April-2026 redesign)

The worker handles three input shapes deterministically. No more 600 ms timing guesses on the Windows path:

1. **Passthrough** — text doesn't end in `\r`. Single write. (Single-byte ESC, raw escapes, in-progress typing.)
2. **Atomic submit** — text ends in `\r` AND total length ≤ `SAFE_ATOMIC_LEN` (56 bytes; 8-byte margin under the threshold). One write; `\r` is unambiguously a keystroke regardless of how the kernel reads it. Common case for short chat messages.
3. **Echo-driven submit** (desktop) / **600 ms split** (Android) — text ends in `\r` AND total length > 56. Chunk the body in 56-byte pieces, then send `\r` separately.
   - **Desktop:** wait for the body's tail to echo back from CC's stdout (proving CC has drained the body bytes from its input pipe), then write `\r` as a single byte. `\r` is guaranteed below the paste threshold AND guaranteed to arrive in a fresh kernel read. No timing assumption. On echo timeout (`ECHO_TIMEOUT_MS`, 12 s) the worker **suppresses the CR entirely** — no echo usually means CC isn't showing an input bar that accepted the body (a live Ink select menu has focus), and a blind `\r` would answer that menu. Recovery for genuinely lost sends is the renderer-side `useSubmitConfirmation` retry, which is gated on `canRetrySubmit()`. Do NOT reintroduce the blind fallback CR (2026-07-09 stray-Enter fix, youcoded#110).
   - **Android:** still uses the 600 ms gap. Linux PTY doesn't have ConPTY's gap-collapse issue, so the timing is reliable. Mirroring desktop's echo-driven approach is a planned follow-up; see TODO in `PtyBridge.writeInput`.

### Invariants

- **Don't reintroduce a 600 ms enter-split in the desktop worker.** Echo observation makes the gap superfluous and timing-fragile under busy state. The current `pty-worker.js` handles all three paths; do not "simplify" back to the timing-based version.
- **Don't atomic-write any `body + \r` longer than 56 bytes.** Even if it would happen to fit under the threshold for a given CC version, the constants are version-coupled and can shift downward. Always route through the echo-driven path for >56-byte sends.
- **Don't reintroduce bracketed-paste markers** (`\x1b[200~...\x1b[201~`) on Windows. Verified mangled by ConPTY (markers stripped or asymmetrically preserved). Independent of the length issue.
- **`useSubmitConfirmation` is the second-line defense, not the primary fix.** With echo-driven send, primary failure modes are absent for short messages and rare for long ones. The renderer-side retry catches the residual lost-send case — sends a bare `\r` when `pending` stays set 8 s after submit AND `canRetrySubmit()` (in `state/pty-input-gate.ts`) passes: `attentionState === 'ok'`, no awaiting-approval or running current-turn tools, no in-flight assistant turn, no uncompleted interactive prompt. `attentionState === 'ok'` ALONE is not an idle signal — it's also the normal state mid-turn and while a permission/AskUserQuestion menu is up (both `thinking-active` and `unknown` buffer classes map to `'ok'`); gating on it alone was how the retry auto-answered prompts. Don't gate retry on `!isThinking`: `isThinking` is set on `USER_PROMPT` and only cleared by `endTurn()`, which never fires if CC never received the message — so that gate is broken for the very state it's supposed to recover from. (See `useSubmitConfirmation.ts` and `pty-input-gate.ts` comments.)
- **CC's Ink select menu is LIVE in the PTY while a hook permission card is up — never write to the PTY programmatically during a pending interaction.** The chat card (permission / AskUserQuestion / plan approval) answers via the hook socket, but until it does, the terminal-side menu still consumes keystrokes: a bare `\r` selects the highlighted option (auto-answering the question or auto-approving the permission), and typed text is eaten as menu input. Every automated writer must consult `state/pty-input-gate.ts` (`hasPendingInteraction` / `canRetrySubmit`) or, in the main process, `HookRelay.hasPendingPermission(sessionId)` — that's how `useSubmitConfirmation`, InputBar chat sends (refused with a toast, draft kept), and `SessionManager.broadcastReloadPlugins` (per-session deferral with bounded retries) are gated. Deliberate menu-driving writes (ToolCard plan-approval arrow keys, TrustGate buttons, terminal-view xterm keystrokes) intentionally bypass the gate. Fixed 2026-07-09 (youcoded#110) after AskUserQuestion prompts were observed answering themselves.
- **The optimistic USER_PROMPT bubble and the PTY send must derive from ONE sanitized string** (`components/outgoing-message.ts`). The transcript confirms the bubble by EXACT content match; the PTY send replaces newlines with spaces. When the bubble kept its newlines, every multiline (Shift+Enter) message stayed `pending` forever and armed a guaranteed stray retry `\r`. Don't rebuild either string inline in InputBar.

## PTY Resize (Desktop Windows)

- **`TerminalView.fitAndSync` must dedup on unchanged cols/rows before calling the PTY resize IPC.** Windows node-pty uses ConPTY, which reflows its visible buffer on every resize and re-emits the repainted contents to the terminal. The ResizeObserver fires for any layout shift (font load, sibling resize, 1-pixel container jitter) and `fitAddon.proposeDimensions()` frequently returns the same cell grid across ticks. Without dedup, each spurious resize causes ConPTY to re-emit Claude Code's Ink-rendered UI (banner + input bar) into xterm scrollback — so scrolling back through terminal history shows the same chunk repeated between every user message. The dedup lives as a closure (`lastCols`/`lastRows`) inside the TerminalView mount effect; keep it there, not in `session-manager.resizeSession` — the renderer owns "what dimensions should the PTY be" and main-side dedup invites confusion about ownership.
- **Android is not affected.** Termux's Linux PTY doesn't reflow on SIGWINCH. The dedup still runs harmlessly on Android; it's a pure perf win there.

## Remote Access State Sync

- **Remote clients receive chat state via `chat:hydrate` on connect — don't add parallel replay paths.** On new WebSocket auth, `remote-server.ts::replayBuffers()` calls `requestChatSnapshot(webContents)` to ask the renderer (`RemoteSnapshotExporter`) for a serialized `ChatState` snapshot, then pushes it as a single `chat:hydrate` WebSocket message to the connecting client only. The old `transcriptBuffers` replay buffer was removed because two sources of truth create ordering and dedup bugs. If you need to backfill new state types for remotes, extend `serializeChatState` / `deserializeChatState` in `chat-types.ts` — don't reintroduce a sidecar buffer.
- **`attentionState` is authoritative on desktop only — remote browsers get it via `attentionMap` in `status:data`.** `useAttentionClassifier` reads the xterm PTY buffer every 1s and runs ONLY on Electron. Remote browsers have no PTY access and MUST NOT run their own classifier (CLI-version-sensitive regex in `attention-classifier.ts` would drift across two sites). Flow: desktop reducer diff → `useRemoteAttentionSync` fires `remote:attention-changed` IPC → main caches in `lastAttentionBySession` → `buildStatusData()` folds `attentionMap` into `status:data` → broadcast on change (not just the 10s timer). Remote shim diffs `attentionMap` vs `prevAttentionRef` before dispatching `ATTENTION_STATE_CHANGED` — the diff is load-bearing; without it every 10s tick would thrash the reducer.
- **`RemoteSnapshotExporter` is Electron-only by design.** It's mounted in `App.tsx` inside `ChatProvider` and guards on `typeof window.claude.onChatExportSnapshot === 'function'`. On remote browsers that API doesn't exist (remote-shim doesn't expose it), so the exporter short-circuits — this is intentional, not a parity bug.
- **`chat:export-snapshot` has a 2s timeout.** `requestChatSnapshot()` resolves with `{ sessions: [] }` on timeout. Combined with the 500ms PTY-replay delay in `replayBuffers`, worst-case window before a new remote sees PTY output is ~2500ms when the renderer is unresponsive. If you shorten the timeout, the degenerate case (renderer still booting at connect) hydrates empty and remote falls back to live events.

## Build-Type Parity (Android)

Debug and release Android builds are NOT equivalent. The release flavor enables R8 minification (`isMinifyEnabled = true` in `app/build.gradle.kts`); debug skips it. Anything that depends on R8-stable behavior works in `assembleDebug` and silently breaks in `assembleRelease`.

- **Don't use string-based reflection against your own code.** `getMethod("name")`, `getDeclaredMethod`, `Class.forName`, `KClass`, `KFunction`, `::class.declaredMembers` — R8 obfuscates the target's name and the lookup throws. The `PluginInstaller.buildEnv()` bug (commit `912f5ca7`, 2026-04-30) was this exact pattern: `bootstrap.javaClass.getMethod("buildRuntimeEnv")` returned a NoSuchMethodException in release, the silent `try/catch` fallback shipped a stripped env without `LD_PRELOAD`, and every marketplace install died with `cannot exec 'remote-https': Permission denied`. Bug shipped from 2026-03-25 (`e18ab861`, R8 first enabled) through v1.2.2 because every dev/CI build was debug.
- **Direct calls always; reflection only when unavoidable.** When unavoidable (third-party libs, generic IPC), add an explicit `-keep` rule in `app/proguard-rules.pro` for the targeted class/methods. Don't rely on the `try { reflection } catch { fallback }` pattern — silent fallbacks mask R8-induced failures.
- **`Bootstrap` has a defensive `-keep` rule** (`app/proguard-rules.pro`) so future code that re-introduces reflection against it can't silently break. Costs negligible APK size for one class. Don't remove it without an audit confirming nothing reflects against `Bootstrap`.
- **`./gradlew :app:assembleReleaseTest` is the parity check.** Same R8/shrinker/proguard config as production release, signed with the debug keystore, installs side-by-side as `com.youcoded.app.releasetest` ("YouCoded ReleaseTest", bridge port 9961). Runs automatically on every push via `android-ci.yml` and on demand via `android-test-build.yml`. Run it locally before tagging if you've touched code that involves reflection, annotation processing, or any sensitive symbol-name dependency.
- **Don't rely on the GitHub runner image for `node`.** Android workflows now `setup-node@v4` explicitly so `bundleWebUi` (which shells to `npm` via `scripts/build-web-ui.sh`) doesn't depend on whatever ubuntu-latest happens to ship.

## Prerequisite Installer (First-Run)

The desktop installer's first-run flow lives in `youcoded/desktop/src/main/prerequisite-installer.ts` and `first-run.ts`. A real-user `spawn EINVAL` on a clean Windows 11 machine motivated the rules below; don't regress.

- **Always spawn `.cmd`/`.bat` via `runCommand`, not raw `execFile`.** Node 18.20.2+ / 20.12.2+ / 21.7.1+ ship the CVE-2024-27980 mitigation, which makes `child_process.spawn` / `execFile` REFUSE to launch `.cmd` / `.bat` files on Windows unless `shell: true` is passed — failure surfaces as `Error: spawn EINVAL`. The `runCommand` helper at the top of `prerequisite-installer.ts` auto-flips `shell: true` when `process.platform === 'win32'` AND the resolved path matches `/\.(cmd|bat)$/i`. Real `.exe` paths take the no-shell route so the no-shell-injection guarantee is preserved for those. Adding a new `execFile` call in this file? Route it through `runCommand`. Adding a new file that spawns Node-CLI shims (npm, gh, claude-when-installed-via-npm, etc.) on Windows? Replicate the helper or set `shell: true` conditionally on the same condition.
- **`installClaude` uses Anthropic's native installer, not `npm i -g`.** Switching to `claude.ai/install.ps1` (Windows) / `claude.ai/install.sh` (POSIX) eliminates the `.cmd` shim chain entirely — `npm.cmd` during install AND `claude.cmd` during every subsequent `--version` / `auth status` / `auth set-key` call. The bootstrap is two-stage (downloads the per-version binary from `downloads.claude.ai`, runs `<binary> install` to register PATH). Don't reintroduce the npm path "for backwards compat" — every `.cmd` we add re-exposes EINVAL plus the latent paths the wrapper covers.
- **Native installer URL is CC-coupled — see `youcoded/docs/cc-dependencies.md` "Native installer bootstrap script (Desktop)".** If Anthropic moves distribution off `claude.ai/install.{ps1,sh}` (winget-only, store-only, custom URL scheme), or makes the bootstrap interactive, `installClaude` breaks. Refresh the cc-dependencies entry on each CC version review.
- **Post-install detection failure means "PATH didn't propagate" — surface the message, don't loop.** When `detectClaude` reports not-installed after a successful native-installer run, the binary is on disk but its directory hasn't propagated to the running Electron process's PATH (Windows: HKCU registry update; POSIX: shell rc file change). `installClaude` returns a clear "Quit and reopen YouCoded" error rather than retrying. Don't add a polling loop or a forced PATH-rebuild — restart is the deterministic fix and the only one that's correct.
- **Android still installs Claude Code via npm.** Per `youcoded/docs/cc-dependencies.md` "npm package entry-point layout (Android)" — the Android runtime depends on `lib/node_modules/@anthropic-ai/claude-code/cli.js`, currently pinned to CC 2.1.112. Migrating Android to the native binary is a separate, larger refactor (the linker64 + claude-wrapper.js exec path needs the Node-CLI shape). Don't accidentally switch `Bootstrap.installClaudeCode` on Android to the bootstrap script — the desktop and Android paths intentionally diverge here.
- **`getRegPath()` returns a QUOTED path; `getPowerShellPath()` returns UNQUOTED — do not "unify" them.** Both helpers in `prerequisite-installer.ts` resolve to an absolute `%SystemRoot%\System32\...` path so the installer doesn't depend on a clean PATH. But they're consumed differently: `getRegPath` is interpolated into a `execSync(\`${reg} query ...\`)` template that routes through `cmd.exe`, which parses the embedded quotes correctly. `getPowerShellPath` is passed to `runCommand(ps, args)` → `execFile(ps, args, { shell: false })`, which hands the literal string to `CreateProcess` as the application path — embedded quotes become part of the filename and you get `ENOENT spawn "C:\\...\\powershell.exe"`. Verified empirically 2026-05-21 (the bug shipped briefly in the first PATH-hardening pass). Rule: if a helper returns a path that will be used as the `file` arg of `execFile` / `spawn` with `shell: false`, return it UNQUOTED. Quote only when the path will be interpolated into a shell-parsed string.
- **`detectWinget()` is the upfront guard for winget-dependent installs — wire new winget callsites through it.** `winget.exe` is an MSIX App Execution Alias, not a guaranteed-present Win32 binary. It can be absent on Windows Server, older LTSC builds, sandboxed environments, or machines where "App Installer" was disabled by policy. Bare `winget` invocation in those cases surfaces as cryptic `spawn ENOENT`. `detectWinget()` probes `winget --version` upfront and returns an actionable error message ("install App Installer from the Microsoft Store"). Current callers: `installNode`, `installGit` (in `prerequisite-installer.ts`), `RemoteConfig.installTailscale` (in `remote-config.ts`), `installRclone` (in `sync-setup-handlers.ts`). Any new winget-dependent install flow must run `detectWinget()` first — absent-winget is a real user state and hardcoded paths don't solve it.

## Framed Shell & Chrome Glass

The chat region renders inside `.framed-shell` (in `desktop/src/renderer/styles/globals.css`). When the artifact drawer is open the shell holds `[frame-edge] [chat-pane] [frame-divider] [drawer-pane] [frame-edge]`. The visible "chrome" (panel-colored frame around the chat + drawer windows) is painted by a **single** absolute element — `<div class="chrome-glass">` mounted at the top of the chat container in `App.tsx` — clipped to a rounded donut shape via `clip-path: polygon()`.

- **One backdrop-filter, ever.** Prior to the May 2026 chrome-glass refactor, HeaderBar / frame-edge / frame-divider / drawer-pane / corner-pseudos each carried their own translucent panel + backdrop-filter. At non-100% zoom, the element boundaries land at fractional screen pixels and the anti-aliased seams created either tiny overlaps (darker double-blurred regions) or tiny gaps (sharp wallpaper bleed-through). `chrome-glass` consolidates the entire frame chrome into one element with one backdrop-filter — no inter-element compositing seams. Do NOT reintroduce per-element backdrop-filters on HeaderBar / InputBar / StatusBar / frame-edge / frame-divider / drawer-pane in framed-chrome mode.
- **`destination-out` is NOT a valid `mix-blend-mode` value.** It's a Porter-Duff *compositing* operator, available in Canvas / SVG `<feComposite>`, but the CSS `mix-blend-mode` whitelist is `normal | multiply | screen | overlay | darken | lighten | color-dodge | color-burn | hard-light | soft-light | difference | exclusion | hue | saturation | color | luminosity | plus-darker | plus-lighter`. An earlier chrome-glass draft used `mix-blend-mode: destination-out` cutouts; the browser silently ignored the invalid value and rendered the cutout's `background: black` straight through, painting the chat area black. Use `clip-path` if you need to cut a shape out of an element.
- **Compound attribute selectors must be on the same element.** `data-wallpaper` is set on `<html>`; `data-chrome-style` / `data-input-style` are set on `<body>`. `[data-wallpaper][data-chrome-style='floating']` requires both attributes on the SAME element — it never matches anything. Use `[data-wallpaper] [data-chrome-style='floating']` (descendant) instead. This bit Devil's Garden's header glassmorphism — opaque rather than translucent because the wallpaper-translucent rule never fired.
- **`chrome-glass` is hidden in floating-chrome modes.** When `chrome-style='floating'` (full floating, e.g. halftone-dimension) OR `input-style='floating'` (partial floating, e.g. devils-garden), `chrome-glass` is `display: none` and HeaderBar / InputBar / StatusBar paint their own pills/strips. Adding a new "always-floating" element? Mirror the same gating so chrome-glass doesn't double up with it.
- **The TR corner of the chat-pane cutout polygon needs an explicit closer.** The `clip-path: polygon()` builds the chrome donut as outer-CW + inner-CCW with the inner rounded rectangle approximated by 5 points per corner. The other three corners share their endpoint with the start of the next side; TR's endpoint is the same as the FIRST inner point at the top of the polygon list. Without an explicit `(R-corner, T)` repeated at the end of the polygon, the implicit close jumps straight from the last TR intermediate to `(0, 0)` and the TR arc visually distorts. Keep the trailing closer point.
- **drawer-pane sits ABOVE chrome-glass via z-index: 11.** chrome-glass at z-10 only cuts out the chat-pane area (single polygon cutout). drawer-pane has its own `--inset` background and `border-radius` and renders on top of chrome-glass in the chrome region. The corner cutoffs from `border-radius` reveal chrome-glass underneath — that's where the rounded corner visual comes from. Adding a second cutout to the polygon for drawer would also work but at the cost of ~20 polygon points; z-index is cheaper.
- **`.chrome-wrapper` must be transparent regardless of theme.** The Tailwind `bg-canvas` class on HeaderBar / bottom chrome wrappers (in App.tsx) was the right answer pre-chrome-glass. With chrome-glass providing the surface, the bottom wrapper at `z-index: 20` (from `.bottom-float`) sits ABOVE chrome-glass at z-10 — if it paints `--canvas`, the bottom strip reads canvas-colored while the rest of the chrome reads panel-colored, visibly different. Universal `.chrome-wrapper { background-color: transparent !important; }` covers solid and wallpaper themes alike.
- **The framed-shell's right slot holds EITHER the artifact drawer OR the games panel — width is the `--right-pane-width` CSS var.** As of June 2026 the multiplayer games panel renders inside the framed-shell as a `.drawer-pane game-pane` (same chrome as the artifact drawer) instead of a full-height `border-l` slide-out. `.drawer-pane` width and `chrome-glass--drawer-open`'s `--chat-right-offset` BOTH read `var(--right-pane-width, 480px)`; App sets it on the main-area div (`400px` when the games panel is the active right pane, else `480px`). If you hardcode the pane width in either place again, the frosted-chrome cutout and the actual pane width drift apart at non-default widths. The games panel and artifact drawer are **mutually exclusive** (two transition-gated effects in App close the other on open) and `chrome-glass--drawer-open` is gated on `activeDrawerOpen || gameState.panelOpen`. ChatView receives the games panel via a `gamePane` slot prop (only the ACTIVE session's ChatView gets it) and renders it with precedence over the artifact drawer. Caveat: the games panel only shows in chat view (it lives in ChatView), not terminal view — a behavior change from the old always-visible slide-out.

## Theme Color Contrast Rules

Themes must meet contrast thresholds enforced by `wecoded-themes/scripts/audit-contrast.mjs` (also wired into the `validate-theme.yml` CI workflow, non-blocking for now). The same script lives in dev under `desktop/scripts/audit-theme-contrast.mjs` for local audits including the four built-in themes.

- **`panel` vs `canvas` must be ≥1.07:1.** Both are surface colors — chrome (panel) wraps content (canvas). Below 1.07:1 the two surfaces blur together visually. Crème originally had 1.046:1 — the panel and canvas were effectively identical and the frame disappeared.
- **Text contrast on canvas:**
  - `fg` / `fg-2`: ≥4.5:1 (WCAG AA normal text)
  - `fg-dim`: ≥3:1 (AA large text)
  - `fg-muted`: ≥3:1 (used for small/secondary text — must be readable)
  - `fg-faint`: ≥1.8:1 (intentionally decorative, but still recognizable)
- **User bubble text:** `on-accent` vs `accent` must be ≥4.5:1 (AA normal text). Strawberry Kitty's original accent `#D94E6B` against white `on-accent` was 4.0:1 — slightly below AA; fixed by darkening accent to `#CC4060` (4.7:1).
- **chat-pane bg == drawer-pane bg.** Both should render as `--canvas` so the two "windows" read as one content surface inside the chrome. drawer-pane briefly used `--inset` during the chrome-glass refactor — looked like a tonally different window. Change drawer-pane bg in the SAME edit you change chat-pane bg; the audit doesn't catch this mismatch on its own.

## Theme Marketplace Entries

- **`ThemeRegistryEntryWithStatus.isLocal` is distinct from `installed`.** `installed: true` means a `manifest.json` exists at `~/.claude/wecoded-themes/<slug>/`. `isLocal: true` means the entry was synthesized in `desktop/src/main/local-theme-synthesizer.ts` because no marketplace registry entry exists for that slug — it's user-built (typically via `/theme-builder`) and never been published. A marketplace-installed theme has `installed: true, isLocal: undefined`. Code that branches on "is this user-deletable forever vs. reinstallable" must read `isLocal`, NOT `installed`. The deletion-confirmation copy in `MarketplaceDetailOverlay.tsx::ThemeBody.handleUninstall` is the canonical reader; the "Local" badge in `MarketplaceCard.tsx` is the other.

The merge order `[...marketplaceEntries, ...synthesized]` is locked by `desktop/tests/local-theme-synthesizer.test.ts` — don't reorder without updating the test, and don't change "marketplace wins" without considering downstream consumers (the publisher PR-flow assumes the published version is canonical).
- **Synthesizer is pure; the provider does I/O.** `local-theme-synthesizer.ts` MUST stay free of `fs`/`path`/`os` imports — that's how it stays unit-testable without mocking. All filesystem reads (manifest.json + preview.png existence check) happen in `theme-marketplace-provider.ts::listThemes()`. Don't bleed I/O into the synthesizer "for convenience."

Local user themes are invisible in the Android Library tab. Pre-existing gap, not addressed by Phase 1 of the local-themes-in-library plan; tracked in `docs/knowledge-debt.md`.

## Plugin Installation & Claude Code Registries

- **Claude Code v2.1+ does NOT filesystem-scan `~/.claude/plugins/`.** Its plugin loader (`GD_` in the binary) iterates `enabledPlugins` from `settings.json` only. Dropping files into `~/.claude/plugins/<id>/` without writing the four registries leaves the plugin invisible to the CLI — `/reload-plugins` will report "0 new plugins."
- **Four registries must be written for a plugin to load.** `ClaudeCodeRegistry` (`src/main/claude-code-registry.ts`) handles all four atomically: (1) `~/.claude/settings.json` → `enabledPlugins["id@youcoded"]: true`, (2) `~/.claude/plugins/installed_plugins.json` → v2 entry with absolute `installPath`, (3) `~/.claude/plugins/known_marketplaces.json` → marketplace source config, (4) `~/.claude/plugins/marketplaces/youcoded/.claude-plugin/marketplace.json` → plugin manifest list. Always call `registerPluginInstall()` / `unregisterPluginInstall()` — don't write these files by hand.
- **Plugin install location is `~/.claude/plugins/marketplaces/youcoded/plugins/<id>/`.** The marketplace subtree lives under the plugin cache dir (`~/.claude/plugins/`), NOT directly under `~/.claude/`. The non-cache loader (`t71`) computes the path as `<marketplaceInstallLocation>/<source>` and errors if that directory doesn't exist. Two exceptions sit OUTSIDE the marketplace subtree: (a) the bundled `youcoded-core` plugin at `~/.claude/plugins/youcoded-core/` (cloned by `youcoded-core/bootstrap/install.sh`, not via the plugin installer — being deprecated), and (b) legacy top-level installs for other Claude-Code-installed plugins.
- **Reconcilers must scan both roots.** `listInstalledPluginDirs()` in `claude-code-registry.ts` enumerates top-level children of `~/.claude/plugins/` (for the `youcoded-core` clone) AND direct children of `YOUCODED_PLUGINS_DIR` (for marketplace-installed packages). Scanning only one misses half the installed plugins — this was the decomposition-v3 reconciler bug.
- **`installed_plugins.json` lives at `~/.claude/plugins/installed_plugins.json`** (inside the plugin cache dir). Historical commentary said `~/.claude/installed_plugins.json`; that was the pre-v2.1 location and is wrong for current Claude Code.
- **Bundled plugin list is two-way duplicated.** `BUNDLED_PLUGIN_IDS` lives in `desktop/src/shared/bundled-plugins.ts` (imported by both main and renderer) and `app/src/main/kotlin/com/youcoded/app/skills/BundledPlugins.kt`. Both must stay in sync. The list is intentionally hardcoded, not marketplace-driven: (a) offline-first launches can't fetch a remote list, and (b) a remote config would grant the marketplace repo authority to force-install plugins on every client. Changing the list requires an app release. Also update this PITFALLS entry when the list changes.

## MCP Plugin Authoring (Cross-Platform)

Captured from the spotify-services real-world install pass on Windows (2026-04-26). Six independent footguns surfaced in sequence — every commit `941f9b4..e152dcc` on `feat/spotify-services` in `wecoded-marketplace` is a real-world fix. If you're shipping a stdio MCP server in a marketplace plugin, read every entry below before submission.

- **`bash` is NOT on the Windows system PATH.** Git for Windows installs only `C:\Program Files\Git\cmd\` (which has `git.exe`) to the system PATH. `bash.exe` lives at `C:\Program Files\Git\usr\bin\` and is only on PATH inside an interactive Git Bash session. Don't write `command: "bash"` in `mcp-manifest.json` — it works on macOS but the Windows spawn fails silently with "command not found." Use a real binary that IS on PATH (e.g. `node`, `uvx`, `python`) when possible, or invoke bash by absolute path.
- **Spaces in `command` paths break under Claude Code's spawn (assumed `shell:true`).** `C:\Program Files\Git\usr\bin\bash.exe` gets word-split by `cmd.exe` at the first space → `'C:\Program' is not recognized`. The fix is to use the Windows 8.3 short name: `C:\PROGRA~1\Git\usr\bin\bash.exe`. Verify the short name on the dev box with `powershell -Command "(New-Object -ComObject Scripting.FileSystemObject).GetFile('<path>').ShortPath"`. The 8.3 short name has been stable across every Windows version since NTFS — safe to bake into the manifest.
- **MSYS-style `/c/Users/...` paths only work as args, never as `command`.** `bash.exe` translates them internally; Windows kernel `CreateProcess` and `cmd.exe` don't. So a `command` field MUST be a Windows-native path (`C:\...` or `C:/...`); MSYS paths are fine in `args` because bash gets to translate them after spawn.
- **`${PACKAGE_DIR}` is expanded by YouCoded's reconciler, not by Claude Code.** `mcp-manifest.json` can use `${PACKAGE_DIR}/...` placeholders, but they only resolve when the YouCoded desktop app's `reconcileMcp()` writes the resolved entry to `~/.claude.json` on app launch. If a user runs Claude Code as a CLI outside YouCoded, the placeholder reaches `spawn()` literally and `/mcp` reports `Missing environment variables: PACKAGE_DIR`. Don't ship plugins that depend on this expansion path for non-YouCoded users; bake an absolute path or a real-on-PATH command into the manifest instead.
- **MCP SDK 0.x → 1.x is a breaking change (dict → Pydantic).** `mcp` 0.x accepted plain dicts from `@server.list_tools()` and `@server.call_tool()`. mcp 1.x requires `mcp.types.Tool` and `mcp.types.TextContent` Pydantic objects, accessed via attribute (`tool.name`). If `pyproject.toml` pins `mcp>=0.9.0` (unbounded), pip silently pulls 1.x and `tools/list` crashes with `'dict' object has no attribute 'name'`. **Pin `mcp>=1.0.0,<2.0.0`** explicitly. Same risk applies to any other Python SDK with a major-version Pydantic migration.
- **`claude mcp list` is a liar — it only verifies `initialize`.** The CLI's connection check stops after the first JSON-RPC step. The in-session `/mcp` host runs the full handshake including `tools/list`. A server can show `✓ Connected` in the CLI and `✗ Failed` in `/mcp` simultaneously when `tools/list` is broken. Always verify with `/mcp` (or a manual three-message Node spawn-probe — see `wecoded-marketplace/spotify-services/docs/plan.md` Phase 2 for one) when changing the tool registry.
- **`PYTHONIOENCODING=utf-8` is required for any Python that prints Unicode under Git Bash.** Git Bash's stdout to Python defaults to cp1252; Python 3.13 raises `UnicodeEncodeError` on the first ✓/✗ glyph and exits before any work happens. Set it at the bash invocation: `PYTHONIOENCODING=utf-8 python script.py`. No-op on macOS/Linux. The two heredocs in `spotify-services/setup/{ingest-oauth,smoke-test}.sh` do this.
- **pywinrt namespace packages don't transitively pull each other.** Each WinRT namespace ships as a separate PyPI package: `winrt-Windows.Media.Control`, `winrt-Windows.Foundation`, `winrt-Windows.Foundation.Collections`, etc. Installing one does NOT pull others. If your code calls a method that returns an `IVectorView` (e.g. `sessions.get_sessions()`), you also need the Collections package. Pip resolution doesn't catch the gap because the imports defer to runtime; a smoke-test on real Windows hardware does.
- **Don't gate setup scripts on system Python.** `python3` resolves to a Microsoft Store stub on Windows that prints an install hint and exits non-zero. A prereq check like `! python3 -c '...'` falsely fails for users with `uv`-managed Python (which is the recommended path). Either drop the system-Python check entirely (`uv venv --python X.Y` fetches the version on demand) or also try `py` and `python` as fallbacks. The spotify-services install script went with the former.
- **`rsync` is not on Git Bash's PATH.** Use `tar | tar` pipe or `cp -r` for portable file-copy in setup scripts. Apply excludes consistently (`.venv`, `__pycache__`, `*.egg-info`, `.pytest_cache`).
- **Don't source the venv `activate` script in MCP launchers.** The activate script calls coreutils (`basename`, `dirname`) at `/usr/bin/`. Claude Code's MCP spawn passes a stripped PATH that omits `/usr/bin`, so activate fails with `basename: command not found` and python never starts. Symptom: `/mcp` shows ✗ Failed but the same launcher run from an interactive shell works. Fix: skip activate entirely — call the venv's python by absolute path (`$VENV/Scripts/python.exe` on Windows, `$VENV/bin/python` on Unix) and set `VIRTUAL_ENV` manually for any library that introspects it. Reference: spotify-services launcher.sh in `wecoded-marketplace`.
- **Test the MCP handshake end-to-end before submission.** Unit tests on the tool handlers don't catch protocol-layer issues (Tool/TextContent shapes, SDK version skew, stdio framing). Write a Node-based probe that spawns the server with the actual command/args from `mcp-manifest.json`, sends `initialize` + `notifications/initialized` + `tools/list`, and asserts all three succeed. The reference probe is `wecoded-marketplace-spotify/<temp>/spawn-tools-list.js` from the spotify-services PR.

## Header Bar

The left and right `flex-1` columns must stay symmetric — both omit `min-w-0`. If you need to truncate something inside, put `min-w-0` on the individual child, not the flex parent.

Do not reintroduce `@media`, `hidden sm:`, or `window.innerWidth` checks in header children — they lie when the app window is narrow but the viewport is wide, which is the default state on desktop.

The gamepad pill and caption buttons stay in the right cluster on all platforms — only the chat/terminal toggle moves.

## Keyboard Routing

- **ESC handling flows through `useEscClose` stack → chat-passthrough guard.** The stack's capture-phase listener (in `EscCloseProvider`, mounted at App root) pops the top overlay and calls `preventDefault()`; the App-level bubble-phase listener reads `defaultPrevented` to decide whether to forward `\x1b` to the active session's PTY. Do NOT add parallel window-level ESC listeners on overlays — they break LIFO ordering and can race the chat-passthrough. Exception: **xterm's terminal view** still forwards ESC natively via node-pty when the terminal pane is focused; the chat-passthrough listener explicitly returns when `viewMode === 'terminal'` to avoid a double-send. The buddy window has its own React root and its own `window`, so its ESC handler does not compete with the main window's.
- **Chat-to-PTY interrupt is single-byte — no chunking applies.** `window.claude.session.sendInput(sessionId, '\x1b')` sends 1 byte. Single-byte writes are well below the empirically-observed paste-classification threshold (see "PTY Writes" above) and reach Ink as a fresh keystroke regardless of timing. Don't mistakenly wrap the single ESC in the paste-splitter workaround.
- **Interrupt markers in the transcript end the turn in the reducer.** `transcript-watcher.ts` detects `[Request interrupted by user]` and `[Request interrupted by user for tool use]` in user messages (exact match only) and emits `user-interrupt` events instead of `user-message`. The reducer's `TRANSCRIPT_INTERRUPT` handler attaches `stopReason: 'interrupted'` to the in-flight turn and calls `endTurn(session, 'Turn interrupted')`. Don't "simplify" either by removing the interception — the marker would then render as a user bubble and in-flight tools would stay `running` forever.

## Cloudflare Workers (Marketplace Backend)

Gotchas discovered shipping `wecoded-marketplace/worker/` — the Cloudflare Worker backing marketplace install counts and ratings.

- **Never put a key in both `[vars]` and `wrangler secret put` with the same name.** On every `wrangler deploy`, the var from wrangler.toml wins and silently clobbers the secret. Symptom: OAuth breaks in prod because `GH_CLIENT_ID` ends up as the literal placeholder string from wrangler.toml instead of the real secret. **Fix:** leave the key entirely out of wrangler.toml — define it ONLY as a secret. The missing-var pattern is load-bearing; don't "helpfully" add a placeholder.
- **In CI, `wrangler deploy` must run BEFORE `wrangler secret put`.** Deploy reconciles the Worker's bindings to wrangler.toml, removing stale vars left by earlier deploys. If a stale var with the same name exists when you try `secret put`, Cloudflare errors with "Binding name already in use" (code 10053). The safe order is: `migrations apply --remote` → `deploy` → `secret put`. Don't swap deploy to last.
- **`[env.test]` wrangler section is required to run tests against miniflare with `@cloudflare/vitest-pool-workers` 0.5.x.** It can't resolve wrapped bindings like `[ai]`; the test env must omit AI entirely. Consequence: any code path that depends on `env.AI` must fail open or be test-covered another way. The `test/setup.ts` trick of mutating `env` does NOT propagate to `c.env` inside the running worker — set test vars via `[env.test.vars]` in wrangler.toml instead.
- **Rate limits via the Cache API are per-colo, not global.** `checkRateLimit()` uses `caches.open("rl")` which is edge-local. For casual abuse prevention this is fine; for true global throttling you'd need a Durable Object or D1-backed counter.

## Announcements

- **Source of truth is `youcoded/announcements.txt`** (app repo), not `youcoded-core`. Ownership moved in the 2026-04 rebuild. `/announce` writes to the app repo; pushing to `youcoded-core` does nothing. Public URL: `https://raw.githubusercontent.com/itsdestin/youcoded/master/announcements.txt`.
- **Single fetcher per platform.** Desktop runs `announcement-service.ts` (Electron main, 1h cadence); Android runs `AnnouncementService.kt` launched from `SessionService` (also 1h). Both write `~/.claude/.announcement-cache.json` in their respective home dirs. **Never reintroduce a parallel fetcher to the same cache file** — the legacy `desktop/hook-scripts/announcement-fetch.js` + its 6h `setInterval` spawner in `ipc-handlers.ts` were removed because two writers mean redundant work and divergent freshness. Both platforms now share a single 1h cadence owned by the app's main process.
- **Two expiry filters, both required.** Fetch-time in the parser (drops past-date lines before writing cache) and render-time in `StatusBar.tsx` via the shared `isExpired()` helper at `desktop/src/shared/announcement.ts`. Removing the fetch-time filter lets stale entries enter the cache; removing the render-time filter lets entries linger past local midnight until the next fetch. Keep both.
- **Clear propagation is an explicit null-write.** When the remote `announcements.txt` is empty (or all lines are expired/comments), `fetchAnnouncement` writes `{ message: null, fetched_at }` to the cache rather than skipping the write. The prior behavior (`return` without touching disk) let cleared announcements linger for up to a full refresh interval because the stale cache entry was never overwritten. If you "simplify" back to a no-op-on-empty write, clears won't propagate to the status bar within the refresh interval.
- **Cache shape contract:** `{ message: string | null, fetched_at: string, expires?: string }`. `message: null` means explicitly cleared; missing cache file means never-fetched. Both result in no pill; distinction matters only for diagnostics.
- **Android writes its own cache in its own `~/.claude`.** Android's `~/.claude` lives inside the Termux env (`context.filesDir/home` / whatever `$HOME` resolves to under Bootstrap), separate from desktop's `~/.claude`. `SessionService.startStatusBroadcast()` reads that cache into the `status:data` WebSocket payload so the React status-bar widget receives it — mirror of desktop's `ipc-handlers.ts` `buildStatusData()`. If either writer goes missing, that platform shows no announcement even though the other one does.

## Settings → Development

- **Six new IPC types must stay in parity across `preload.ts`, `remote-shim.ts`, `ipc-handlers.ts`, and `SessionService.kt`:** `dev:log-tail`, `dev:summarize-issue`, `dev:submit-issue`, `dev:install-workspace`, `dev:install-progress`, `dev:open-session-in`. Phase 6 originally drifted to the old shape and shipped silently-empty issue bodies on Android — caught only by the next code review.

Do not "simplify" by always re-cloning — wastes ~500MB and breaks any in-progress local edits the user has. The `alreadyInstalled: true` flag in the response distinguishes pull from fresh clone for UI copy.

This reuses Claude Code's own OAuth token, avoids cracking open `~/.claude/.credentials.json`, and sidesteps the Windows ~32KB arg-length cap. Do not switch to passing the prompt as a positional arg — both desktop and Android use stdin. If you ever switch to direct `api.anthropic.com` calls, add a `cc-dependencies.md` entry — the credentials file format is a private CC contract.
- **`DevTools.runStreamed` writes stdin synchronously before reading stdout — fine for current callers (sub-64KB prompts), fragile for future callers.** The Linux pipe buffer is 64KB. If a future caller passes stdin larger than that, the write will block waiting for the child to drain its stdout, but no one is reading stdout yet — classic stdin/stdout deadlock. Current sole caller is the `claude -p` summarizer prompt, which is bounded upstream by `smartTruncateLog`. If you ever pass a large stdin blob, drain stdout on a separate thread while writing.
- **The plan-file-to-implementation drift in Phase 6 is the canonical cautionary tale.** Cross-platform tasks that batch desktop + android work need to port the *current* implementation across, not the *original plan*. Always re-read the up-to-date desktop signatures before writing the parallel Kotlin handler.

## Analytics

YouCoded ships opt-outable anonymous device-hash + DAU/MAU telemetry via the marketplace Worker (`wecoded-marketplace/worker/`). Specs: `docs/superpowers/specs/2026-04-23-privacy-analytics-design.md` (original install_id design, historical) and `docs/superpowers/specs/2026-05-01-device-id-analytics-design.md` (current device-hash redesign — read this first).

- **`blob2` is `device_id_hash`, not `install_id`.** Every client computes `HMAC_SHA256(SALT, machine_id || platform)` locally and sends only the 64-char hex hash. The raw machine_id never reaches the Worker. SALT is baked into the desktop and Android source — see `analytics-salt.ts` (desktop) and `Salt.kt` (Android), kept in sync.
- **Never log raw `device_id_hash` server-side outside of `count(DISTINCT)`.** The privacy-by-construction guarantee is that `blob2` appears only inside cardinality aggregates in admin SQL queries. Breaking this — even in a debug log — breaks the promise made in the in-app Privacy section (`AboutPopup.tsx`).
- **Only `lib/analytics.ts#writeAppEvent` may call `env.APP_ANALYTICS.writeDataPoint`.** Any new field must go through that wrapper so the blob order stays in sync with the SQL queries. The current blob order — `["heartbeat" (vestigial), deviceIdHash, appVersion, platform, os, country, region]` — is the contract; changing it silently breaks every admin query. blob1 is a constant `"heartbeat"` (single event type post-redesign); blob7 is `region` (ISO 3166-2 from `CF-IPRegionCode`).
- **One event type only.** The retired `install` event is gone — `/app/install` returns 410. "New installs per day" is derived at query time as `MIN(timestamp) GROUP BY blob2`. The discriminator slot (blob1) is kept as a constant for forward-compat.
- **`writeDataPoint` is fire-and-forget and silent on failure.** No receipt, no retry queue, no durability. Use D1 if you need durable, queryable-by-id data instead.
- **`[env.test]` deliberately omits `APP_ANALYTICS`.** `vitest-pool-workers` / miniflare can't resolve the binding in this toolchain version — same footgun as the `[ai]` binding. The `lib/analytics.ts` wrapper uses `env.APP_ANALYTICS?.writeDataPoint()` — the `?.` is load-bearing; don't remove it.
- **Mutating `env` in a test body does NOT propagate to `c.env` inside a worker request.** Test runtime snapshots env at start. Values that need to flow to admin route handlers (e.g. `CUTOVER_TIMESTAMP`, `KNOWN_DEV_DEVICES`) must be set in `[env.test.vars]` in `wrangler.toml`, not via `(env as any).X = ...`.
- **CF Analytics Engine SQL is a narrow subset, NOT full ClickHouse.** Quirks learned the hard way (422 responses): (1) cardinality is `count(DISTINCT col)` — `uniq()` and `COUNT_DISTINCT()` both raise "unknown function call"; (2) `INTERVAL '30' DAY` takes a QUOTED STRING literal — unquoted integers raise "Expected literal string, found: 30"; (3) the account ID + narrow-scope `CF_ANALYTICS_TOKEN` are distinct from the broader `CF_API_TOKEN` used by `wrangler deploy`. **Subqueries in `FROM` may or may not be supported** — the derived-installs query depends on this; validate via deploy smoke test before assuming it works.
- **`adminFilterClause` and `cutoverClause` are the SQL safety boundary.** `worker/src/lib/admin-filter.ts` exports two helpers that return `""` or a sanitized `AND ...` fragment. Hex-only and ISO-only sanitization is mandatory because we string-interpolate into SQL (AE has no parameter binding). A `KNOWN_DEV_DEVICES` secret holds Destin's own device hashes, filtered out of admin queries by default; `?include_admins=1` bypasses for debugging.
- **Admin auth has two paths via `requireAdminAuth`.** Existing cookie-flow session `Authorization: Bearer <token>` still works. The admin CLI skill in `youcoded-admin` authenticates via `X-GitHub-PAT: <gh auth token>`, which the Worker trades for a platform account id by resolving the GitHub `/user` numeric id through the `identities` table (cached 60s by SHA-256 hash in `auth/pat.ts`; requires the admin to have signed into the marketplace at least once). The admin allowlist check (`isAdminAccount()` in `auth/admin.ts`, keyed on bare GitHub numeric ids via `identities` — accounts Phase 1 rebuild, 2026-07-07) stays inline in each route so 401 (not auth'd) and 403 (auth'd but not admin) stay distinguishable. `ADMIN_USER_IDS` holds bare GitHub numeric ids (repo secret is named `MARKETPLACE_ADMIN_USER_IDS`), NOT `github:<id>` strings.
- **AE free-tier retention is 90 days.** Pre-cutover (install_id-keyed) rows age out automatically; the `CUTOVER_TIMESTAMP` env var explicitly excludes them from queries until they do. Privacy copy does not promise any specific retention — if a future copy change adds one, verify the current AE retention (or upgrade the plan) before shipping.
- **Worker JSON 500 via `app.onError`.** Non-HTTPException errors return `{ ok: false, error: <message> }` with status 500 (not Hono's default plain-text). Critical for debugging AE SQL failures — the response body carries the CF error. `HTTPException` responses (400/401/403/429) still emit their own status+body unchanged.
- **Client `device_id_hash` is stable across reinstalls.** Reinstalling YouCoded does NOT regenerate the hash because `~/.claude/youcoded-analytics.json` persists across uninstall/reinstall AND because the hash is derived from the machine's hardware ID, not from a UUID in that file. Wiping `~/.claude` only matters when machine_id read fails (rare; falls back to a persisted UUID). Don't add code that rotates the hash on launch — that would defeat the entire device-counting metric.
- **Country and region never sent from client.** The Worker reads `CF-IPCountry` and `CF-IPRegionCode` server-side. Region is ISO 3166-2 (e.g., `US-CA`). The privacy copy promises country + approximate region (state-level granularity); finer geography is out of scope.
- **No cross-tabulation of region with other dimensions.** Region + version + platform + OS could become a fingerprint at low cell counts. Single-dimension queries only — current six endpoints all GROUP BY exactly one dimension.

## Artifact Viewer

- **`canonicalize()` is the single source of truth for path equality.** All paths in sidecars and the central index are stored canonical. All comparisons (orphan detection, exclude match, dedup, click routing) go through `canonicalize()`. The function lives in TWO places: TypeScript at `desktop/src/shared/artifacts/canonicalize.ts` and Kotlin at `app/src/main/kotlin/com/youcoded/app/artifacts/PathCanonicalize.kt`. Both consume the same `shared-fixtures/artifacts/canonicalize-cases.json` for parity. Bypassing the canonicalizer is the most likely cause of "the path didn't match even though it should have."
- **`appendVersion` retries on CAS conflict — don't add a second retry loop.** `MAX_RETRIES = 5` is intentional. Increasing it doesn't help under sustained contention; surfacing a warning is the right escape valve. Same applies on the Kotlin side.
- **`casWrite` uses a mkdir-based lock (not just compare-and-set).** The TS implementation went through one revision: the initial CAS-then-rename pattern had a TOCTOU race where two writers could both pass CAS, then both rename — one overwrites the other (commit `2d44fc3d` initial → `a75492ea` fix). The Kotlin port (`CasWrite.kt`) was implemented with the lock from the start. Both use `<target>.lock` directory + 30s stale-lock breaking. Don't "simplify" the lock away — the bug WILL recur under concurrent writers.
- **`transcriptRef` is NOT in v1.** Earlier drafts pointed versions at CC's JSONL lines for historical content. Dropped because CC's transcript format is version-coupled (per the Transcript Parser pitfall) and CC compacts/rotates transcripts. v1 reads current content from disk only. v2's version-history dropdown will snapshot content into `<project>/.youcoded/content/<versionId>` at write time — under our control, not CC's. Don't reintroduce transcriptRef.
- **`bash mv` / `bash rm` produce zero artifact events.** The Tracker only subscribes to Write/Edit/Delete tool calls. Filesystem side-effects via Bash are opaque in v1. If you need rename tracking, that's a v2 task.
- **Session Drawer auto-tracks ALL files Claude touches; Project View filters externals.** Original spec said externals are never auto-tracked. **That rule changed during implementation.** The Session Drawer is a session activity log — every Write/Edit/Delete in the session shows up, with `kind: 'internal'` for files inside cwd and `kind: 'external'` (+ `absolutePath`) for files outside. The Project View filter (in `LIST_PROJECT` IPC handler) excludes external artifacts unless they appear in `manualIncludes`. So external auto-captures live in the sidecar but only surface in the per-session list, never in the project browser unless explicitly added. **Don't restore the old "externals never tracked" rule** — the Session Drawer becomes wildly confusing if Claude edits a file in the session and it doesn't show up.
- **Transcript events don't carry `cwd`.** The transcript-watcher emits `{ type, sessionId, uuid, timestamp, data }`. The Artifact Tracker resolves cwd by looking up the session in the `sessions` state via a `sessionsRef` (so the handler always sees the latest list without re-subscribing). Don't try to read `event.cwd` — it's always undefined and the handler will silently bail.
- **`artifacts:append-version` is the IPC that actually populates the sidecar.** `listSession` is read-only — it just filters the sidecar by sessionId. Without `appendVersion` being called by the renderer Tracker on every Write/Edit/MultiEdit tool-use event, the sidecar stays empty forever. The whole chain (Tracker → appendVersion → ensureProject → applyGitTreatment → broadcast `changed`) lives in the main-process handler. If artifacts mysteriously don't materialize from Claude's activity, this is the first thing to check.
- **The framed-shell visual chrome overlays the content region; the drawer-pane insets itself with margins to clear it.** YouCoded's HeaderBar (`.header-bar { position: absolute; top: 0; z-index: 20 }`) and bottom chrome (`.bottom-float`, also `z-index: 20`) overlay the content rather than being flex siblings. `.chat-scroll` compensates with `padding-top: var(--top-chrome-height)` / `padding-bottom: var(--bottom-chrome-height)` so messages scroll behind frosted chrome. The drawer-pane is a flex sibling that fills the full `.framed-shell` height, so on its own its top/bottom would slip behind the absolute chrome. **Resolved by the chrome-glass refactor:** `.framed-shell > .drawer-pane` carries `margin-top: var(--top-chrome-height, 3rem)` + `margin-bottom: var(--bottom-chrome-height, 5rem)` (globals.css), insetting the whole drawer into the gap between header and bottom chrome using the same vars the chat padding uses. Margins are on the drawer ONLY, not chat-pane, so the chat doesn't shift when the drawer opens — that chat-shift was the "jank" that killed the earlier Option C (padding-on-framed-shell) attempt, reverted 2026-05-22. **Don't move these insets onto `.framed-shell` or `.chat-pane`** — that reintroduces the layout shift. The historical Option B (restructure HeaderBar/StatusBar to flex siblings) is no longer needed for the drawer.
- **`.youcoded/` is auto-gitignored.** `ProjectManager.applyGitTreatment()` (both TS and Kotlin) appends `.youcoded/` to `.gitignore` on first sidecar materialization in a git repo. Privacy guarantee: agent file-touch history doesn't leak into commits by default. User can opt in to sharing via Settings → Privacy.
- **The drawer is layout-level, NOT an overlay.** Don't wrap `SessionDrawer` in `<OverlayPanel>`. It's a flex sibling of `ChatPanel` inside `.framed-shell`. The framed chrome (HeaderBar fill extends down the sides + divider) is the load-bearing visual — that's why it's layout-level.
- **`--frame-edge` is a real CSS variable (default 10px).** Theme authors can tune it. Setting it to 0 effectively hides the chrome edges (useful for fullscreen-feeling themes). Floating themes opt out via `data-theme-layout='floating'` on `<html>`, which is set from the active theme's `layout.frame-style` field (default `'framed'` until a theme opts in).
- **Android `artifacts:list-project` / `:list-all-files` / `:list-projects-index` / `:include-external` / `:exclude` / `:delete-project` / `:rename` / `:remove-record` are no-op stubs.** Mobile Project View is v2. The Kotlin handlers return `{ok: false, error: 'not-implemented-on-mobile'}`. Don't "fix the bug" of stubs returning errors — that's the contract. (`artifacts:read-binary`, `get`, `save`, `append-version`, `check-existence`, `list-session` are REAL on Android.)

The renderer-side `Artifact Tracker` (in `state/artifact-tracker.ts`) subscribes to TranscriptWatcher events alongside the chat reducer's subscribers. - **Drawer open/closed AND the selected artifact are per-session, keyed by `sessionId`.** `artifactState` holds `drawerOpenBySession: Record<sessionId, boolean>` and `activeArtifactBySession: Record<sessionId, string | null>` (NOT the old global `drawerOpen` / `activeArtifactId`). Fixed height + `shrink-0` children guarantees every card is the same size and always shows its preview + filename. Don't drop the fixed height "to let cards size to content."
- **The HeaderBar artifact-count badge excludes BOTH `status:'deleted'` AND orphaned (bash-rm'd) files.** Claude Code has no Delete tool, so files are almost always removed via `bash rm`, which produces no artifact event — the record stays `status:'active'` with the file gone from disk (an "orphan"). The badge runs its own `artifacts.checkExistence(projectRoot, liveIds)` (mirroring the drawer) and subtracts missing ids, re-checking when the list changes or the drawer toggles. `projectRoot` comes from the active session's `cwd`. Counting only `status !== 'deleted'` is NOT sufficient — it misses orphans, which is the common case.
- **`ContentFindBar` is reused for BOTH the artifact viewer and chat Ctrl+F — keep the highlight registries distinct.** It takes a `highlightName` prop (`'artifact-find'` default, `'chat-find'` for the chat timeline) because `CSS.highlights` is a single global registry; two bars sharing a name would clobber each other's ranges. ### Binary viewers, ExcelJS xlsx, open-externally, artifactify (2026-06-17, `feat/artifact-viewer`)

- **Binary viewers load bytes via the `artifacts:read-binary` IPC (base64), NEVER `fetch('file://…')`.** The renderer origin (http in dev, app/asset in prod/Android) can't fetch a `file://` URL — it fails silently. The shared `useArtifactBytes(absolutePath)` hook calls `read-binary` → `atob` → `Uint8Array`. `XlsxView`/`DocxView`/`PdfView` consume the bytes; `ImageView` builds a `blob:` URL (NOT `<img src="file://…">`). The IPC is full 4-surface parity (preload/remote-shim/ipc-handlers/`SessionService.kt` base64 read) — pinned by the artifact-parity describe in `ipc-channels.test.ts`. Remote works because the shim routes it to the host over WS.
- **The lazy viewers are now actually MOUNTED.** Previously `ActiveArtifactView` fell back to `BinaryFallback` for every `{lazy}` registry entry (pdf/docx/xlsx never rendered). `RendererRegistry` now wraps them in `React.lazy` (real components) and `ActiveArtifactView` renders inside a `<Suspense>`. `BinaryFallback` is only for truly unknown types and its "Open externally" uses `shell.openPath` (not the nonexistent `platform.openExternal`).
- **xlsx viewer is ExcelJS, not SheetJS — SheetJS (`xlsx`) was REMOVED.** SheetJS's free build can't read cell styles (fills/fonts/borders = Pro only); ExcelJS reads them free, so it's the fidelity path. `exceljs-cell.ts` (pure) maps a cell → display text + CSS (fills, fonts, borders, alignment, number formats, merges, col widths); `XlsxView` renders a DOM grid with A/B/C + row-number gutters, gridlines, a formula bar (reveals `=…`), bottom sheet tabs, click-to-select, and is **padded to ≥50 rows × 26 cols** (`MIN_ROWS`/`MIN_COLS`) of empty gridlined cells so it reads like a real sheet, not a table in whitespace. Don't reintroduce `sheet_to_html` (no gutters/styling) or add SheetJS back.
- **Formula cells must be evaluated — openpyxl/pandas write formulas WITHOUT cached results.** ExcelJS gives `{ formula }` with no `result`; rendering that raw stringifies to `[object Object]`. `xlsx-formula.ts` is a lightweight evaluator (cell refs, ranges, arithmetic, `SUM/AVERAGE/MIN/MAX/COUNT/PRODUCT/ROUND/IF/…`) used by a memoized, cycle-guarded resolver in `XlsxView.buildSheet`: prefer a cached `result` when present, else evaluate, else fall back to blank (unsupported fns like `VLOOKUP` throw → blank, never a wrong number). Unit-tested in `xlsx-formula.test.ts`. `cellRawValue` MUST handle the formula/richText/hyperlink object shapes or `[object Object]` returns.
- **"Open externally" (`shell:open-path`) opens a local file in its OS default app (HTML→browser, .docx→Word).** Desktop-only — remote-shim + (no) Android handler are no-ops; the button is `isElectron`-gated in `SessionDrawer`, shown in `FilesTab` detail tools. `xls`/`ods` are NOT rendered in-app (ExcelJS is xlsx-only) — they fall back to this button.
- **Clicking ANY filepath pill "artifactifies" it.** `FilepathToken` resolution order: session match → project (`listProject`/`listAllFiles`) match → else **appendVersion** (author `'user'`, type `'read'`) to create a sidecar entry, then `listSession` + select. So a file Claude made via a Bash/python script (untracked) or in a temp dir (external) still opens + appears in the drawer. `html`/`htm`/`svg` are in the `useInlineFilepathDetector` WHITELIST (consistent with `categorizeArtifact`) so those paths become pills.
- **`SessionDrawer` excludes `discovered` records from `checkExistence`** (`filter(a => !a.discovered)`). Discovered ids are relative paths, not sidecar ids, so the handler's "unknown id → missing" branch wrongly flagged on-disk files as **deleted** (the file was literally open + rendering). The general rule was already documented under the All-files bullet; this is the SessionDrawer enforcement.
- **In-app xlsx EDITING is deliberately deferred.** Decision 2026-06-17: keep the themed DOM viewer; do NOT adopt a canvas spreadsheet engine (Univer/x-spreadsheet) — canvas breaks the theme-token system (un-themeable, esp. community themes), kills DOM Ctrl+F / accessibility / native selection, fights app keyboard routing, and adds bundle/perf weight that ships to the Android WebView. Revisit only if "serious spreadsheet editing" becomes a headline product feature; otherwise a small single-cell editor (ExcelJS `writeBuffer` + a binary-write IPC) is the on-brand path.

### Merge-readiness hardening pass (2026-07-08, `feat/artifact-viewer`)

## Sync Spaces (Cross-Device Sync Phase 1a)

Shipped on youcoded `feat/sync-spaces` (plan: `docs/superpowers/plans/2026-07-03-sync-spaces-phase1a.md`). Module tree: `desktop/src/main/sync-spaces/`.

- **The hidden repo uses GIT_DIR env, NOT `--separate-git-dir`.** separate-git-dir writes a `.git` FILE into the worktree, which collides with a developer's own repo in the same project folder. Every git call in `git-transport.ts` sets `GIT_DIR=<root>/.youcoded/sync.git` + `GIT_WORK_TREE=<root>`. Never "simplify" to a normal `git init`. The contract suite's init test pins "nothing outside .youcoded/".
- **`info/attributes` is `* -text`, NOT the spec's original `* text=auto`.** Verified empirically on Windows: `text=auto` OVERRIDES `core.autocrlf=false` and converts LF→CRLF on checkout, breaking the byte-fidelity the setting existed to guarantee. `* -text` disables all EOL conversion. The contract suite's byte-exact assertions pin this — if you touch line-ending config, run `tests/sync-spaces-git-transport.test.ts` on a Windows box.
- **Ignores live in `$GIT_DIR/info/exclude`, attributes in `$GIT_DIR/info/attributes`.** Nothing of ours is ever written into the user's tree (no .gitignore, no .gitattributes). info/exclude is rewritten on every init; oversize exclusions are appended after and survive because excluded files stop being staged.
- **Conflict policy is convergent: REMOTE wins the canonical filename; LOCAL content becomes a visible conflict copy** (`name (from <device>, <date>).ext`). Both devices converge to identical trees — do NOT flip to local-wins-canonical, which never converges (each device keeps its own canonical and they re-conflict forever). Pinned by the transport contract suite.
- **`merge --allow-unrelated-histories` is load-bearing.** Two devices that both have pre-existing content in the same space (the Personal space on a second machine is the mainline case) have unrelated root commits at first contact; without the flag git refuses the merge and both devices are silently stuck forever (`updated:false` / `pushed:false`, no error). Contract test pins first-sync convergence for this shape. Companion fix: `pull()` adopts the remote via `checkout -B main origin/main` when local `main` is unborn (fresh second device).
- **Conflict-copy content is read as a Buffer via `showStage()` with `maxBuffer ≥ maxFileBytes` — both halves are load-bearing.** The string-returning `git()` helper must never be used for file CONTENT: utf8 decoding corrupts binary bytes, and Node's default 1MB `maxBuffer` kills `git show` on larger files — which made the conflict loop silently SKIP the local copy while `checkout --theirs` overwrote the file (silent data loss in exactly the scenario the convergent rule protects). Contract tests pin >1MB and binary conflict copies byte-for-byte.
- **`tests/sync-transport-contract.ts` is the transport compatibility boundary** (10 tests). Any new `SyncTransport` (SyncHub-assisted, YouCoded Cloud) must pass it unchanged. Changing the contract = changing the spec (§15) — do both in the same commit. The suite owns its own `vi.setConfig({ testTimeout: 30_000, hookTimeout: 30_000 })` — real transports are I/O-bound; do NOT move this into the global vitest config (it would mask hung unit tests).
- **`repoNameForSpace` appends a short hash of the LOWERCASED space id — do not "simplify" it away.** The repo name IS the sync identity. The slug alone is neither unique ('My App' and 'My-App' both slug to `my-app` — two different projects would silently cross-sync into one repo) nor total (all-symbol / non-Latin names slug to ''). Lowercased-id hashing keeps the name stable across devices (same folder name, any case → same repo) while distinct folder names always get distinct repos.
- **`provisionGithubRemote` treats an already-existing repo as SUCCESS** (falls back to `gh repo view`). The provisioned-URL state file (`~/.claude/toolkit-state/sync-spaces.json`) is per-device, so the user's second device re-runs provisioning for repos the first device created — without the recovery path, sync could never start on any device but the first.
- **Engine invariants:** single-flight per space with exactly one coalesced rerun (don't add a second queue); `addSpace` waits for chokidar `ready` (with `ignoreInitial:true`, writes landing before `ready` are silently swallowed as initial-scan entries — verified on Windows); a persistent `watcher.on('error')` handler is required (post-ready errors bypass `'all'`, and an unhandled EventEmitter `'error'` crashes the main process); `stop()` clears the state map FIRST, then awaits in-flight sync chains (their git subprocesses hold handles inside the space root — on Windows that blocks folder removal and races app-quit teardown).
- **Enable/disable transitions are serialized through a module-level promise chain in `service.ts`.** `enable(true)` racing `enable(false)` (two windows, panel double-click, remote client) would otherwise interleave — the disable stops the half-started engine and the resuming start adds watchers to a dead instance. `startEngine` is also supersession-safe (bails and self-cleans when a newer transition replaced it).
- **`isIgnoredPath()` in guards.ts keeps backups scrub-consistent with sync.** The iCloud backup's cpSync filter interprets the same `DEFAULT_IGNORES` the git layer enforces — a narrower hardcoded filter shipped briefly and leaked `*.pem`/`id_rsa*`/credentials into iCloud backups while Drive excluded them. New backup targets must route through it (or rclone `--exclude` derived from `DEFAULT_IGNORES`).
- **Android has NO `syncspaces:*` Kotlin handlers in 1a (deliberate).** The shared renderer still works: remote-shim `invoke()` rejects after a 30s timeout, and the SyncPanel/FolderSwitcher actions all catch rejections and surface messages inline. When mobile sync-spaces starts, add SessionService stubs returning `{ok:false, error:'not-implemented-on-mobile'}` first (the artifact-viewer stub pattern) so invokes reject fast instead of timing out.
- **The friendly gh error messages in `space-manager.ts` are a UI contract.** "GitHub CLI (gh) is not installed…" / "Not signed in to GitHub…" are shown verbatim by SyncPanel's error note (which renders the latest `error`-type event from `recentEvents`). Enable-path provisioning failures do NOT reject `enable()` — they arrive as broadcast error events; if you add a new failure path, make sure it lands in one of those two channels or the user sees nothing.
- **The legacy `sync-service.ts` is completely untouched in 1a.** GitHub-backup migration and the spec-§12 deletions are Phase 2 (when conversations move into the personal space) — flipping backup to daily-only earlier would regress conversation backup freshness to 24h. The dated daily space backup (`daily-backup.ts`) runs ALONGSIDE it, driven from the same backend config via `getSyncConfig()` (which is async and exposes `.backends` — not the on-disk `storage_backends` key).

### Import existing folders (Phase 1-followup, `feat/sync-import`, 2026-07-09)

The spec-§3 import flows: "Sync this project" on saved-folder rows + folder-picker import, both MOVING the folder into `~/YouCoded/Projects/<name>/` via `desktop/src/main/sync-spaces/import-project.ts` and the `syncspaces:import-project` IPC. Plan: `docs/superpowers/plans/2026-07-09-sync-spaces-import-existing-folders.md`.

- **Import MOVES the folder — never copy-and-keep-both.** The only copy is the EXDEV (cross-drive) fallback, which deletes the source afterward and surfaces a WARNING if that delete fails. A silent surviving copy at the old path forks the user's work (muscle memory keeps editing the old one while the synced copy rots). Don't "soften" the move to a copy.
- **The EXDEV branch re-checks `existsSync(dest)` BEFORE cpSync — load-bearing.** `rename(2)` reports EXDEV from the filesystem comparison before it ever looks at dest, so a dest created in the checkImport→move window (the sync engine materializing a same-named project from another device; dev + built app both running) would be silently MERGED into by cpSync and then DELETED by the copy-failure cleanup. The guard refuses instead. Never remove it, and never let the failure cleanup run against a dest this call didn't create.
- **The CC transcript slug dir must be renamed on move (`remapTranscriptDir`).** `~/.claude/projects/<slug>/` is keyed by a slug DERIVED from the cwd (not stored), so past conversations orphan silently unless the dir is renamed to the new path's slug (via `ccProjectSlug` — drive-case-normalized). When the new slug dir already exists, merge file-by-file, never clobber.
- **Remap failures degrade to WARNINGS, never errors.** After the move succeeds, the four store remaps (saved-folders path, central-index path, sidecar manualIncludes/Excludes prefixes, transcript slug dir) are each individually try/caught; a failure means stale pointers, never lost content. Each failure MUST surface as a user-facing warning — the sidecar CAS-miss throw exists so even that path lands in a warning; don't reintroduce a silent drop.
- **`remapProjectPath` (central-index) drops any stale entry already at the destination path.** Two index entries sharing one `path` (the store key) shadow each other and detach artifact history. The filter inside the `mutateIndex` lock is the guard.
- **`countFilesBounded` is bounded three ways: file count, depth (100), wall-clock (2s).** `isSymbolicLink()` does NOT detect NTFS junctions — a junction cycle containing no files never trips the file-count limit and recurses forever without the depth cap. Depth/time exhaustion returns the OVER-limit signal (fail closed), not a partial count — a partial count would let a monster tree into the sync engine. Don't "simplify" the caps away.
- **`FOLDERS_LIST` badges managed rows by projectsRoot PREFIX, not only via the synthesized merge.** The import flow rewrites the saved entry's path to its new managed home; prefix-badging is what keeps the "synced project" label on it. The synthesized-managed merge below it stays for managed folders with no saved entry.
- **`saved-folders.ts` is the single store module for `~/.claude/youcoded-folders.json`** (extracted from ipc-handlers; `writeFolders` is atomic via unique-tmp + rename because the dev + built app share `~/.claude`). Known remaining duplication: `remote-server.ts` still hand-rolls the format inline in its folders:* cases — migrate it there if you touch those.
- **`ImportProjectModal` close semantics are state-dependent.** Pre-move: Cancel/Scrim/ESC all cancel (ESC is gated `!busy`, matching Scrim + Cancel — don't let any dismiss path bypass the busy guard mid-move). Post-success (warnings shown): EVERY dismiss routes to `onDone(path)` so the folder list reconciles; only the pre-move state may bare-cancel.

- **Hero Rename edits the picker NICKNAME only (`folders.rename`), never the folder on disk.** A folder rename changes the sync identity (`repoNameForSpace` derives from the folder name) — that flow is deferred, as is move-out-of-sync (hence Remove is hidden for synced projects; "Managed by sync" shows instead, and ProjectSwitcher hides the remove-× for rows with a space).

### SyncHub (Plan 1b, 2026-07-10, wecoded-marketplace#21 + youcoded#114)

Instant cross-device sync signals: a `SyncGroupRoom` Durable Object in the marketplace worker relays metadata-only `{kind, spaceKey}` signals between one account's devices; the desktop's `sync-hub-socket.ts` (service-owned) turns them into immediate pulls.

- **The DO is per-account — `idFromName(userId)`, never global.** Contrast the social `PresenceRoom` (one global room + D1 friend graph): SyncHub has **no D1 at all** and no cross-account visibility (isolation is pinned by a worker test). Auth happens in the Hono route (`requireAuth`); the DO trusts only the internal `X-Sync-User`/`X-Sync-Device` headers the route sets — never add a direct DO entry point.
- **The DO is an ACCELERANT, never a source of truth.** Losing its state loses nothing: devices reconcile-on-connect (sync every space) and the engine's 120s poll stays as the fallback. **Don't "optimize" the poll away** — SyncHub down must only make sync less instant, never block it.
- **spaceKey across devices is `repoNameForSpace()`, NEVER the local space id.** Local ids (`project:<folderName>`) differ across devices by folder-name case; the repo name (slug + sha1 of lowercased id) is the stable cross-device identity. The service resolves inbound spaceKeys back to spaces by comparing repo names.
- **Signal ONLY on `pushed:true` — that's the loop breaker.** A signal-triggered pull that pushes nothing broadcasts `pushed:false` and re-signals nothing, so the signal loop converges. Don't widen the guard in `broadcast()` (it also keeps `hub-status`/`error` events from recursing into a send).
- **The hub send in `broadcast()` runs LAST and isolated (own try/catch).** `broadcast()` is the single fan-out chokepoint from the engine's `onEvent`; a hub failure must never block or kill local window/remote event delivery. Signalling is best-effort — the poll covers misses. Keep the ordering.
- **A superseded `startEngine` run owns NO global state.** The supersession guard (`engine !== e`) sits BEFORE any hub work; everything after it is synchronous. Moving hub-socket creation or `hubStatus` writes above the guard reintroduces the boot-race clobber where a stale run stamps `'off'` over a newer live socket's `'connected'` (race-pinned by a service test using the `initialEnabled` boot knob).
- **`hub-status` events carry the sentinel `spaceId: 'hub'`.** The literal never collides with real space ids, so `sync-dot-state.ts`, ProjectHero, and per-space event scans ignore them naturally. SyncPanel filters them from visible notices (the "Instant sync" line is their surface) but keeps them in `recentEvents` data. If you add a new per-space event consumer, match on real space ids and hub-status stays inert for free.
- **The relay `kind` allowlist (`ALLOWED_KINDS` in `worker/src/sync/room.ts`) is where Phase 2's lease/takeover kinds get added** — the relay/ring logic is deliberately kind-agnostic; extending is a one-line allowlist change, not a structural one.
- **`device` is a client-chosen label (from `?device=`), connection-pinned via the socket attachment — NOT authenticated identity.** Two devices can legitimately send the same label. Phase 2 lease logic must not key on it as verified/unique.
- **Any new DO binding must be mirrored under `[env.test.durable_objects]`** in wrangler.toml (env.test does not inherit) or every worker test fails; new DO classes need a NEW `[[migrations]]` tag (v2 added `SyncGroupRoom`; v1 untouched). Migrations are NOT repeated in env.test.
- **The hub socket is SERVICE-owned (works with zero windows open), unlike the renderer-driven presence socket.** Its no-token path retries on a 30s timer (clear-then-reschedule — a fired timeout leaves the variable non-null, so an `if (!retryTimer)` guard kills the poll after one cycle; pinned by a socket test) so mid-session sign-in connects without a restart. Auth arrives via the `setSyncSpacesAuthStore` facade wired in main.ts BEFORE `startSyncSpaces`.
- **Enable-time pushes send no signal (socket not open yet)** — other devices catch up via their poll. Inherent and by design; don't "fix" by delaying the engine start on the socket.

### Conversation Store (Phase 2a, youcoded#116)

Per-conversation JSON records at `~/YouCoded/Personal/Conversations/claude/<id>.json` + CC transcript mirroring, riding the personal sync space. Module tree: `desktop/src/main/conversations/`. Spec: `docs/superpowers/specs/2026-07-10-phase2-conversation-sync-design.md` §1–§2. Leases are Plan 2b; legacy demolition is Plan 2c.

- **Mirror-in is add/update-only AND shrink-guarded — CC `cleanupPeriodDays` deletion and `/clear` rewrites must NEVER shrink the durable space copy.** `transcript-mirror.ts` compares sizes: local missing → no-op (deletion never propagates); local smaller → `{copied:false, shrunk:true}`, space untouched; equal → no write; larger/space-missing → copy. Both directions never delete. The space copy is the durable one — the load-bearing invariant, byte-pinned in `transcript-mirror.test.ts`.
- **`lastActive` comes from the transcript content timestamp, NEVER file mtime.** The reconciler reads `readSessionTranscriptMeta().lastTimestampMs`; a corrupt `>500B` transcript with no parseable tail timestamp is SKIPPED (creating an EPOCH-dated record would re-upsert every scan forever). The 627-file mtime-rebump incident is why.
- **Merges are convergent (a lattice join), not positional.** `store-core.ts mergeRecords` breaks exact-`lastActive` ties with a total-order content comparison (`JSON.stringify`), so `merge(a,b) === merge(b,a)`. `foldConflictCopies` picks EVERY field group (activity/spread, flags, createdAt, title) over the ORIGINAL input set, not a mutated reduce accumulator — otherwise directory-enumeration order changed the healed record and two devices ping-ponged forever. Literal `'Untitled'` is a placeholder that always loses a real title (the legacy topic-file placeholder lesson).
- **The record id/provider is a path-traversal boundary.** `conversation-store.ts` charset-allowlists both (`[A-Za-z0-9._-]`, rejects `.`/`..`, separators, NUL, Windows reserved names) AND `path.resolve`-contains the result under the provider dir. Reachable over the remote WS after the Task 6 wiring — don't bypass. Writes throw on a bad id; reads fail soft (null/[]).
- **`heal()` claims each conflict copy by quarantine-rename (`<name>.healing-<pid>`) BEFORE reading it.** Without it a multi-process heal (dev + built app share `~/YouCoded`) could unlink a copy the engine had just re-created with NEW content at the same day-granular name — deleting it unfolded. Quarantine atomically claims the exact bytes; crash-orphaned quarantines are adopted + folded on the next heal. Read paths (`get`/`list`) swallow heal failures (fail-soft); `upsert` throws on lock timeout.
- **`upsert` treats caller-provided fields as LOCAL TRUTH.** After `mergeRecords` decides activity fields by recency, provided `projectName`/`originalPath`/`transcriptRef` are re-applied unconditionally and a non-empty/non-`'Untitled'` title is re-applied — the merge is for cross-device convergence, not for overruling the caller. Metadata-only upserts default `lastActive` to the EPOCH sentinel so they never outrank real activity.
- **The materialize sweep and the Resume Browser union both SKIP live sessions (no leases in 2a).** Sweep: `if (sessions.has(rec.id)) continue;` (`service.ts`); union: `if (activeSessionIds?.has(rec.id)) continue;` (`session-browser.ts`). Without these, a pull could `materializeOut` over — or the browser could offer `--resume` against — a transcript CC is actively appending to (Windows EPERM containment is unreliable; POSIX rename detaches CC from its inode, freezing chat view and losing turns). Accepted caveat: nothing removes `sessions` entries on session exit, so ended sessions stay sweep-guarded until restart (fine for 2a — `mirrorIn` keeps the space current from the local side). `noteSessionEnded` refinement lands with leases in 2b.
- **The reconciler recovers the EXACT projectKey from known folders; last-segment truncation is only a fallback.** The live path keys transcripts by `basename(cwd)` (`youcoded-dev`); the slug last segment truncates hyphenated names to `dev`. A mismatch produces an ORPHAN duplicate space transcript and a cross-device materialize gap (a `dev`-keyed record never matches the peer's managed `youcoded-dev`). The reconciler builds a `ccProjectSlug(folder) → basename` map (lowercased for Windows case drift) from managed projects + saved folders; truncation applies only to genuinely unknown paths.
- **The LEGACY sync system symlinks every conversation into the HOME-dir slug — the reconciler MUST skip symlinks (youcoded#118).** NOT a Claude Code behavior (an earlier note wrongly attributed it to CC). `SyncService.aggregateConversations()` (`sync-service.ts:2062`, run from `pull()` on every app launch/restore) symlinks each non-home slug's `.jsonl` into the home slug `C--Users-desti/<id>.jsonl -> ../C--Users-desti-<proj>/<id>.jsonl` "for /resume from ~"; `rewriteProjectSlugs()` (`:2026`) junctions whole foreign-device slug dirs similarly. Verified on a real machine: 687 such 73-byte symlinks, ALL in the home slug. The reconciler's `statSync` FOLLOWED them (symlink → real target size passes the gate), tagging every linked conversation with the home basename (`desti`) and bucketing all transcripts into `transcripts/desti/` (the dev smoke test mis-keyed 921/921 records this way). Fix: `lstatSync` (does NOT follow) + `isSymbolicLink()` skip. The real transcript is processed in its true cwd slug; a genuine home-dir session is a REAL file and is kept as `desti`. This machinery (slug rewriting + foreign-slug symlinks) is on the **Plan 2c demolition list** — the FULL four-part task (delete the two creators, a one-time on-disk sweep of the ALREADY-created symlinks, keep this reconciler skip, accept the `/resume from ~` CLI consequence) is spelled out in **design §4a**. Do NOT stop creating the symlinks before 2c — `sync-service.ts` is frozen (the live app's `/resume from ~` + legacy backup still rely on the aggregation), and the built app keeps creating them until the next release ships 2c anyway, so the reconciler skip is the necessary-and-sufficient defense until then. **Note for 2c: deleting the creator does NOT remove the existing symlinks — the one-time sweep is required, and must remove only `lstat`-symlinks/junctions, never real files.**
- **Fire-and-forget store writes MUST `.catch()`** — `upsert`/`setFlag`/`setTitle` reject on lock timeout, and an uncaught rejection in Electron main is fatal-ish. `service.ts` routes every write through a catch; never `void store.x()`.
- **The reconciler preloads all records via ONE `store.list()` pass, not `store.get()` per file.** `get()` heals (readdirs the whole provider dir); per-file gets made the scan O(n²) in dirents (~2.8s at 600 records every startup + 30-min tick). Preload is ~200× fewer reads; the map updates after each upsert (the same UUID can appear under two slug dirs).
- **`sync-service.ts` is byte-untouched until Plan 2c.** The legacy conversation-index runs in parallel; flags dual-write (legacy + store) during the transition. Flipping backup to daily-only before 2c regresses conversation backup freshness — the standing ordering guard.
- **The store works with sync OFF.** `startSyncSpaces()` always runs and populates `getManagedRoots()`; only the engine is gated on the enable flag. Records write locally regardless; they don't travel until sync is enabled.
- **Known 2a limitations:** a store-only browser row on a second device can transiently read "Not synced to this device yet" until the same-pull sweep materializes the transcript (self-corrects next open); transcripts >50 MB hit the engine's pre-existing `MAX_SYNC_FILE_BYTES` cap and never reach peers.

### Cross-device project discovery / rename / stop (2026-07-12, youcoded `feat/sync-project-discovery`)

Spec: `docs/superpowers/specs/2026-07-12-cross-device-project-discovery-design.md`. Modules: `desktop/src/main/sync-spaces/{project-registry,materialization-planner}.ts` + `engine.ts` (`removeSpace`/`liveSpaceIds`) + `service.ts` wiring.

- **The project registry lives at `~/YouCoded/Personal/ProjectSync/<name>.json` — VISIBLE per-file, mirroring the Conversation Store, NEVER under `.youcoded/`** (that basename is the transport's hidden git dir AND a `DEFAULT_IGNORES` entry, so anything under it never syncs). `sync-spaces-project-discovery.test.ts` pins that the registry round-trips through the transport.
- **The record is convergent and mutable (`{schemaVersion,name,repoName,displayName,state,updatedAt}`) — `name` is the IMMUTABLE identity (folder name), `displayName`/`state` are synced+mutable.** The field-wise merge: **`state` is `stopped`-dominates monotonic** (NOT last-writer-wins — a stale rename on a device that hasn't pulled the stop must never un-stop it; consequence: no Resume without a per-field `state` timestamp), **`displayName` is last-writer-wins** by `updatedAt`. Reuses `laterOf`/`isConflictCopyName`/`extractConflictBase` (`conversations/store-core`, `laterOf` now exported) + `mutateFileUnderLock` (`artifacts/cas-write`).
- **Fold-on-read is load-bearing — provable against the transport.** The transport's conflict policy is remote-wins-canonical, so it can leave the WRONG winner as the canonical `<name>.json` (a peer's older `active` overwriting a local `stopped`). `readProjectRegistry` folds every conflict copy into the canonical in memory (stopped dominates), so a stopped project can't read active and resurrect. Copy files are left in place (rare, inert). Don't "optimize away" the fold.
- **Rename is display-name only; the folder never moves** (`repoName` derives from the folder name, which stays fixed — that's why no ULID). True on-disk folder rename is deferred (spec §15).
- **Stop = tombstone + `engine.removeSpace` + KEEP the folder.** The folder stays under `Projects/`, still a project, just detached. `activeManagedSpaces()` (the ONE gate, routed through all three `roots.spaces()` sites — startEngine add loop, hub-connected sync loop, daily-backup loop, via the service's `activeSpaces()` helper) keeps stopped projects out of the live engine so they never respawn. Don't filter at the sites individually and don't make `ManagedRoots.spaces()` registry-aware.
- **`planReconcile`'s third arg is live project NAMES, not space ids.** `runDiscovery` (and every caller, including the integration test) converts `engine.liveSpaceIds()` `'project:app'` → `'app'` before passing it. Passing raw space ids silently empties `toStop` (a stopped project never detaches).
- **`ensureRemote` runs FIRST in materialize** (it uses only `space.id`, so a gh-auth failure creates nothing); `addSpace` runs BEFORE the first pull (a failed pull leaves a recoverable empty space). Materialize reuses the empty-folder + first-sync-pull path — NO `git clone`.
- **`runDiscovery` is single-flight + one coalesced rerun**, reads the registry ON DISK (triggers guarantee freshness: startEngine awaits a Personal pull; broadcast on Personal `synced`+`updated`, and hub `connected` fire after a Personal sync). Don't make it sync Personal itself (recurses through the broadcast trigger).
- **Rename/stop ride the four-surface IPC parity** (`preload.ts`, `remote-shim.ts`, `ipc-handlers.ts`, `remote-server.ts`) via `syncspaces:rename-project` / `syncspaces:stop-project` (payload objects `{name,displayName}` / `{name}`); the `IPC.SYNC_SPACES_*` constants live in `shared/types.ts` AND the inlined preload map. `ipc-channels.test.ts` pins the parity.
- **UI: the status dot is the sync/stopped signal; location ≠ status** (synced + unsynced projects both live in `Projects/`). `sync-dot-state.ts` adds a gray `"Sync stopped"` state; the ProjectHero "Stop syncing" button is consequence-gated; rename routes through `syncSpaces.renameProject` for synced projects (local `folders.rename` otherwise); rows overlay the synced `displayName` from the status payload (single source of truth — never written back into `youcoded-folders.json`). The richer 3-category Project View labeling (native-synced / native-unsynced / external) is a deferred follow-up (spec §10/§15).
- **Android:** no discovery yet (Phase 3); `syncspaces:rename-project`/`syncspaces:stop-project` are `not-implemented-on-mobile` stubs. Registry format is platform-neutral for later reuse.

## Local Dev & Launch Environment

- **Spawned `claude` sessions must NOT inherit Claude Code's own session-identity env vars — `pty-worker.js` strips them.** If YouCoded is launched from *inside* a Claude Code session (the common case when a Claude session runs `bash scripts/run-dev.sh` via the Bash tool, or when you start the app from a terminal that is itself a CC session), the Electron process inherits `CLAUDECODE=1`, `CLAUDE_CODE_CHILD_SESSION=1`, `CLAUDE_CODE_SESSION_ID=<parent>`, `CLAUDE_CODE_ENTRYPOINT`, `CLAUDE_CODE_EXECPATH`. `pty-worker.js` spread `...process.env` into every `claude` it spawns, so the child believed it was a **nested/child session**. **Nested interactive Claude Code does NOT write a top-level transcript** to `~/.claude/projects/<slug>/<id>.jsonl`. Since the `TranscriptWatcher` is the *sole* source of chat-view state, the chat view stays **permanently empty** — messages reach Claude and replies show in **terminal view**, but never in **chat view**. The PTY works and hooks still fire (auto-title, session-start sidecars get written), which masks the real cause and makes it look like only chat rendering broke. **Fix lives at the spawn chokepoint** (`desktop/src/main/pty-worker.js`, case `'spawn'`): it deletes those vars from the child env before `pty.spawn`, so every session is a clean top-level session regardless of how YouCoded was launched. Don't re-add a raw `...process.env` spread there. (`CLAUDE_DESKTOP_SESSION_ID` / `CLAUDE_DESKTOP_PIPE` are still set fresh per-session — those are ours, not CC's.) Shipped in youcoded#106.
- **Diagnostic signature:** terminal view works + chat view empty + `find ~/.claude/projects -name '<dev-session-id>*.jsonl'` returns nothing, even though `~/.claude/topics/topic-<id>`, `~/.claude/.gitbranch-<id>`, `~/.claude/.session-stats-<id>.json` exist (hooks ran). That combination = the spawned CC ran as a child and wrote no transcript. Plain `claude -p` is NOT a useful repro — print/`sdk-cli` mode always writes a transcript even with the child markers set; only the **interactive** path suppresses it.
- **Android is not affected.** Android spawns CC via `Bootstrap.buildRuntimeEnv()` in the Termux runtime, not from an inherited desktop shell env; there is no CC-inside-CC launch path on device.

## Working With Destin

- **"Merge" means merge AND push to origin.** Don't stop at a local merge.
- **Always sync before working.** `git fetch origin && git pull origin master` in every repo you'll touch. Prevents working against stale state.
- **Annotate non-trivial code edits with a WHY comment.** Destin is a non-developer and relies on comments to understand what code does and why it was changed. Example: `// Fix: prevent stale tool IDs from coloring the status dot`.
- **Verify fix consequences before shipping.** Batch fixes — especially network/permission changes — can silently break cross-cutting features. Check both platforms.
