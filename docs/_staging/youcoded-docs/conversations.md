# Conversation store, leases & Resume Browser identity — depth

Per-conversation JSON records at `~/YouCoded/Personal/Conversations/claude/<id>.json` + CC transcript mirroring, riding the personal sync space; one-holder-per-conversation leases + cross-device takeover; and the Resume Browser's name/path resolution. Modules: `desktop/src/main/conversations/`, `session-browser.ts`, `device-identity.ts`. The rule (`.claude/rules/conversations.md`) carries the load-bearing invariants; this is the why.

## Conversation Store (Phase 2a, `conversations/{transcript-mirror,store-core,conversation-store,reconciler,service}.ts`)

- **Mirror-in is add/update-only AND shrink-guarded** — CC `cleanupPeriodDays` deletion and `/clear` rewrites must NEVER shrink the durable space copy. `transcript-mirror.ts` compares sizes: local missing → no-op (deletion never propagates); local smaller → `{copied:false, shrunk:true}`, space untouched; equal → no write; larger/space-missing → copy. Both directions never delete. The space copy is the durable one. Byte-pinned in `transcript-mirror.test.ts`.
- **`lastActive` comes from the transcript content timestamp, NEVER file mtime.** The reconciler reads `readSessionTranscriptMeta().lastTimestampMs`; a corrupt `>500B` transcript with no parseable tail timestamp is SKIPPED (an EPOCH-dated record would re-upsert every scan forever). The 627-file mtime-rebump incident is why.
- **Merges are convergent (a lattice join), not positional.** `store-core.ts mergeRecords` breaks exact-`lastActive` ties with a total-order content comparison (`JSON.stringify`) so `merge(a,b) === merge(b,a)`. `foldConflictCopies` picks EVERY field group (activity/spread, flags, createdAt, title) over the ORIGINAL input set, not a mutated reduce accumulator — otherwise directory-enumeration order changed the healed record and two devices ping-ponged forever. Literal `'Untitled'` is a placeholder that always loses a real title.
- **The record id/provider is a path-traversal boundary.** `conversation-store.ts` charset-allowlists both (`[A-Za-z0-9._-]`, rejects `.`/`..`, separators, NUL, Windows reserved names) AND `path.resolve`-contains the result under the provider dir. Reachable over the remote WS — don't bypass. Writes throw on a bad id; reads fail soft (null/[]).
- **`heal()` claims each conflict copy by quarantine-rename (`<name>.healing-<pid>`) BEFORE reading it.** Without it a multi-process heal (dev + built app share `~/YouCoded`) could unlink a copy the engine just re-created with NEW content at the same day-granular name — deleting it unfolded. Quarantine atomically claims the exact bytes; crash-orphaned quarantines are adopted + folded next heal. Read paths (`get`/`list`) swallow heal failures; `upsert` throws on lock timeout.
- **`upsert` treats caller-provided fields as LOCAL TRUTH.** After `mergeRecords` decides activity fields by recency, provided `projectName`/`originalPath`/`transcriptRef` are re-applied unconditionally and a non-empty/non-`'Untitled'` title is re-applied — the merge is for cross-device convergence, not for overruling the caller. Metadata-only upserts default `lastActive` to the EPOCH sentinel so they never outrank real activity.
- **The materialize sweep and the Resume Browser union both SKIP live sessions (no leases in 2a).** Sweep: `if (sessions.has(rec.id)) continue;` (`service.ts`); union: `if (activeSessionIds?.has(rec.id)) continue;` (`session-browser.ts`). Without these, a pull could `materializeOut` over — or the browser could offer `--resume` against — a transcript CC is actively appending to (Windows EPERM containment unreliable; POSIX rename detaches CC from its inode → frozen chat view + lost turns). Accepted caveat: nothing removes `sessions` entries on session exit, so ended sessions stay sweep-guarded until restart (fine for 2a). `noteSessionEnded` refined in 2b.
- **The reconciler recovers the EXACT projectKey from known folders; last-segment truncation is only a fallback.** The live path keys transcripts by `basename(cwd)` (`youcoded-dev`); the slug last segment truncates hyphenated names to `dev`. A mismatch produces an ORPHAN duplicate space transcript + a cross-device materialize gap. The reconciler builds a `ccProjectSlug(folder) → basename` map (lowercased for Windows case drift) from managed projects + saved folders; truncation applies only to genuinely unknown paths.
- **The LEGACY sync system symlinks every conversation into the HOME-dir slug — the reconciler MUST skip symlinks (youcoded#118).** NOT a Claude Code behavior. `SyncService.aggregateConversations()` symlinks each non-home slug's `.jsonl` into the home slug "for /resume from ~"; `rewriteProjectSlugs()` junctions whole foreign-device slug dirs. Verified in the wild: 687 such 73-byte symlinks, all in the home slug. The reconciler's `statSync` FOLLOWED them → tagged every linked conversation with the home basename (`desti`) and bucketed all transcripts into `transcripts/desti/`. Fix: `lstatSync` + `isSymbolicLink()` skip. Plan 2c (unmerged) additionally deletes the symlink CREATORS and sweeps existing ones; this reconciler symlink-skip is KEPT as a cheap defense against any future symlink source — do NOT remove it.
- **Fire-and-forget store writes MUST `.catch()`** — `upsert`/`setFlag`/`setTitle` reject on lock timeout, and an uncaught rejection in Electron main is fatal-ish. `service.ts` routes every write through a catch; never `void store.x()`.
- **The reconciler preloads all records via ONE `store.list()` pass, not `store.get()` per file.** `get()` heals (readdirs the whole provider dir); per-file gets made the scan O(n²) in dirents (~2.8s at 600 records every startup + 30-min tick). Preload is ~200× fewer reads.
- **The store works with sync OFF.** `startSyncSpaces()` always runs + populates `getManagedRoots()`; only the engine is gated on the enable flag. Records write locally regardless; they don't travel until sync is enabled.
- **Known 2a limits:** a store-only browser row on a second device can transiently read "Not synced to this device yet" until the same-pull sweep materializes the transcript (self-corrects next open); transcripts >50 MB hit `MAX_SYNC_FILE_BYTES` and never reach peers.

## Session leases & takeover (Plan 2b, `conversations/{takeover,lease-client,service}.ts`, `device-identity.ts`)

One-holder-per-conversation coordination + cross-device takeover, on the Conversation Store (2a) + SyncHub (1b). DORMANT behind `native.supported=false`/`YOUCODED_NATIVE` EXCEPT the materialize-on-release, Bug-1 browse filter, and SessionStart-lease-acquire paths, which run for real CC sessions (null-safe when sync/hub absent).

- **Lease ops are DO-AUTHORITATIVE request/response, NOT client-relayed signals.** `{type:"lease", op:acquire|renew|release|takeover|force-acquire|get}` hits the `SyncGroupRoom` DO, which owns the lease table with server-computed lazy 300s expiry (30s heartbeat) — the DO is the single arbiter, unlike the `space-updated` SIGNAL relay that just fans out. Don't route lease decisions through the signal path.
- **`lease-event` broadcasts (released/taken/takeover-request) NEVER enter the replay ring.** A replayed stale lease frame would LIE about current state (assert a holder that already released). Lease state is re-queried live via `op:get`, never reconstructed from replay. `ALLOWED_KINDS` stays `{space-updated}` — leases are a separate message TYPE, so the "lease-acquired is a disallowed signal kind" worker test still holds.
- **Leases key on the per-INSTALL `deviceId`, NEVER the client-chosen device LABEL.** `device-identity.ts` mints a UUID in Electron `userData/device-id.json`. Per-install (not per-machine) is load-bearing: the dev instance and built app share `~/.claude` but have SEPARATE `userData`, so distinct deviceIds coordinate them cross-process on ONE machine. The SyncHub `device` label is a display string two devices can share — never key a lease on it.
- **`noteSessionEnded`'s quiescence gate: `session-exit` fires BEFORE the PTY worker dies (CC may still be flushing a final turn), so materialize-on-end MUST skip on timeout.** `noteSessionEnded` (service.ts) releases the materialize guard + runs a TARGETED `materializeOne` that polls the local transcript size until stable across one 750ms probe (max 6s). On TIMEOUT it MUST return without materializing (a `quiesced` flag) — a space→local `renameSync` over a still-open transcript detaches CC's inode on POSIX and loses the flush (a real data-loss bug caught in review). The reconciler/startup sweep catch up a skipped round.
- **`flushSessionToSpace` (holder-takeover local→space) reuses the same wait but pushes REGARDLESS of timeout.** mirrorIn is grow-only and never touches CC's open local file, so that direction is always safe — only the space→local materialize has the inode-detach hazard. The asymmetry is deliberate; don't unify behind one skip-on-timeout rule.
- **Bug-1 browse filter: `sessionIdMap` is a CACHE, not truth — filter its entries to LIVE sessions.** `SESSION_BROWSE` builds its active-set by filtering `sessionIdMap.entries()` to mappings whose desktop session still exists (`sessionManager.getSession(desktopId)`). A stale entry used to hide a CLOSED session from the Resume Browser until restart. Don't revert to raw `sessionIdMap.values()`.
- **Holder-takeover ordering is load-bearing: interrupt(ESC) → flush(local→space) → release(lease) → pushMoved → destroySession** (`createHolderTakeover`, takeover.ts). MIRROR-BEFORE-RELEASE: the requester pulls the moment it sees the lease free, so the final turn must already be in the space. RELEASE/push-moved-BEFORE-DESTROY: the enriched `session:moved` push must reach the renderer while the session still exists (so its cwd/claude-id are readable AND the renderer records the moved entry before the destroy arrives). Every step is independently try/caught + an OUTER backstop — the handler is fire-and-forget from a hub event, so a throw = an unhandled rejection.
- **Never-block: any lease query/takeover failure degrades to "proceed with the resume" (+ a warning), never a hard block.** `createRequesterTakeover` runs op:'takeover' → poll `query` every 1s up to 10s (checks free BEFORE the first sleep) → on free: syncNow('personal') → `materializeOne` → acquire; on timeout → the "take over anyway?" force dialog → force-acquire. A lease ERROR at any point lets the resume proceed — never hard-block a user from their OWN conversation. Device registry: `Personal/Devices/<deviceId>.json` (per-file, convergent, name LWW + lastSeen max) → the "Your devices" list (plain text, no glyphs).
- **Hub-down `requester.force` returns `{ok:true}` even when `forceAcquire` was null — a deliberate never-block degradation, NOT a bug to fix into a hard block.** If the hub is unreachable the real holder never got the `takeover-request` either, so both devices can end up live: grow-only mirror churn (visible conflict copies, never silent loss), or a genuine same-file double-writer only on a shared-`~/.claude` dev+built-app machine. The `{ok:true}` is optimistic; surfacing a distinct "hub was down" outcome is an optional refinement, not required.

### MovedGate (the "taken over" surface, App.tsx — renderer, not in these modules but tied to 2b)

- **The "taken over" surface is App.tsx's MovedGate (a full-page gate), NOT a timeline marker — and `destroyedHandler` MUST read `movedSessionsRef.current`, never the `movedSessions` state.** The original marker was appended by `SESSION_MOVED` then wiped back-to-back by the holder's own destroy (`SESSION_REMOVE`), so it never rendered — a taken-over session silently vanished. Fix: the holder KEEPS the pill + still destroys the CC session; clicking it renders `<MovedGate>` (modeled on `TrustGate`: content-area gate at z-10 below chrome, opaque `bg-canvas`, plain words) with **Exit Session** (`session.destroy()` no-ops on the dead session) and **Resume on this device** (runs `handleResumeSession`). No second writer, no un-removable zombie, no competing "session died" banner (the moved branch `return`s before `SESSION_PROCESS_EXITED`/`SESSION_REMOVE`). STALE-CLOSURE TRAP: the handlers are registered once in a `[dispatch]`-dep mount effect, so a `movedSessions` STATE read inside `destroyedHandler` is permanently the initial empty Map — read `movedSessionsRef.current` (`recordMoved`/`clearMoved` keep it in lockstep, updated SYNCHRONOUSLY so the ref is populated before the async post-PTY-death destroy fires). The enriched push carries `claudeSessionId`/`projectSlug`/`projectPath` so Resume needs no second lookup.

## Resume Browser & conversation identity (`session-browser.ts`; from the 2026-06-12 missing-sessions investigation)

- **Topic-file mtime IS the conversation-index's `lastActive` source.** `updateConversationIndex()` upserts whenever the topic file's mtime is newer than stored `lastActive`. Anything that (re)writes topic files MUST preserve the original timestamp (`fs.utimesSync`) — `regenerateTopicCache()` is the canonical writer and does. "Now" mtimes create a feedback loop that bumps EVERY session's lastActive each cycle → the 30-day prune never fires + the recent-50 backup pull fetches the wrong sessions (observed: all 627 topic files ≤2 days old).
- **`sessionIdMap` remaps ONLY on `SessionStart` hook events.** CC rotates its session id mid-PTY on `/clear` (verified 2026-06-12; `--resume` does NOT rotate, `/compact` rewrites the same file). The remap gate must stay SessionStart-only: subagent/tool hook events carry child session ids and would poison the map. The remap also assumes the rotated transcript starts EMPTY.
- **Resume Browser name precedence: topic file > conversation-index topic > derived-from-first-user-message > "Untitled".** The derived title (`readTranscriptMeta`, mirrored in Kotlin `SessionBrowser`) + content-timestamp ordering exist because (a) the auto-title hook is PostToolUse-gated (chat-only sessions never fire it) + model-compliance-dependent, and (b) sync restores clobber file mtimes so `stat.mtimeMs` lies about recency. Don't simplify ordering back to mtime or drop the head/tail parse. Reads are bounded (256KB head only when untitled / 64KB tail). Android applies the final precedence in `SessionService`'s `session:browse` handler.
- **Literal `Untitled` topic-file content is a placeholder.** Both desktop `readTopic` and Kotlin `SessionBrowser` normalize it (alongside blank + `New Session`) so the index/derived fallback chain applies. Treating the literal string as a real name silently bypasses both fallbacks.
- **Conversation-index keys are CC session UUIDs — `SESSION_UUID_RE` gates the topic scan.** The auto-title flow has the model hand-type `echo "Title" > topics/topic-<id>`; a typo'd id created phantom index entries pointing at no transcript. The scan guard blocks CREATION from malformed ids (existing entries may still be updated — `setSessionFlag` seeds flag-bearing entries before a topic file exists); the prune loop self-heals malformed entries that carry no flags. Flagged malformed entries are KEPT (deleting a user's tag is worse than a dead row).
- **`cleanupPeriodDays` is seeded (365) when absent — never overwritten.** CC's built-in default is 30 days and silently deletes transcripts the Resume Browser lists from. `retention-default.ts` (desktop) + `RetentionDefault.kt` (Android) seed only when the key is missing, respect explicit user values, and never rewrite a corrupt settings.json (it carries hooks + enabledPlugins).

### Slug→path resolution for resume (2026-07-12 two-device dogfood, `session-browser.ts` + `conversations/resolve-local-project.ts`)

A CC slug is the cwd with every separator flattened to `-`, so reversing a slug is AMBIGUOUS when a folder's own name contains a hyphen (`C--Users-desti-youcoded-dev` = `…\desti\youcoded-dev` OR `…\desti\youcoded\dev`). Resume depends on this direction.

- **`walkSlugParts` MUST try the LONGEST leading segment first — do NOT revert to shortest-first.** Shortest-first greedily descends into a stray shorter sibling: with `C:\Users\desti\youcoded` present, the slug resolved to the nonexistent `…\youcoded\dev`, and `session-manager` silently launched `claude --resume` in `$HOME` → every hyphenated-folder session "resumed from the home directory." Pinned by `desktop/tests/slug-path-resolution.test.ts`.
- **Store-backed rows override the slug-walk with the store's exact-basename resolution** (`resolveLocalProject`, shared by the materialize sweep AND the Resume Browser via `buildLocalProjectResolver`), but only when the resolved folder actually holds the transcript. Unambiguous; makes materialize + resume agree on every OS. Cross-OS store-only rows resolve by `projectName` basename too (originalPath is the foreign device's path). Pinned by the cross-OS test in `session-browser.test.ts` + `resolve-local-project.test.ts`.
- **`session-manager.ts`'s `existsSync(cwd) ? cwd : os.homedir()` is a SILENT MASK — it now WARNS on a resume with a missing cwd.** Behavior unchanged; the warning makes a future upstream regression visible instead of a silent wrong-directory resume. Don't remove the warn.
- **ANDROID MIRRORS THE ORIGINAL (PRE-FIX) BUG — all three failure modes still live there (deferred with Android sync, Phase 3).** `SessionBrowser.kt`'s `walkSlugParts` is still SHORTEST-first (its doc comment wrongly claims longest-first); the `session:create` bridge handler DROPS `resumeSessionId` (Android "resume" launches a FRESH session) + passes the raw cwd with no existence check; there is NO `resolveLocalProject`/store overlay; and `listPastSessions` does NOT skip the legacy home-slug symlinks. `SessionRegistry.resumeSession` has the CORRECT cwd derivation but is dead code. When Android resume/sync is picked up, port: longest-first `walkSlugParts`, thread `resumeSessionId` + a cwd existence guard, add a store/basename resolver.
