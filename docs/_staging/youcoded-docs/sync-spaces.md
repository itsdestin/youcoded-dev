# Sync Spaces, SyncHub, backup & GitHub-connect — depth

Cross-device sync = a hidden per-space git repo the app pushes/pulls, plus SyncHub (instant signals) and a daily backup snapshot. Modules: `desktop/src/main/sync-spaces/`, `sync-service.ts`, `sync-hub-socket.ts`, `sync-error-classifier.ts`, `github-{auth,connect}.ts`. The rule (`.claude/rules/sync-spaces.md`) carries the load-bearing invariants; this file carries the why.

## Git transport (`sync-spaces/git-transport.ts`)

- **`GIT_DIR` env, not `--separate-git-dir`.** separate-git-dir writes a `.git` FILE into the worktree, colliding with a developer's own repo in the same folder. Every git call sets `GIT_DIR=<root>/.youcoded/sync.git` + `GIT_WORK_TREE=<root>`. The contract init test pins "nothing outside `.youcoded/`".
- **`info/attributes` = `* -text`, NOT `* text=auto`.** On Windows `text=auto` overrides `core.autocrlf=false` and converts LF→CRLF on checkout, breaking byte fidelity. `* -text` disables all EOL conversion. Byte-exact contract assertions pin this — run `tests/sync-spaces-git-transport.test.ts` on Windows if you touch EOL config.
- **Ignores in `$GIT_DIR/info/exclude`, attributes in `$GIT_DIR/info/attributes`.** Nothing of ours is ever written into the user's tree (no `.gitignore`/`.gitattributes`). info/exclude is rewritten on every init; oversize exclusions appended after survive because excluded files stop being staged.
- **Convergent conflict policy: REMOTE wins the canonical filename; LOCAL content becomes a visible conflict copy** `name (from <device>, <date>).ext`. Both devices converge to identical trees. Local-wins-canonical never converges (each keeps its own canonical → re-conflict forever). Pinned by the transport contract suite.
- **`merge --allow-unrelated-histories` is load-bearing.** Two devices both with pre-existing content in one space (Personal on a second machine = mainline) have unrelated root commits at first contact; without the flag git refuses the merge and both are silently stuck (`updated:false`/`pushed:false`, no error). Companion: `pull()` adopts remote via `checkout -B main origin/main` when local `main` is unborn.
- **Conflict-copy content read as a Buffer via `showStage()` with `maxBuffer ≥ maxFileBytes`.** The string-returning `git()` helper must never carry file CONTENT: utf8 decode corrupts binary; Node's default 1MB maxBuffer kills `git show` on larger files — which made the conflict loop silently SKIP the local copy while `checkout --theirs` overwrote the file (silent data loss in exactly the scenario the convergent rule protects). Contract tests pin >1MB + binary conflict copies byte-for-byte.
- **`tests/sync-transport-contract.ts` is the transport compatibility boundary (10 tests).** Any new `SyncTransport` (SyncHub-assisted, YouCoded Cloud) must pass it unchanged; changing it = changing spec §15 (do both in one commit). The suite owns its own 30s `vi.setConfig` timeout — don't move that into global vitest config (would mask hung unit tests).
- **`repoNameForSpace` appends a short hash of the LOWERCASED space id.** The repo name IS the sync identity. The slug alone is neither unique ('My App'/'My-App' both slug to `my-app`) nor total (all-symbol/non-Latin → ''). Lowercased-id hashing keeps the name stable across devices while distinct folder names always get distinct repos.

## Engine & service (`engine.ts`, `service.ts`, `space-manager.ts`)

- **Engine invariants:** single-flight per space + exactly one coalesced rerun (don't add a second queue); `addSpace` waits for chokidar `ready` (with `ignoreInitial:true`, writes before `ready` are swallowed as initial-scan entries — verified on Windows); a persistent `watcher.on('error')` handler is required (post-ready errors bypass `'all'`; an unhandled EventEmitter `'error'` crashes main); `stop()` clears the state map FIRST then awaits in-flight sync chains (their git subprocesses hold handles inside the space root — on Windows that blocks folder removal and races app-quit teardown).
- **Enable/disable serialized through a module-level promise chain in `service.ts`.** `enable(true)` racing `enable(false)` would interleave — disable stops the half-started engine, resuming start adds watchers to a dead instance. `startEngine` is supersession-safe (bails + self-cleans when a newer transition replaced it).
- **`provisionGithubRemote` treats an already-existing repo as SUCCESS** (falls back to `gh repo view`). The provisioned-URL state file (`~/.claude/toolkit-state/sync-spaces.json`) is per-device, so the second device re-runs provisioning for repos the first created — without the recovery path sync could never start on any device but the first.
- **`isIgnoredPath()` (guards.ts) keeps backups scrub-consistent with sync.** The iCloud backup's cpSync filter interprets the same `DEFAULT_IGNORES` the git layer enforces — a narrower hardcoded filter once leaked `*.pem`/`id_rsa*`/credentials into iCloud while Drive excluded them. New backup targets must route through it (or derive rclone `--exclude` from `DEFAULT_IGNORES`).
- **Friendly gh error strings in `space-manager.ts` are a UI contract.** "GitHub CLI (gh) is not installed…" / "Not signed in to GitHub…" are shown verbatim by SyncPanel's error note (latest `error`-type event in `recentEvents`). Enable-path provisioning failures do NOT reject `enable()` — they arrive as broadcast error events; a new failure path must land in one of those two channels or the user sees nothing.
- **Android has NO `syncspaces:*` Kotlin handlers (deliberate, pre-Phase-3).** The shared renderer still works: remote-shim `invoke()` rejects after 30s and SyncPanel/FolderSwitcher catch + surface inline. When mobile starts, add SessionService `{ok:false, error:'not-implemented-on-mobile'}` stubs first so invokes reject fast instead of timing out.

## SyncHub — instant signals (Plan 1b; `sync-hub-socket.ts` + worker `SyncGroupRoom` DO)

Metadata-only `{kind, spaceKey}` signals relayed between one account's devices → immediate pulls.

- **DO is per-account (`idFromName(userId)`), never global.** Unlike the social `PresenceRoom` (one global room + D1 friend graph), SyncHub has NO D1 and no cross-account visibility (worker test pins isolation). Auth in the Hono route (`requireAuth`); the DO trusts only internal `X-Sync-User`/`X-Sync-Device` headers — never add a direct DO entry point.
- **The DO is an ACCELERANT, never a source of truth.** Losing its state loses nothing (devices reconcile-on-connect + the engine's 120s poll is the fallback). Don't "optimize" the poll away — SyncHub down must only make sync less instant, never block it.
- **spaceKey across devices = `repoNameForSpace()`, NEVER the local space id.** Local ids (`project:<folderName>`) differ across devices by folder-name case; the repo name (slug + sha1 of lowercased id) is the stable cross-device identity. The service resolves inbound spaceKeys back to spaces by comparing repo names.
- **Signal ONLY on `pushed:true` — the loop breaker.** A signal-triggered pull that pushes nothing broadcasts `pushed:false` and re-signals nothing → converges. Don't widen the guard in `broadcast()` (it also keeps `hub-status`/`error` events from recursing into a send).
- **The hub send in `broadcast()` runs LAST + isolated (own try/catch).** `broadcast()` is the single fan-out chokepoint from the engine's `onEvent`; a hub failure must never block/kill local window or remote delivery. Signalling is best-effort — the poll covers misses.
- **A superseded `startEngine` run owns NO global state.** The supersession guard (`engine !== e`) sits BEFORE any hub work; everything after is synchronous. Moving hub-socket creation or `hubStatus` writes above it reintroduces the boot-race clobber (a stale run stamps `'off'` over a live socket's `'connected'`) — race-pinned by a service test using the `initialEnabled` boot knob.
- **`hub-status` events carry the sentinel `spaceId: 'hub'`** — never collides with real space ids, so `sync-dot-state.ts`, ProjectHero, per-space scans ignore them naturally. SyncPanel filters them from visible notices (the "Instant sync" line is their surface) but keeps them in `recentEvents`.
- **The relay `kind` allowlist (`ALLOWED_KINDS` in worker `sync/room.ts`) is where Phase 2 lease/takeover kinds get added** — the relay/ring logic is kind-agnostic; extending is a one-line change.
- **`device` is a client-chosen label (`?device=`), connection-pinned — NOT authenticated identity.** Two devices can send the same label; lease logic must never key on it as verified/unique.
- **Any new DO binding must be mirrored under `[env.test.durable_objects]`** in wrangler.toml (env.test doesn't inherit) or every worker test fails; a new DO class needs a NEW `[[migrations]]` tag (v2 added `SyncGroupRoom`; v1 untouched; migrations NOT repeated in env.test).
- **The hub socket is SERVICE-owned (works with zero windows open)**, unlike the renderer-driven presence socket. Its no-token path retries on a 30s timer (clear-then-reschedule — a fired timeout leaves the var non-null, so an `if (!retryTimer)` guard kills the poll after one cycle; pinned by a socket test) so mid-session sign-in connects without restart. Auth arrives via `setSyncSpacesAuthStore` wired in main.ts BEFORE `startSyncSpaces`.
- **Enable-time pushes send no signal** (socket not open yet) — other devices catch up via poll. By design.

## Import existing folders (`sync-spaces/import-project.ts`)

"Sync this project" + folder-picker import both MOVE the folder into `~/YouCoded/Projects/<name>/`.

- **Import MOVES the folder — never copy-and-keep-both.** The only copy is the EXDEV (cross-drive) fallback, which deletes the source afterward and WARNS if that delete fails. A silent surviving copy forks the user's work (muscle memory edits the old one while the synced copy rots).
- **The EXDEV branch re-checks `existsSync(dest)` BEFORE cpSync — load-bearing.** `rename(2)` reports EXDEV before it looks at dest, so a dest created in the checkImport→move window (the engine materializing a same-named project from another device; dev + built app both running) would be silently MERGED into by cpSync then DELETED by the copy-failure cleanup. The guard refuses instead. Never remove it; never let the failure cleanup run against a dest this call didn't create.
- **The CC transcript slug dir is renamed on move (`remapTranscriptDir`).** `~/.claude/projects/<slug>/` is keyed by a slug DERIVED from cwd (not stored), so past conversations orphan silently unless renamed to the new slug (via `ccProjectSlug`, drive-case-normalized). When the new slug dir exists, merge file-by-file, never clobber.
- **Remap failures degrade to WARNINGS, never errors.** After a successful move, the four store remaps (saved-folders path, central-index path, sidecar manualIncludes/Excludes prefixes, transcript slug dir) are each individually try/caught; a failure means stale pointers, never lost content. Every failure MUST surface a user-facing warning (the sidecar CAS-miss throw exists so even that lands in a warning) — don't reintroduce a silent drop.
- **`remapProjectPath` (central-index) drops any stale entry already at the destination path.** Two index entries sharing one `path` (the store key) shadow each other and detach artifact history; the filter inside the `mutateIndex` lock is the guard.
- **`countFilesBounded` is bounded three ways: file count, depth (100), wall-clock (2s).** `isSymbolicLink()` does NOT detect NTFS junctions — a junction cycle with no files never trips the file-count limit and recurses forever without the depth cap. Depth/time exhaustion returns the OVER-limit signal (fail closed), not a partial count (a partial would let a monster tree into the engine).
- **`FOLDERS_LIST` badges managed rows by projectsRoot PREFIX, not only via the synthesized merge.** The import flow rewrites the saved entry's path to its managed home; prefix-badging keeps the "synced project" label on it.
- **`saved-folders.ts` is the single store module for `~/.claude/youcoded-folders.json`** (`writeFolders` atomic via unique-tmp + rename since dev + built app share `~/.claude`). Known duplication: `remote-server.ts` still hand-rolls the format in its folders:* cases — migrate if you touch those.
- **`ImportProjectModal` close semantics are state-dependent.** Pre-move: Cancel/Scrim/ESC all cancel (ESC gated `!busy`, matching Scrim + Cancel — no dismiss path may bypass the busy guard mid-move). Post-success (warnings shown): EVERY dismiss routes to `onDone(path)` so the folder list reconciles; only pre-move may bare-cancel.

## Project & sync management UX + cross-device discovery

Session picker slimmed to rows + dots + one footer; Project View is the management hub. Discovery/rename/stop: `sync-spaces/{project-registry,materialization-planner}.ts`.

- **Sync dots (green/red/gray) are the ONE sanctioned status-color use.** The tooltip/aria-label carries the full plain phrase; the no-●◐○ rule is about glyph TEXT, not colored dots (Destin chose the dots for the compact picker). All three surfaces (FolderSwitcher, ProjectSwitcher, ProjectHero) derive from the pure `renderer/components/sync-dot-state.ts` — don't re-derive inline; don't change its label strings without the spec (tests byte-pin them).
- **The session picker has NO add/create/import actions** (Project View owns adding, via the "Manage projects…" footer). EXCEPTION: when `onManageProjects` is absent (the buddy window, no Project View), the footer falls back to a minimal "Browse for folder…" escape hatch — delete it if buddy ever gains Project View, don't let it become a second add path.
- **FolderSwitcher's dropdown is PORTALED to document.body at zIndex 9001.** The host SessionStrip menu is the documented z-[9000] exception; a popover spawned FROM it must render above its host, and the menu's `overflow-hidden` clips a non-portaled child. Don't "fix" either z-index.
- **The `data-folder-switcher-portal` marker on the panel is load-bearing (youcoded#113).** A HOST menu's outside-click handler can't see the portaled panel via `ref.contains()`, so SessionStrip checks `target.closest('[data-folder-switcher-portal]')`. Without it a mousedown on the dropdown counts as "outside", the menu unmounts the picker BEFORE its click fires — row selection silently does nothing.
- **Picker rows are pick-only** — rename/remove live in Project View (hero Rename/Remove). Don't re-add per-row actions.
- **`SpaceSyncEvent.at` is stamped ONLY in `service.broadcast()`** (ms epoch at emit) — the SOLE source for "Last synced N min ago". No persistence; goes blank after restart until the next sync. By design.
- **`syncSpacesSyncNow(spaceId?)` narrows to one space; no-arg syncs everything.** Rides all five parity surfaces (service/ipc-handlers/preload/remote-shim/remote-server). SyncPanel's no-arg call is the compat contract — don't make the arg required.
- **ProjectView refreshes sync status live via `syncSpaces.onEvent`** (500ms trailing debounce, gated on view open). Without it "Sync now" produces zero visible feedback.
- **The project registry lives at `~/YouCoded/Personal/ProjectSync/<name>.json` — VISIBLE per-file, NEVER under `.youcoded/`** (that basename is the hidden git dir AND a `DEFAULT_IGNORES` entry → never syncs). `sync-spaces-project-discovery.test.ts` pins round-trip through the transport.
- **The registry record is convergent + mutable** (`{schemaVersion,name,repoName,displayName,state,updatedAt}`): `name` is immutable identity (folder name); `state` is **`stopped`-dominates monotonic** (NOT LWW — a stale rename on a device that hasn't pulled the stop must never un-stop it; consequence: no Resume without a per-field state timestamp); `displayName` is LWW by `updatedAt`. Reuses `laterOf`/`isConflictCopyName`/`extractConflictBase` (`conversations/store-core`) + `mutateFileUnderLock` (`artifacts/cas-write`).
- **Fold-on-read is load-bearing.** The transport's remote-wins-canonical can leave the WRONG winner as canonical `<name>.json` (a peer's older `active` over a local `stopped`). `readProjectRegistry` folds every conflict copy into the canonical in memory (stopped dominates) so a stopped project can't read active and resurrect. Copy files left in place (inert). Don't optimize away.
- **Rename is display-name only; the folder never moves** (`repoName` derives from the folder name → no ULID). True on-disk folder rename deferred (spec §15).
- **Stop = tombstone + `engine.removeSpace` + KEEP the folder.** `activeManagedSpaces()` is the ONE gate, routed through all three `roots.spaces()` sites (startEngine add loop, hub-connected sync loop, daily-backup loop) via the service's `activeSpaces()` helper. Don't filter at the sites individually; don't make `ManagedRoots.spaces()` registry-aware.
- **`planReconcile`'s third arg is live project NAMES, not space ids.** `runDiscovery` converts `engine.liveSpaceIds()` `'project:app'` → `'app'` before passing. Raw space ids silently empty `toStop` (a stopped project never detaches).
- **`ensureRemote` runs FIRST in materialize** (uses only `space.id`, so a gh-auth failure creates nothing); `addSpace` runs BEFORE the first pull (a failed pull leaves a recoverable empty space). Materialize reuses the empty-folder + first-sync-pull path — NO `git clone`.
- **`runDiscovery` is single-flight + one coalesced rerun**, reads the registry ON DISK (triggers guarantee freshness). Don't make it sync Personal itself (recurses through the broadcast trigger).
- **Rename/stop ride four-surface IPC parity** via `syncspaces:rename-project`/`stop-project` (`{name,displayName}`/`{name}`); `IPC.SYNC_SPACES_*` in `shared/types.ts` AND the inlined preload map; `ipc-channels.test.ts` pins it. Android stubs `not-implemented-on-mobile`.
- **UI: the status dot is the sync/stopped signal; location ≠ status** (synced + unsynced both live in `Projects/`). `sync-dot-state.ts` adds a gray "Sync stopped"; the ProjectHero "Stop syncing" button is consequence-gated; rows overlay synced `displayName` from the status payload (single source of truth — never written back into `youcoded-folders.json`).

## Legacy backup + demolition (Plan 2c — branch `feat/sync-legacy-demolition`, NOT YET MERGED to youcoded master as of 2026-07-15)

Modules `snapshot-retention.ts`, `conversations/symlink-sweep.ts`, `sync-spaces/gc-policy.ts` do not exist in master yet — these invariants apply once the branch lands.

- **`sweepProjectSymlinks()` is `lstat`-only, removes ONLY symlinks/junctions, NEVER recursive.** Walks `~/.claude/projects/*` one level deep. Removal is `unlinkSync` → `rmdirSync` fallback (Windows dir junctions need rmdir) — NEVER `rm -rf`/`{recursive:true}`, because recursion through a junction deletes the TARGET's real transcripts (the workspace `git worktree remove`-follows-junctions lesson). Per-entry try/catch; idempotent (no once-marker); wired detached in `main.ts`. Don't switch to `statSync` (follows links) or add recursion. **(Highest-consequence: a wrong refactor here irreversibly deletes real conversation transcripts.)**
- **Drive/iCloud backup is WRITE-ONLY dated snapshots — restore was REMOVED, not rewired.** The `~/.claude` personal set (memory, encyclopedia, settings, skills, config — NOT conversations, which sync-spaces `DailyBackup` covers) writes to `Backup/<YYYY-MM-DD>/`, gated by a once-per-UTC-day stamp, with tiered grandfather-father-son pruning (`snapshot-retention.ts`: ≤7d all → 8–28d newest-per-ISO-week → 29–90d newest-per-month → >90d delete; unparseable folder names NEVER pruned; pruning targets EXACT dated paths via rclone `purge`, never a wildcard). The entire restore path was DELETED (Restore Wizard UI, 9 `sync:restore:*` IPC channels, `restore-service.ts` + adapters). Don't re-add a Restore Wizard or auto-restore-on-launch pull — Drive/iCloud are backup-only; GitHub space sync is the live recovery. (Android Kotlin restore backend left as dead handlers; Phase-3 follow-up demolishes it.)
- **The personal snapshot has an AUTOMATIC daily trigger — an hourly poll, NOT the deleted 15-min loop.** A re-introduced hourly `setInterval` in `SyncService.start()` (+ one launch call) calls `push()` NON-force, so the 15-min `debounceCheck` throttles and the once-per-day stamp gates the dated copy → at most one snapshot/UTC day.
- **The >500MB sync-history warning rides a dedicated `notice` `SpaceSyncEvent` kind, NOT `error`.** `error` drives the red dot; a healthy-but-large space must not flash red. `notice` is emitted after a successful sync (one-per-launch, Set-deduped), IGNORED by `sync-dot-state.ts`, surfaced non-alarmingly in SyncPanel. `git gc` here is **local `git gc --auto` only** (every 50th successful sync per space, counter in the space's `.youcoded/gc-counter`) — repacks local objects, NEVER rewrites history or desyncs peers. Remote/history compaction stays a DEFERRED manual procedure.

## Sync Warnings (`sync-service.ts`, `sync-error-classifier.ts`)

- **`~/.claude/.sync-warnings.json` is authoritative** — a typed `SyncWarning[]` consumed by SyncPanel, StatusBar chips, gear red dot. The legacy string-code `~/.claude/.sync-warnings` is still written by bash `statusline.sh` for the terminal statusline only.
- **Two writers, non-overlapping codes.** `runHealthCheck()` owns `OFFLINE`, `PERSONAL_NOT_CONFIGURED`, `PERSONAL_STALE`, `SKILLS_UNROUTED`, `PROJECTS_UNSYNCED`. Push methods own per-backend codes (`CONFIG_MISSING`, `AUTH_EXPIRED`, `QUOTA_EXCEEDED`, `NETWORK`, `RCLONE_MISSING`, `UNKNOWN`) + a `backendId`. Health-check merge replaces only its own codes each run — a push-path code must never appear in `runHealthCheck`.
- **`.sync-error-<backendId>` is retired** — deleted on startup by `cleanupStaleBackendErrorFiles()`; all per-backend error state lives in `.sync-warnings.json` keyed by `backendId`. Don't reintroduce.
- **Push-failure warnings are non-dismissible** — `dismissWarning()` enforces `dismissible: false` server-side (silencing "Google Drive isn't connected" risks data loss). No force-dismiss path.
- **Remove a backend → clear its warnings** — `removeBackend()` calls `clearWarningsByBackend(id)`; skipping leaves a phantom warning with no way to dismiss.
- **Classifier stderr patterns are rclone-version-sensitive** (`sync-error-classifier.ts`). Major rclone bumps can silently drop patterns to `UNKNOWN`; `sync-error-classifier.test.ts` is the source of truth for which stderr each code matches.
- **Node-killed timeouts produce empty stderr — use `extractStderr()`.** `execFileAsync(..., {timeout})` SIGTERMs the child before it flushes stderr, so the rejection has empty `e.stderr` + only `Command failed: <cmd>` in `e.message`. Naively returning `e.stderr || e.message` classifies every timeout as `UNKNOWN` — which broke Drive backups when the youcoded-dev slug grew past 156 MB. `extractStderr(e, timeoutMs)` checks `e.killed && e.signal` and prepends a `TIMEOUT_SENTINEL`. Don't reintroduce raw `e.stderr || e.message` in `rclone()`/`gitExec()`/future helpers.

## Connect-GitHub Modal (`github-auth.ts`, `github-connect.ts`)

In-app GitHub device-flow auth so non-developers never hit the gh-missing/not-signed-in dead end when enabling Sync.

- **The access token NEVER leaves the main process.** It flows ONLY into `gh auth login --with-token`'s stdin — never logged, never in a thrown Error, never in the `github:connect-done` payload, never over remote WS. Both `github-auth.ts` and `github-connect.ts` have a test forcing a failure whose message contains the token and asserting it's absent from the surfaced error/payload.
- **Reuse gh's own public client id (`178c6fc778ccc68e1d6a`) + gh's default scopes** — the minted token is indistinguishable from a normal `gh login`, so sync's `gh repo create|view` + `.netrc` keep working. Don't stand up a second OAuth app.
- **Do NOT wrap `gh auth login --web` (interactive).** The device flow + `--with-token` is what gh does under the hood, is non-interactive, gives us the code/URL to render, and works while gh is still installing. `completeLogin` MUST `stdin.end()` after writing the token or gh blocks forever on EOF.
- **The orchestrator is a module SINGLETON shared by desktop + remote** (`setGithubConnect`/`getGithubConnect`) so a remote `connect-start` + its `connect-cancel` drive the SAME flow. Its settle guard is PER-FLOW (monotonic `activeFlowId`), NOT a single `settled` bool: two starts without an intervening cancel must let the aborted prior flow no-op while the new one settles — else a superseded flow emits a spurious `cancelled` that blocks the real done. Pinned by the "superseding start silences the aborted prior flow" test.
- **gh-missing install is winget→restart, never poll.** Windows: `detectWinget()` (import from `prerequisite-installer.ts`) → `winget install --id GitHub.cli`; if gh still not found, PATH hasn't propagated to the running Electron process → return `restartRequired`. No PATH-rebuild loop. mac/linux return a `manual` one-liner.
- **SyncPanel triggers gate on the STRUCTURED `github:status`, never on parsing space-manager's error strings.** The "Connect GitHub…" button is additive beside the pinned error strings and keyed on `!githubStatus.authed`. After a successful connect the panel auto re-kicks a failed enable (`wasMidEnableRef`, cleared before re-invoke — one retry, no loop).
- **Modal cancels main's poll on EVERY user-initiated close** (X/Scrim/ESC/"Never mind" → `connectCancel()`); the success auto-close path calls `onClose` directly (no cancel — no live flow, and a cancel would emit a spurious `cancelled`). `navigator.clipboard` is undefined on plain-HTTP remote (non-secure context) so Copy no-ops there — `select-all` for manual copy is the fallback.
- **4-surface IPC parity + Kotlin stub.** `github:status`/`connect-start`/`connect-cancel`/`install-gh` request channels; `github:connect-done` push asserted in preload + remote-shim only (Android never emits it, matching `session:moved`). Android's four request channels are a combined `not-implemented-on-mobile` stub.
