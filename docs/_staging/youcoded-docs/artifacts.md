# Artifact Viewer — depth

The artifact system tracks every file Claude Write/Edit/Deletes into per-project sidecars + a central index, and renders them in the chat Session Drawer and the Project View hub. `ArtifactStore` state lives in the renderer (`state/ArtifactContext.tsx`, `state/artifact-tracker.ts` subscribes to TranscriptWatcher events alongside the chat reducer); all file I/O is main-process behind `window.claude.artifacts.*` IPC — the same pattern as the chat reducer, which is what lets the shared React UI render identically on desktop + Android.

## Tracking pipeline

- **`artifacts:append-version` is the IPC that populates the sidecar.** `listSession` is read-only (filters the sidecar by sessionId). Without the renderer Tracker calling `appendVersion` on every Write/Edit/MultiEdit tool-use event, the sidecar stays empty. Chain: Tracker → `appendVersion` → `ensureProject` → `applyGitTreatment` → broadcast `changed`. First thing to check when artifacts don't materialize.
- **Transcript events don't carry `cwd`.** The watcher emits `{ type, sessionId, uuid, timestamp, data }`; the Tracker resolves cwd via a `sessionsRef` lookup into `sessions` state. Reading `event.cwd` always bails silently.
- **`bash mv`/`bash rm` produce ZERO artifact events** — the Tracker only sees Write/Edit/Delete tool calls. Bash filesystem side-effects are opaque in v1. Because CC has no Delete tool, files are usually removed via `bash rm` → the record stays `status:'active'` with the file gone ("orphan"). Badges/counts subtract orphans via `checkExistence`.
- **`sessionArtifacts` is backfilled from the sidecar when a session is active** (`ChatView` calls `listSession(sessionId, cwd)` ungated by drawer-open) — the live tracker only APPENDS, so after a reload the drawer showed "0 artifacts" until the next write. Don't remove.
- **`.youcoded/` is auto-gitignored** by `ProjectManager.applyGitTreatment()` (TS + Kotlin) on first sidecar materialization in a git repo — agent file-touch history doesn't leak into commits. User opts in via Settings → Privacy.
- **`transcriptRef` is NOT in v1** — dropped because CC's transcript format is version-coupled and CC compacts/rotates. v1 reads current content from disk; v2's version history will snapshot into `<project>/.youcoded/content/<versionId>` at write time. Don't reintroduce.

## Concurrency (`cas-write.ts`)

- **`casWrite` uses a mkdir-based lock, not just compare-and-set.** The initial CAS-then-rename had a TOCTOU race (two writers both pass CAS, both rename — one overwrites; commit `2d44fc3d`→`a75492ea`). Both TS + Kotlin (`CasWrite.kt`) use a `<target>.lock` dir + 30s stale-lock breaking. Don't simplify the lock away — the bug WILL recur.
- **Central-index writers use `mutateFileUnderLock` (read-modify-write INSIDE the lock)** — the index has no CAS version field, and the dev instance + built app share `~/.claude` (cross-process interleaving is normal). `casWrite` + `mutateFileUnderLock` share `acquireLock`/`atomicWrite`.
- **`appendVersion` retries CAS `MAX_RETRIES=5`** — intentional; increasing it doesn't help under sustained contention (surfacing a warning is the escape valve). Same on Kotlin.

## Artifacts vs All files (2026-06-15 split)

- **CORE: two DISTINCT concepts, never merged; All files is ALWAYS a superset.** Artifacts = files Claude created/edited (sidecar, `kind:'internal'` or manually-included external). All files = every real file on disk (all types incl. source code). A Claude-authored file legitimately appears in BOTH. `LIST_PROJECT` returns tracked ONLY; `LIST_ALL_FILES`/`countAllFiles` go through `projectAllFiles(root)` = discovery UNIONed with tracked-on-disk internal artifacts (deduped by `canonicalize()`) so it can never report fewer files than Artifacts (a real bug: a code-heavy/nested-repo project had more artifacts than "all files"). The union restores artifacts inside a skipped nested sub-repo.
- **Renderer shows the split as a 4th segment `Artifacts | All files | Conversations | Context`**, both file tabs rendered by one `FilesTab` with a `mode` prop. Both are a navigable folder TREE (`listDir(files, currentDir)`): single files sort before folders; breadcrumb navigates up; search FLATTENS. The seg-row search pill + type filter (`fileTypeGroup`) + sort + "Hide code & configs" (DEFAULT ON, suspended under the Code filter) live in `FileFilterPopover`; search matches FILE NAMES only. Badge counts stay folder TOTALS, not filtered counts. Show-deleted/Add-file chips gate to the Artifacts tab.
- **Discovery (`project-file-discovery.ts#discoverProjectFiles`) scans every file type** (only `isNoiseFile` byproducts skipped — lockfiles, `.map`, `.min.js/.css`; NO extension allowlist), excludes `node_modules`/dot-dirs/`SKIP_DIRS`, never follows symlinks, **STOPS at nested git repos** (`.git` dir OR file — so youcoded-dev lists its own tree, not a crawl through cloned sub-repos), hard-bounded (file/dir/depth caps + 1.5s budget), cached 10s. The nested-repo stop makes the count DETERMINISTIC (before it: hero said 282, switcher said 1209). Discovered records carry `discovered: true` (NEVER persisted) — consumers skip them in `checkExistence` (ids are relative paths, not sidecar ids → wrongly flagged missing) and count them as always-present. "Exclude" is hidden for discovered files.

## Project list & counts

- **Project list = SAVED FOLDERS (`~/.claude/youcoded-folders.json`), NOT the central index.** `LIST_PROJECTS_INDEX` builds via `buildSavedFolderProjects(savedFolders, indexProjects)` (pure): each saved folder is a project; a central-index match by canonical path is reused (real ULID + stats), else a SYNTH entry whose `id` IS the canonical path. The index only gains an entry once Claude writes a tracked artifact, so folders with only conversations were invisible. Index-only projects that aren't saved folders are DROPPED.
- **`LIST_PROJECT`/`GET`/`SAVE` fall back to id-as-PATH (traversal-guarded)** — synth projects have no index entry; `path.resolve(projectRoot, artifactId)` REFUSES anything escaping the root. This is also how discovered files load/edit by path.
- **Two single-source count helpers in `ipc-handlers.ts`:** `countArtifacts(root)` (non-deleted tracked files still on disk — orphans excluded) = the ARTIFACTS number; `countAllFiles(root)` (discovery length) = the ALL FILES number. Hero segment badges/stat line + switcher row each read the right one. Don't recompute inline (the 282-vs-1209 drift). The sidebar/hero count is LIVE, never the stale seeded `stats.artifactCount`.
- **Project switcher rows show "N files · M chats" where files = `fileCount` (ALL FILES)**, via `listProjectsIndex({ withCounts: true })` — a heavy global `listPastSessions` + per-project counts, too heavy for ChatView's frequent cwd-resolution (uses the fast no-arg form). `fileCount`/`conversationCount` on `CentralIndexProject` are read-time only, never persisted.
- **Conversations/Context tabs are CONTROLLED** (data lifted to `ProjectView`, per-project cached). `listProjectConversations` uses a bounded 64KB head-read per session (NOT `loadHistory(all)` — the dominant stall). Hero runs five IPCs in `Promise.all`.

## Visibility (`visible-artifacts.ts`, 2026-07-08)

- **`trackedArtifacts()` is the ONLY place Artifacts-tab visibility is decided.** Order: manually INCLUDED (any kind — includes WIN over excludes, so "+ Add file" recovers a mistaken exclude) → manually EXCLUDED hidden (sticky across Claude re-edits) → internal with ≥1 NON-READ version → externals hidden. Pill-click `read`-only views deliberately don't appear in Artifacts (they live in All files + the drawer). Include/exclude entries are canonical ABSOLUTE paths — canonicalize BOTH sides (Windows drive-case made "Add file"-ed externals vanish).
- **`manualExcludes` MUST stay wired into `trackedArtifacts`** — it shipped write-only once (Exclude button inert). `EXCLUDE` un-pins + adds the exclude; `INCLUDE_EXTERNAL` also ENSURES a record exists (`appendVersion` `read`/`user`).
- **`SessionDrawer` excludes `discovered` records from `checkExistence`** (`filter(a => !a.discovered)`) — relative ids hit the "unknown id → missing" branch and wrongly flagged on-disk files as deleted. SessionDrawer resolves the OPEN document against the UNFILTERED list (toggling a filter must not blank the content pane).
- **`artifacts:changed` broadcasts carry the REAL artifact id** (`appendVersion` returns it) — it was `null`, so the "Claude also edited this file" conflict banner could never fire. A `read` version records the view but does NOT bump `lastModified` (bumping reordered "recently modified" on every pill click). `artifacts:remove-record` deletes the RECORD only, never the file.

## Binary viewers & security (2026-06-17 / 2026-07-08)

- **`artifacts:read-binary` is GUARDED on desktop** (`read-binary-access.ts`, pure + tested): reachable over the remote WS, so reads are restricted to (a) saved-folder + central-index roots, (b) tracked external paths/manual includes, with well-known secret locations (`.ssh`, `.gnupg`, `.aws`, `.netrc`, `.credentials.json`, `.config/gh`, …) refused even inside allowed roots (the seeded "Home" saved folder makes most of home a root — the sensitive check carries the weight). 50MB gate (`too-large`) since base64 inflates 33% over IPC/WS. Kotlin handler is unguarded but the Android bridge is localhost-only; port the guard first if Android gains a remote transport.
- **Binary viewers load bytes via `artifacts:read-binary` (base64), NEVER `fetch('file://…')`** — the renderer origin can't fetch `file://` (fails silently). `useArtifactBytes(absolutePath)` → `atob` → `Uint8Array`. `ImageView` builds a `blob:` URL. Full 4-surface parity pinned by `ipc-channels.test.ts`.
- **All binary viewers go through the `BinaryContent` shell** (`artifact-views/BinaryContent.tsx`) — owns loading/error (maps `orphan`/`too-large`/`not-allowed`/`unavailable`), keys the inner viewer by `absolutePath`. PdfView once ignored loading/error → blank pane on a corrupt PDF; it must `loadingTask.destroy()` in cleanup. **`ViewerErrorBoundary` wraps the lazy render in `ActiveArtifactView`** — `React.lazy` THROWS chunk-load failures that `<Suspense>` can't catch.
- **xlsx viewer is ExcelJS, not SheetJS (removed).** SheetJS free can't read cell styles; ExcelJS reads them free. `exceljs-cell.ts` (pure) maps cell → text + CSS; `XlsxView` renders a DOM grid padded to ≥50×26. **Formula cells must be evaluated** — openpyxl/pandas write formulas without cached results (`{formula}` → `[object Object]`). `xlsx-formula.ts` evaluates (memoized, cycle-guarded; IF short-circuits; /0 + non-finite THROW → blank, never `Infinity`; unsupported fns → blank). In-app xlsx EDITING is deliberately deferred (canvas engines break the theme-token system). CSV/TSV render via `CsvView` (text path, `csv-parse.ts`). `xls`/`ods` fall back to "Open externally" (`shell:open-path`, desktop-only).

## Filepath pills & attachments

- **Clicking ANY filepath pill "artifactifies" it and ALWAYS opens the artifact viewer, NEVER Project View.** `FilepathToken` order: session match → project (`listProject`/`listAllFiles`, in parallel) match → else `appendVersion` (`user`/`read`) + `listSession` + select. So a file Claude made via Bash/python (untracked) or in a temp dir still opens. cwd comes from `sessionCwd` (`SET_SESSION_CWD` from ChatView). The old `PROJECT_VIEW_OPENED` fallback was the bug — don't reintroduce.
- **`findBestMatch` (`components/filepath-match.ts`) prefers EXACT path matches over suffix**, case-insensitively (exact+suffix OR'd let a same-named file shadow the target). `buildArtifactifyArgs` resolves `..` via `canonicalize()` before internal/external classification; returns `null` for `~/` paths.
- **`detectFilepaths` accepts sentence-final `./!/?`** (only before whitespace/end) and rejects protocol-less domains via a short TLD list (so dotted dir names keep working). `html`/`htm`/`svg` are in the detector whitelist.
- **User-message attachment pills ride `ChatMessage.attachments` (exact picker paths), NOT regex** — picker paths contain spaces the space-joined `content` can't split back out. `content` stays the space-joined string (transcript dedup matches on it). Live-bubble only.

## Project View hub (2026-06-14 redesign)

- **Detail is a centered overlay** (`ProjectDetailOverlay`, `<Scrim>`+`<OverlayPanel>` at `fixed inset-2 sm:inset-8 md:inset-16`, closes via `useEscClose`), NOT a side pane. `ActiveArtifactView` is the shared viewer (SessionDrawer + ProjectView) — surface differences via props, never fork. `ProjectDetailOverlay` has `tools` + `meta` header slots.
- **No project rail — switching goes through the palette (`ProjectSwitcher`).** The hero project name is the trigger and MUST NOT truncate (`whitespace-normal break-words`). Project cards MUST be fixed-height (`h-44`) with `shrink-0` thumbnail+name (else flex-shrink collapses them into blank pills). `ProjectView`/`ProjectSwitcher` register in the `useEscClose` LIFO stack; the delete-confirm is L3 via `Scrim`/`OverlayPanel`.
- **Per-session drawer state keyed by `sessionId`.** `artifactState` holds `drawerOpenBySession` + `activeArtifactBySession` (NOT global). New sessions start closed; switching restores each one's open-state + selection. `ProjectView` keeps its selection under the reserved sessionId `'project-view'`. `drawerExpanded` stays GLOBAL (one drawer mounted at a time). Thread `sessionId` through any new open/select dispatch.
- **Framed-shell chrome overlays the content region; drawer-pane insets via margins.** HeaderBar/bottom chrome are absolute at `z-index:20`; `.chat-scroll` pads for them. `.framed-shell > .drawer-pane` carries `margin-top/bottom: var(--top/bottom-chrome-height)` — margins on the DRAWER only, not chat-pane (chat-shift was the "jank" that killed Option C). The drawer is layout-level, NOT an overlay — don't wrap `SessionDrawer` in `<OverlayPanel>`.
- **Status load-timing badges are plain WORDS, never `●◐○`** (`Always`/`When editing <glob>`/`On recall`/`Index`) — the user dislikes glyphs anywhere in the artifact/context UI.
- **Context discovery is pure-core / IO-shell** (`project/context-discovery.ts#discoverContext` is pure + tested; `project-context.ts` does I/O). `RECOGNIZED_INSTRUCTION_FILES` = `CLAUDE.md`/`AGENTS.md`/`GEMINI.md` (format-agnostic). The `.git/config` `url =` parse + rule `globs:` regex are line-anchored (both caused wrong-value bugs).
- **`project:*` IPC is desktop-only** (`project:list-conversations`/`conversation-history`/`repo-info`/`list-context`/`read-context-file`/`write-context-file`) — STUBBED in `SessionService.kt` (`not-implemented-on-mobile`), parity pinned by `ipc-channels.test.ts`. **`read-context-file`/`write-context-file` are ALLOW-LISTED to the discovered context set** (`isAllowed` re-runs `discoverContextGroups` + canonical compare — a security boundary). The global-file blast-radius confirm is enforced in the renderer (`ContextEditorOverlay`).

## Android parity

- **Android `artifacts:list-project`/`:list-all-files`/`:list-projects-index`/`:include-external`/`:exclude`/`:delete-project`/`:rename`/`:remove-record` are no-op stubs** returning `{ok:false,error:'not-implemented-on-mobile'}` — the CONTRACT, not a bug. `read-binary`, `get`, `save`, `append-version`, `check-existence`, `list-session` ARE real on Android. Mobile Project View is v2. (`canonicalize()` runs on both platforms via the shared fixture; Kotlin `ProjectManager.kt`/`ProjectView` count mirrors pending an Android cleanup pass.)
