# Marketplace Worker тАФ backend depth

The Cloudflare Worker at `wecoded-marketplace/worker/` backs marketplace install counts, ratings, accounts/social, sync-hub signalling, and privacy analytics. It auto-deploys on push to master via `.github/workflows/worker-deploy.yml` (runs `npm test` тЖТ `wrangler d1 migrations apply --remote` тЖТ `wrangler deploy` тЖТ `wrangler secret put` for every required secret). Credentials live in repo secrets (`CF_API_TOKEN`, `CF_ACCOUNT_ID`) тАФ no local `wrangler login`.

## Deploy & config

- **Never put a key in both `[vars]` and `wrangler secret put` with the same name.** Every `wrangler deploy` lets the wrangler.toml var win and silently clobber the secret. Symptom: OAuth breaks in prod because `GH_CLIENT_ID` ends up as the literal placeholder string. Fix: leave shared secrets OUT of wrangler.toml entirely тАФ define them only as secrets. The missing-var pattern is load-bearing; don't "helpfully" add a placeholder.
- **In CI, `wrangler deploy` must run BEFORE `wrangler secret put`.** Deploy reconciles the Worker's bindings to wrangler.toml, removing stale vars from earlier deploys. A stale same-named var makes `secret put` error "Binding name already in use" (code 10053). Safe order: `migrations apply --remote` тЖТ `deploy` тЖТ `secret put`.
- **`[vars]` flips (e.g. `CUTOVER_TIMESTAMP`) ship the same way** тАФ edit wrangler.toml, commit, merge; CI auto-deploys. A new secret means a Repo тЖТ Settings тЖТ Secrets entry plus a one-line addition to the workflow's `Push secrets` step.
- **`[env.test]` is required for `@cloudflare/vitest-pool-workers`** (0.5.x) and must OMIT bindings it can't resolve тАФ wrapped bindings like `[ai]` and `APP_ANALYTICS`. Consequence: any code path depending on `env.AI`/`env.APP_ANALYTICS` must fail open (`?.`) or be test-covered another way.
- **Mutating `env` in a test body does NOT propagate to `c.env` inside a running worker request** тАФ the runtime snapshots env at start. Values that must flow to route handlers (`CUTOVER_TIMESTAMP`, `KNOWN_DEV_DEVICES`) go in `[env.test.vars]`, not `(env as any).X = ...`.
- **Any new Durable Object binding must be mirrored under `[env.test.durable_objects]`** (env.test doesn't inherit) or every worker test fails; a new DO class needs a new `[[migrations]]` tag.
- **Rate limits via the Cache API are per-colo, not global.** `checkRateLimit()` uses `caches.open("rl")`, which is edge-local. Fine for casual abuse prevention; true global throttling needs a Durable Object or a D1-backed counter.
- **Worker JSON 500 via `app.onError`** тАФ non-`HTTPException` errors return `{ ok:false, error:<message> }` at status 500 (not Hono's default plain-text), so a failing AE SQL query carries the CF error in the response body. `HTTPException` responses (400/401/403/429) still emit their own status+body unchanged.

## Analytics (device-hash telemetry)

Opt-outable anonymous device-hash + DAU/MAU via `env.APP_ANALYTICS` (Analytics Engine). Current design: the device-hash redesign (superseded the original install_id design). The in-app Privacy section (`AboutPopup.tsx`) is a promise the code must keep.

- **`blob2` is `device_id_hash`, not `install_id`.** Every client computes `HMAC_SHA256(SALT, machine_id || platform)` locally and sends only the 64-char hex hash тАФ the raw machine_id never reaches the Worker. SALT is baked into the desktop (`analytics-salt.ts`) and Android (`Salt.kt`) source, kept in sync.
- **Client `device_id_hash` is stable across reinstalls** тАФ it's derived from the machine's hardware ID (not a UUID in a file), and `~/.claude/youcoded-analytics.json` persists across uninstall/reinstall anyway. Wiping `~/.claude` only matters when the machine_id read fails (rare; falls back to a persisted UUID). Don't add code that rotates the hash on launch тАФ it would defeat the device-counting metric.
- **Only `lib/analytics.ts#writeAppEvent` may call `env.APP_ANALYTICS.writeDataPoint`.** Any new field goes through that wrapper so the blob order stays synced with the admin SQL. Blob order (the contract): `["heartbeat"(vestigial), deviceIdHash, appVersion, platform, os, country, region]` тАФ blob1 is a constant discriminator kept for forward-compat, blob7 is `region` (ISO 3166-2 from `CF-IPRegionCode`). `writeDataPoint` is fire-and-forget and silent on failure (no receipt/retry/durability тАФ use D1 for durable data). The `?.` in `env.APP_ANALYTICS?.writeDataPoint()` is load-bearing.
- **NEVER log raw `device_id_hash` server-side outside of `count(DISTINCT)`.** The privacy-by-construction guarantee is that `blob2` appears only inside cardinality aggregates in admin SQL. Breaking it even in a debug log breaks the in-app Privacy promise.
- **One event type only.** The retired `install` event is gone тАФ `/app/install` returns 410. "New installs per day" is derived at query time as `MIN(timestamp) GROUP BY blob2`.
- **Country and region are read SERVER-side**, never sent from the client тАФ `CF-IPCountry` + `CF-IPRegionCode` (ISO 3166-2, e.g. `US-CA`). The privacy copy promises country + approximate (state-level) region; finer geography is out of scope.
- **No cross-tabulation of region with other dimensions** тАФ region + version + platform + OS could fingerprint at low cell counts. Single-dimension queries only (the six endpoints each GROUP BY exactly one dimension).
- **`adminFilterClause` + `cutoverClause` (`lib/admin-filter.ts`) are the SQL safety boundary.** Both return `""` or a sanitized `AND ...` fragment; hex-only and ISO-only sanitization is mandatory because AE has no parameter binding (we string-interpolate). A `KNOWN_DEV_DEVICES` secret holds Destin's own device hashes, filtered out of admin queries by default; `?include_admins=1` bypasses for debugging.
- **CF Analytics Engine SQL is a narrow subset, NOT full ClickHouse** (422 responses learned the hard way): (1) cardinality is `count(DISTINCT col)` тАФ `uniq()` and `COUNT_DISTINCT()` both raise "unknown function call"; (2) `INTERVAL '30' DAY` takes a QUOTED STRING literal тАФ unquoted integers raise "Expected literal string, found: 30"; (3) the account ID + narrow-scope `CF_ANALYTICS_TOKEN` are distinct from the broader `CF_API_TOKEN` used by `wrangler deploy`. **Subqueries in `FROM` may or may not be supported** тАФ the derived-installs query depends on it; validate via a deploy smoke test.
- **AE free-tier retention is 90 days.** Pre-cutover (install_id-keyed) rows age out automatically; the `CUTOVER_TIMESTAMP` env var explicitly excludes them from queries until they do. Privacy copy promises no specific retention тАФ verify current AE retention before any copy change that adds one.
- **Admin auth has two paths via `requireAdminAuth`.** Cookie-flow session `Authorization: Bearer <token>` still works; the `youcoded-admin` admin CLI skill uses `X-GitHub-PAT: <gh auth token>`, which the Worker trades for a platform account id by resolving the GitHub `/user` numeric id through the `identities` table (cached 60s by SHA-256 hash in `auth/pat.ts`; requires the admin to have signed into the marketplace once). The `isAdminAccount()` allowlist check (`auth/admin.ts`, keyed on bare GitHub numeric ids) stays inline in each route so 401 (not auth'd) and 403 (auth'd but not admin) stay distinguishable. `ADMIN_USER_IDS` holds bare GitHub numeric ids (repo secret `MARKETPLACE_ADMIN_USER_IDS`), NOT `github:<id>` strings.
