- [ ] Session names revert to "New Session" after a renderer crash `bug` `#renderer` `#native-runtime` (added 2026-07-17)

- [ ] Android ManagedSession bare-phrase screen scans can false-fire prompt cards `bug` `#android` `#terminal-parser` (added 2026-07-16)
  Same class as the fixed "trust" substring collision, found in the 2026-07-16 sweep: ManagedSession.kt's non-menu prompt heuristics match bare phrases against the whole lowercased screen вЂ” the paste-code card fires on ("paste code"|"paste the code"|"browser") + ("sign"|"code"|"authorize"), and the Continue/"Ready" card fires on "press enter to continue" вЂ” so Claude conversation output containing those phrases (e.g. instructions that literally say "press Enter to continue") creates phantom prompt cards on Android. Not fixed in the sweep because these gate the Android sign-in flow and need on-device verification; fix should anchor on option shape / distinctive Ink chrome rather than conversation-plausible phrases.
  Repro: Destin sees the MovedGate ("This session was taken over on another device") after just opening a fresh session and leaving it idle / browsing other menus вЂ” no other device involved. Root cause spans both sides of the lease heartbeat: (1) client `lease-client.ts` `renewTick` (lines 142-165) renews every 30s via an `unref()`'d `setTimeout`, and on ANY `!r.ok` renew response unconditionally treats it as "another device force-acquired it," tearing the session down and calling `opts.onTakeoverRequest(sessionId)` with **no `from`** вЂ” that missing `from` is exactly why `MovedGate.tsx:39` falls back to the generic "another device" text (a genuine cross-device takeover always carries the real device name). (2) server `wecoded-marketplace/worker/src/sync/room.ts` lazily expires any lease record older than the 300s TTL on the NEXT op (`if (rec && rec.expiresAt <= now) rec = null`, line ~168), and `renew` only succeeds `if (rec && rec.deviceId === deviceId)` (line 183) вЂ” so `!r.ok` on renew has two conflated causes: genuinely stolen, OR nobody stole it, our own heartbeat just missed its 300s (ten 30s beats) window. A `setTimeout` heartbeat is suspended by system sleep/lock or OS throttling of an idle/backgrounded process вЂ” any such gap misreports as a hostile takeover of a session nothing else ever touched. Fix: on renew failure, `query()` (op `get`) the actual current holder before tearing down вЂ” only surface MovedGate if held by a genuinely different `deviceId`; if unheld, silently re-acquire and continue. Confirmed against the DO's own lazy-expiry test (`wecoded-marketplace/worker/test/sync-hub.test.ts` ~L279-289).
