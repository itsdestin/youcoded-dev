## Bugs

- [ ] `subagent-watcher` fs.watch test flakes on macOS, failing every desktop build `bug` `#tests` `#ci` (added 2026-07-16)
  `desktop/tests/subagent-watcher.test.ts:90` ("picks up a subagent file that appears after start") does `await wait(1500) // allow fs.watch/poll to fire` then asserts the emitted event — on the macOS runner nothing arrives in time, so `stamped?.data.parentAgentToolUseId` is `undefined` and `npm test` exits 1 (`1 failed | 2343 passed`). Observed on beta run 29553208961 (2026-07-16); Linux + Windows passed the same suite in the same run, and the sibling `replays an existing subagent file on start` passes because it uses the replay path and never arms the watcher. macOS `fs.watch` goes through FSEvents, whose coalescing latency exceeds the fixed 1500 ms budget under runner load. Impact is wider than one red check: `npm test` gates the Build step, so the macOS `.dmg` never gets produced — a beta dispatch yields no macOS artifact, and the whole run reports failure even when Win/Linux are fine. Fix by replacing the fixed sleep with polling until the condition holds (bounded retry) rather than raising the constant; the timeout is the symptom, the fixed-budget assumption is the bug. Not yet confirmed whether it's a hard macOS failure or load-dependent flake — one observation so far, worth a re-dispatch to tell.

- [x] Stale "answer the prompt first" toast permanently blocks chat send `bug` `#renderer` `#terminal-parser` (added 2026-07-16; fixed 2026-07-16, youcoded PR #157 — on menu-id change usePromptDetector now cancels the pending show timer and dismisses the previously SHOWN prompt before the SETUP_PROMPT_TITLES gate can bail; gated on shownPromptRef so streaming-list id churn dispatches nothing.   Repro: resume one or more sessions (ResumeBrowser → `session-manager.ts` launches `claude --resume <id>`, which shows CC's own "Resuming from a summary" Ink prompt — `SETUP_PROMPT_TITLES` has 'Resume Session'). If the underlying PTY screen advances to any OTHER Ink-parseable content (a permission Yes/No/Always-Allow menu, or even a false-positive numbered list in Claude's plain-text output) before the user clicks the Resume Session card, the card gets permanently stuck with `completed: false` and `InputBar.tsx`/`App.tsx` refuse all future sends with "Claude is waiting for your response — answer the prompt first," even though nothing is live on screen. Root cause: `usePromptDetector.ts` lines 117-123 — `lastMenuRef.current.set(sid, menu.id)` runs BEFORE the `SETUP_PROMPT_TITLES` check, so when a new (unrecognized) menu is detected, the ref is overwritten to point at it and the function returns early without dismissing the OLD prompt. - [ ] Android ManagedSession bare-phrase screen scans can false-fire prompt cards `bug` `#android` `#terminal-parser` (added 2026-07-16)
  Same class as the fixed "trust" substring collision, found in the 2026-07-16 sweep: ManagedSession.kt's non-menu prompt heuristics match bare phrases against the whole lowercased screen — the paste-code card fires on ("paste code"|"paste the code"|"browser") + ("sign"|"code"|"authorize"), and the Continue/"Ready" card fires on "press enter to continue" — so Claude conversation output containing those phrases (e.g. instructions that literally say "press Enter to continue") creates phantom prompt cards on Android. Not fixed in the sweep because these gate the Android sign-in flow and need on-device verification; fix should anchor on option shape / distinctive Ink chrome rather than conversation-plausible phrases.
  Repro: Destin sees the MovedGate ("This session was taken over on another device") after just opening a fresh session and leaving it idle / browsing other menus — no other device involved. Root cause spans both sides of the lease heartbeat: (1) client `lease-client.ts` `renewTick` (lines 142-165) renews every 30s via an `unref()`'d `setTimeout`, and on ANY `!r.ok` renew response unconditionally treats it as "another device force-acquired it," tearing the session down and calling `opts.onTakeoverRequest(sessionId)` with **no `from`** — that missing `from` is exactly why `MovedGate.tsx:39` falls back to the generic "another device" text (a genuine cross-device takeover always carries the real device name). (2) server `wecoded-marketplace/worker/src/sync/room.ts` lazily expires any lease record older than the 300s TTL on the NEXT op (`if (rec && rec.expiresAt <= now) rec = null`, line ~168), and `renew` only succeeds `if (rec && rec.deviceId === deviceId)` (line 183) — so `!r.ok` on renew has two conflated causes: genuinely stolen, OR nobody stole it, our own heartbeat just missed its 300s (ten 30s beats) window. A `setTimeout` heartbeat is suspended by system sleep/lock or OS throttling of an idle/backgrounded process — any such gap misreports as a hostile takeover of a session nothing else ever touched. Fix: on renew failure, `query()` (op `get`) the actual current holder before tearing down — only surface MovedGate if held by a genuinely different `deviceId`; if unheld, silently re-acquire and continue. Confirmed against the DO's own lazy-expiry test (`wecoded-marketplace/worker/test/sync-hub.test.ts` ~L279-289).
