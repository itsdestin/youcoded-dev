## Bugs

- [ ] `artifacts:save` bypasses write-guard in both directions `bug` `#native-runtime` `#artifacts` `#hooks` (added 2026-07-20)
  `hook-scripts/write-guard.sh` is registered as a Claude Code `PreToolUse` hook on `Write|Edit` (`scripts/install-hooks.js:200`). It is a same-machine concurrency lock, not a permission guard: it reads `~/.claude/.write-registry.json` and blocks (exit 2) only when a *different, still-alive* Claude PID last wrote the file. Its whitelist explicitly covers `CLAUDE.md`, `settings.json`, and `mcp.json` (`write-guard.sh:31-33`). But **`artifacts:save` calls `fs.promises.writeFile` directly in the Electron main process** (`ipc-handlers.ts:3113-3114`, `:3140-3141`) ŌĆö no hook dispatch, no registry read, **and no registry write**. So the break is symmetric: a user editing in the artifact pane is not subject to the guard, *and* the agent's next write to that file sees no ownership conflict because nothing recorded the pane's write. Today this is nearly harmless because the pane can only edit `md`/`markdown`/`txt` (`ActiveArtifactView.tsx:48`) ŌĆö it becomes live the moment the Tier 1 code-editor workstream lifts that gate, which is why it is filed separately rather than bolted into that PR. **Sequencing caution:** per CLAUDE.md, until the youcoded-core deprecation release ships, write-guard fixes must land in BOTH the youcoded-core copy and the app's bundled copies ŌĆö so coupling the artifact save path to registry participation mid-deprecation is the thing to avoid; wait for the deprecation to complete, or scope the fix to the app's copy only and note it. (found 2026-07-20 in the Tier 1 code-editor consequence review ŌĆö spec ┬¦12.2)

- [ ] Artifact sidecar `versions[]` grows unbounded, and each save rewrites the whole file `bug` `#artifacts` `#perf` (added 2026-07-20)
  `appendVersion` (`artifact-store.ts:58`) pushes to `versions[]` (`:108`) with **no cap, no eviction, no pruning, and no comment acknowledging it** ŌĆö unlike `permission-store.ts:25-26`, which at least documents its unbounded growth as intentional and names the UI that will prune it. Size alone is minor (~250ŌĆō300 bytes per event once `JSON.stringify(next, null, 2)` pretty-prints it at `artifact-store.ts:37`; 500 edits of one file Ōēł 125ŌĆō150KB). **The write pattern is the real cost:** `appendVersion` does a full read-parse-mutate-serialize-CAS-write of the *entire* sidecar on every save, with up to `MAX_RETRIES = 5` attempts (`artifact-store.ts:46`) ŌĆö so cost per save is O(total sidecar size) across all artifacts in the project, not O(this file). Mitigating asymmetry today: the *discovered*-file save branch deliberately skips the sidecar entirely (`ipc-handlers.ts:3131-3133`, so editing a stray doc never silently creates a `.youcoded/` dir), so growth only hits already-tracked files. The Tier 1 code-editor workstream makes this materially worse on two axes at once ŌĆö more tracked records AND more events per record ŌĆö but capping/compacting `versions[]` changes the sidecar's on-disk contract and needs its own migration story, so it is filed here rather than scheduled into that PR. Decide whether old events are dropped, coalesced by session, or moved to a rotated file. (found 2026-07-20 in the Tier 1 code-editor consequence review ŌĆö spec ┬¦12.8)

- [ ] Sync dead-ends on any machine without `gh` ŌĆö the setup panel must not require a terminal `bug` `#sync` `#onboarding` (added 2026-07-20)
  Enabling sync on a stock macOS install fails with no usable signal: `gh` is absent (Xcode CLT ships `git`, not `gh`), `provisionGithubRemote` throws ENOENT, and the user sees "Setting upŌĆ”" then a green "All synced" over an empty space (the two `#sync` bugs below). Destin hit this on the beta.8 VM 2026-07-20 and only got past it by hand-installing the release tarball and running `gh auth login` in Terminal ŌĆö **exactly the terminal step the product exists to avoid.** The same trap awaits stock Windows and most Linux installs. **The auth half is already solved and should not be rebuilt:** `github-auth.ts` runs GitHub's OAuth *device flow* in-app (its header: "Non-developers can't be expected to open a terminal and run `gh auth login`"), mints a token, and pipes it to `gh auth login --with-token` ŌĆö so the app already holds a valid token and merely needs `gh` to *exist* to store it. Two fix shapes; the second is the one worth doing:
  **(A) Stopgap ŌĆö teach `prerequisite-installer.ts` about `gh`.** That module already does user-local, sudo-free installs with PATH registration + absolute-path fallback (built for Claude Code); `gh` ships prebuilt tarballs for darwin x64/arm64, linux and win32, so this is a new entry in an existing pattern, not new machinery. Fastest path to "the panel does it for you", and it's the minimum Destin asked for.
  **(B) Real fix ŌĆö drop the `gh` dependency entirely.** `gh` is used for exactly two things: `repo create`/`repo view` (`space-manager.ts:40,46`), and acting as the git credential store for push/pull (`git-transport.ts` has NO credential logic ŌĆö it relies on the helper `gh auth login` configures). The app already has the token, so both collapse into things it can do itself: repo create/view via the REST API using the same `fetch` the device flow already uses, and push/pull auth via `GIT_ASKPASS` or an inline `credential.helper` in the per-call env the transport already builds alongside `GIT_DIR`/`GIT_WORK_TREE`. That deletes a ~13 MB third-party binary, an install step on every platform, and the whole "gh not installed / not authed" failure class. **Constraint that must survive: token hygiene** (`github-auth.ts` ŌĆö token never logged, never in an error string, never over the remote socket, with a test enforcing it). An askpass/credential-helper env var keeps it in process memory; **do NOT embed it in the remote URL**, which would write the token into `.git/config` on disk. Sequencing: (A) is about a day, (B) is a design pass ŌĆö but shipping (A) alone leaves every user carrying a CLI they never asked for, so treat (A) as a release unblocker and (B) as the plan. (found 2026-07-20 dogfooding beta.8 on the macOS VM)
