- [ ] Buddy floater: Wayland-native one-window rewrite (drag broken on native Wayland) `bug` `#buddy` `#linux` (added 2026-07-17)
  On native Wayland (Electron ≥38.2 default), the buddy floater is unusable: it can't be dragged (stuck centered), and edge-anchored peek/sink/lean animations misfire inside the fixed 112×112 window. Root cause is architectural, not a code bug — Wayland forbids apps from programmatically positioning (`setPosition` is a no-op, `getPosition` returns `[0,0]`) or introspecting global window coordinates, and the floater's entire drag model is renderer-pointermove → `buddy:move-mascot` IPC → main-process `setPosition` every frame across THREE separate transparent frameless BrowserWindows (mascot + chat + bar, see `buddy-window-manager.ts`, `buddy-bar-geometry.ts`, `BuddyMascot.tsx`). Confirmed against Electron docs + issues #52204/#48833/#40886; repro env: CachyOS, KDE Plasma (KWin 6.7.3), Wayland, single 1.5×-scaled panel. The proper fix (per Wayland devs + Electron maintainers) is to collapse the three windows into ONE fullscreen transparent always-on-top window and position the mascot/chat/bar as absolutely-positioned DOM inside it — dragging becomes a local CSS write (no IPC, no platform permission, dodges the transparency-smear bug class too). Hard parts priced in: rewiring the 7 cross-webContents IPC pushes + session-subscription-by-webContents.id into in-app React state, and hand-managing dynamic click-through regions (`setIgnoreMouseEvents(forward:true)` + hover-region tracking) across the dead transparent area — the cursor-timing-sensitive debugging the current per-window design gets for free. Do NOT do the `--ozone-platform=x11` XWayland stopgap as the fix (considered 2026-07-17, rejected): it blurs the whole app at 1.5× fractional scaling AND still needs an Electron bump for the separate transparent-window smear bug (electron#50541, propagating to 41-x). This rewrite SUBSUMES the "scene-companion follow physics — window padding redesign" item under Features (the padding/click-through problem disappears in a one-window model). Prototype the click-through + hover behavior in a workbench first, like the rig/peek work.

## Bugs

- [ ] Native-session takeover destroys the holder and transfers nothing — the lease outran the data path `bug` `#native-runtime` `#sync` (added 2026-07-18)
  youcoded PR #176 (merge `e7b09f60`) correctly identified that native-provider sessions had zero takeover protection and gave them one — `ipc-handlers.ts:541-548` now sets `sessionIdMap.set(info.id, info.id)` + `noteSessionStarted` + `leaseClient.acquire` after a successful native create/resume. But the holder-teardown path they're now enrolled in does not function for native, so the resume gate advertises a handoff the data layer can't perform. Three independently verified breaks: (1) **the interrupt is a silent no-op** — `session-manager.ts:250` is `if (!session || !session.worker) return false; // native sessions have no PTY`; it RETURNS false rather than throwing, so the teardown's `try/catch` never sees it and the return value is discarded, and a native turn keeps streaming straight through step 3. (2) **the flush mirrors nothing** — `flushSessionToSpace` reads `localJsonlPath` = `~/.claude/projects/<ccProjectSlug(cwd)>/<id>.jsonl` (`conversations/service.ts:127`), but native transcripts are written to `~/.youcoded/sessions/<slug>/<id>.jsonl` (`native-home.ts:115`); `mirrorIn` throws on the missing source and the `catch { /* best-effort */ }` swallows it. (3) **the requester pulls nothing** — `materializeOne` bails at `if (!rec?.transcriptRef) return` (`conversations/service.ts:298`), and no record exists because `noteTranscriptEvent` (the only writer, hardcoded `provider: 'claude'`) is fed exclusively by `TranscriptWatcher` over the CC projects dir (`ipc-handlers.ts:1917-1927`) — the native host's own `transcript-event` emitter is never routed into it. Net: device B's handoff destroys device A's live session (`destroySession` still runs) while device B resumes its own stale local copy. **This is a behavior regression, not just a gap** — before the PR, no lease meant no handoff offer, so device A kept running. Root cause is ordering: native transcripts do not participate in space sync AT ALL, so the lease acquire landed ahead of the capability it implies. Cheap fix = gate the native acquire behind native-transcript-sync existing (restores pre-PR behavior, keeps the honest `held:false`). Real fix = make native conversations first-class in the conversation store + space sync (needs a `provider` kind that isn't `'claude'`, a native-aware `localJsonlPath`/`transcriptRef`, a native interrupt on the teardown in place of the ESC byte, and a native branch in `materializeOne`) — that's a design pass, and it's the same work any "resume my native conversation on another device" feature needs. Note the two safe halves of the same PR that DO hold and shouldn't be touched: the DO guards `release` on `rec.deviceId === deviceId` (`wecoded-marketplace/worker/src/sync/room.ts:231`) so a victim's post-force release can't clobber the attacker's lease, and `broadcastLeaseEvent` excludes the sender (`room.ts:268`) so the forcing device never self-tears-down. No test covers the native lease registration. (found 2026-07-18 reviewing youcoded PR #176 post-merge)
  **Corrections + scope update 2026-07-18** (from the full investigation, docs/active/investigations/2026-07-18-native-session-takeover-gap.md): (a) break (2) above is wrong about the mechanism — `mirrorIn` does NOT throw; `sizeOf` catches the `statSync` throw and returns `null`, and `mirrorIn` returns `{copied:false}` early at `transcript-mirror.ts:68`. No throw, no catch, no signal — so adding logging to that `catch` would do nothing, control never reaches it. The holder also does NOT short-circuit: it runs the whole barrier (quiesce on a missing file, no-op mirror, a real 15s awaited space sync) and burns ~16s before releasing. (b) **A fourth, more severe break was missed: the harness is orphaned, not destroyed.** `takeover.ts:80-83` calls `destroySession` directly, skipping `await nativeHost.destroy(sessionId)`. Since `resume()` never consults `this.live`, two live sessions can append to one native JSONL — permanently corrupting it, violating the single-writer invariant at `native-home.ts:5-7` — plus a leaked `releaseModel` pins a local model in VRAM for the app's lifetime. Data loss outranks session loss; this is the severity driver. (c) The suite doesn't merely miss this, it **certifies** it: `holder-takeover.test.ts` passes on a native session that was never interrupted, because its fake `sendInput` records the call and can't express "returned false and did nothing." (d) Fix scope is now the full-parity spec: docs/active/specs/2026-07-18-native-sync-parity-design.md.

- [ ] Native "Always allow" is near-useless for Bash — the remembered rule is a literal command string `bug` `#native-runtime` `#permissions` (added 2026-07-18)
  `harness-session.ts:562` emits `pattern: subject`, and Bash's `permissionSubject` is `(a) => a.command` (`tools/bash.ts:35`), so the remembered rule stores the **exact full command string**, matched anchored `^…$` in `tools/subject-glob.ts` (only `*`/`?` are wildcards). There is no prefix extraction, normalization, or command splitting anywhere in the path. Practical effect: always-allowing `git push origin main` grants nothing for `git push origin dev`, so a user who "already allowed this" keeps being re-prompted — and for deny-listed commands each re-prompt is the full destructive confirm. Note the asymmetry: tools whose `permissionSubject` returns `undefined` (TodoWrite, AskUserQuestion) emit a rule with no `pattern`, and `subjectMatches` treats that as tool-wide — so always-allow is *too broad* there and *too narrow* for Bash. **Needs a design pass, not a quick fix**, because every candidate (prefix rules, argv-head matching, user-editable glob at confirm time) WIDENS a grant that currently outranks `DESTRUCTIVE_DENY_LIST` permanently (`permission-engine.ts:26-41`, remembered rules are the last layer and win). The accidental narrowness is currently the only thing limiting blast radius — don't remove it without replacing it with something deliberate. Pairs with the revocation-UI item under Features: widening the grant is much safer once a grant can be taken back. (found 2026-07-18 tracing the deny-listed confirm for youcoded PR #173)
